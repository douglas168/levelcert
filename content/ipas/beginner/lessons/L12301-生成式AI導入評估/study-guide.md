## Section 3 · 核心概念

### 3.1 四階段導入框架（構方驗實）

**先懂一句話：** 導入 AI 不能一步到位——要先想清楚、再設計方案、再小規模驗證、最後才全面實施。

**定義：** 經濟部產業發展署（Industrial Development Administration，工業及服務業相關主管機關）發布的《製造業AI導入指引》（AI Adoption Guide），定義了 AI 導入的四個循序漸進階段：構想評估、方案設計、驗證/POC、實施/營運。

**四階段框架：**

```
┌──────────────────┐   ┌──────────────┐   ┌──────────────┐   ┌──────────────────┐
│ 1. 構想評估       │ → │ 2. 方案設計   │ → │ 3. 驗證/POC  │ → │ 4. 實施/營運      │
│ Ideation &       │   │ Solution     │   │ Validation   │   │ Implementation   │
│ Assessment       │   │ Design       │   │ & POC        │   │ & Operations     │
│                  │   │              │   │              │   │                  │
│ · 業務痛點分析    │   │ · 方案選擇    │   │ · POC 設計   │   │ · 正式部署        │
│ · 資源盤點        │   │ · 技術選型    │   │ · 成功標準   │   │ · 上線推廣        │
│ · AI 適配度評估   │   │ · KPI 定義   │   │ · MVP        │   │ · KPI 追蹤        │
│ · 場景選定        │   │ · 成本估算    │   │ · 退出標準   │   │ · 持續監控        │
└──────────────────┘   └──────────────┘   └──────────────┘   └──────────────────┘
                                                                        │
                          ← ← ← 可回到前一階段調整（非跳過）← ← ← ← ←┘
```

參見 diagrams/01-four-phase-framework.md

| 階段 | 核心問題 | 關鍵產出 |
|---|---|---|
| 構想評估（Ideation & Assessment） | 這個問題適合用AI嗎？ | 應用場景清單 |
| 方案設計（Solution Design） | 用什麼？怎麼做？ | 專案計畫書、KPI |
| 驗證/POC（Validation） | 小規模能做到嗎？ | POC（Proof of Concept，概念驗證）報告 |
| 實施/營運（Implementation & Operations） | 如何全面落地？ | 上線系統、KPI（Key Performance Indicator，關鍵績效指標）追蹤 |

🗣️ **白話說明：** 想像你要開手搖飲店。你不會第一天就花 200 萬裝潢吧？一定是先想清楚「要開幾坪？賣什麼品項？目標客群？」（構想）→ 設計菜單和供應鏈（方案）→ 先在市集擺一週的攤位測試看看（驗證 / POC）→ 確認有生意才租固定門市（實施）。《AI導入指引》就是這個常識的系統化版本。

**Quick check：** POC（Proof of Concept，概念驗證）的目的是在**小規模**驗證可行性，**不是**正式上線。

### 3.2 四大評估維度（技工模效）

**先懂一句話：** 導入AI前，必須從技術、工具、模型、效能四個角度全面評估。

**定義：** 四大評估維度是決定「技術做得到嗎」（技術評估）、「用哪個工具」（工具評估）、「這個模型夠用嗎」（模型評估）、「有沒有帶來效益」（效能評估）的四道關卡。口訣：**技工模效**。

```
參見 diagrams/02-evaluation-four-dimensions.md
```

#### 3.2.1 技術評估（Technical Assessment）

**核心問題：** 我們的基礎設施和資料支撐得了嗎？

- 技術可行性（Technical Feasibility）：現有伺服器、算力、網路頻寬是否足夠
- API（Application Programming Interface，應用程式介面）整合：現有系統能否串接 AI 服務
- 資料品質（Data Quality）：資料夠多、夠乾淨、格式統一嗎？

🗣️ **白話說明：** 就像你想裝工業咖啡機——先確認電壓夠不夠（算力）、水管連得上嗎（API 整合）、有好咖啡豆來源嗎（資料品質）。三缺一都跑不動。

**Quick check：** 技術評估不只看硬體設備，「資料品質」和「API整合難度」也是技術評估的範疇。

#### 3.2.2 工具評估（Tool Evaluation）

**核心問題：** 哪個AI工具或平台最適合我們的需求？

- 功能完整度：能否滿足業務需求
- 整合能力：能否與現有系統串接
- 供應商穩定度（Vendor Stability）：廠商是否可靠，服務是否持續
- 成本結構：按Token計費 vs 月費訂閱？

🗣️ **白話說明：** 就像你要選專案管理工具——Notion、Trello、Asana 都能用，但你的團隊都用 Google 生態系，選 Google 有整合的工具可能更適合。工具評估就是「貨比三家」。

**Quick check：** 工具評估比的是「市面上哪個工具最適合」——不一定選最貴的，要選最適配的。

#### 3.2.3 模型評估（Model Evaluation）

**核心問題：** 這個 AI 模型能勝任這個任務嗎？

- 模型能力（Model Capability）：適合哪種類型的任務
- 幻覺風險（Hallucination Risk）：模型產生不實輸出的機率，在高準確度要求場景中尤其重要
- 準確率（Accuracy）：在實際業務場景中的表現

🗣️ **白話說明：** 就像你叫外送選餐廳——A 餐廳超好吃但要 40 分鐘又貴、B 餐廳普通但 15 分鐘到又便宜。模型評估就是「在這個場景下，哪個模型的表現最符合需求」，不是選「最強」的。

**Quick check：** 初級只需理解「根據場景需求選最適合的，不是最貴最強的」，不需要計算準確率。

#### 3.2.4 效能評估（Performance Evaluation）

**核心問題：** AI系統導入後，是否真的帶來了預期的效益？

- ROI（Return on Investment，投資報酬率）：效益 ÷ 成本
- KPI 追蹤：效率提升、錯誤率降低、客戶滿意度
- 成本效益（Cost-Benefit）比較：導入前後對照

🗣️ **白話說明：** 你在蝦皮開店後裝了AI客服，怎麼知道有沒有用？看數據——回覆速度從 2 小時變 30 秒、客訴量下降 40%、客服人力從 3 人變 1 人。這就是效能評估。

**Quick check：** 效能評估是**導入後**才做的，不是導入前！技術評估、工具評估、模型評估是導入前的三個面向。

### 3.3 解決方案選擇：Build / Buy / SaaS

**先懂一句話：** 導入AI時，要根據客製化需求、資料隱私、預算和技術能力，選擇自建、購買或訂閱。

**定義：** Build（自建）、Buy（購買授權）、SaaS（Software as a Service，雲端訂閱服務）是導入 AI 的三種主要路徑，各有其適用情境。

```
參見 diagrams/03-build-buy-saas-decision.md
```

| 路線 | 英文 | 說明 | 適合情境 | 典型例子 |
|---|---|---|---|---|
| 自建 | Build | 自己組團隊開發 | 核心競爭力、高度客製、資料絕不外傳 | 自建內部文件問答系統 |
| 購買授權 | Buy | 買現成解決方案 | 標準化需求、中度客製化 | 購買 Salesforce AI CRM |
| 雲端訂閱 | SaaS | 訂閱雲端服務 | 快速上線、非核心功能、預算有限 | 訂閱 ChatGPT Enterprise |

**決策因素（概念層級）：**

1. **安全需求與資料隱私** — 若資料不能離開公司 → 傾向 Build 或私有部署
2. **預算** — 預算有限、先試水溫 → SaaS
3. **技術能力** — 沒有 AI 開發團隊 → Buy 或 SaaS
4. **客製化程度** — 高度客製化、核心業務 → Build
5. **上線速度** — 需要快速上線 → SaaS

🗣️ **白話說明：** 就像午餐選擇：自己煮（Build，最客製但最耗時）、去便利商店買便當（Buy，現成可用）、叫 Uber Eats（SaaS，最彈性但長期可能更貴）。要看你有多少時間、多少錢、有多特別的需求。

**Quick check：** 「核心競爭力 + 高度客製 + 資料不外傳」→ Build；「快速上線 + 預算有限 + 非核心」→ SaaS。

### 3.4 AI Agent vs 一般 GenAI 工具

**先懂一句話：** AI Agent 能自主規劃和執行多步驟任務，一般 GenAI 工具只能一問一答。

**定義：**
- **一般 GenAI 工具**（General GenAI Tool）：接收單一提示，產生一次性輸出，需人工判斷下一步。例如：問 ChatGPT「幫我寫一封 Email」，它給你一封 Email，任務結束。
- **AI Agent**（AI 代理）：結合大型語言模型（LLM）、記憶（Memory）、工具呼叫（Tool Calling）與自主規劃（Autonomous Planning），能自動執行多步驟任務。例如：指令「幫我安排下週去台北的出差行程」，Agent 會自己查高鐵、找飯店、排時間、發提醒。

**AI Agent 運作循環（觀決行）：**

```
    ┌──────────┐
    │  觀察     │ ← 接收任務目標
    │ Observe  │
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │  決策     │ ← LLM 思考下一步行動
    │ Decide   │
    └────┬─────┘
         │
         ▼
    ┌──────────┐
    │  行動     │ ← 呼叫工具 / 執行步驟
    │  Act     │
    └────┬─────┘
         │
         └──→ 循環直到任務完成
```

**AI Agent 四大核心組件（腦筆手主）：**

| 組件 | 功能 | 白話比喻 |
|---|---|---|
| LLM（大型語言模型） | 思考與推理 | 助理的腦袋 |
| Memory（記憶） | 跨步驟保持上下文 | 助理的筆記本 |
| Tool Calling（工具呼叫） | 執行外部操作 | 助理的雙手 |
| Autonomous Planning（自主規劃） | 自行決定下一步 | 助理的主見 |

**核心比較表：**

| 比較項目 | 一般 GenAI 工具 | AI Agent |
|---|---|---|
| 互動模式 | 一問一答，被動 | 給目標，自主完成 |
| 步驟數 | 單步驟 | 多步驟，自行規劃 |
| 記憶 | 僅限當次對話 | 跨步驟保持上下文 |
| 工具使用 | 不能（或極有限） | 能呼叫外部工具 |
| 自主決策 | 被動等待指令 | 自行判斷下一步 |
| 適用場景 | 文案寫作、翻譯、問答 | 重複性高、多步驟、跨系統整合 |
| 白話比喻 | 7-11 店員（你問他答） | 私人助理（自己規劃） |

**何時選擇 AI Agent：**
- 任務需要多個步驟才能完成
- 需要跨系統整合（例：查庫存 → 比價 → 下單）
- 高重複性的自動化工作流
- 需要持續監控並自動回應的場景

🗣️ **白話說明：** AI Agent 就像你雇了一個能自己想事情的助理——你說「幫我安排出差」，他會自己查時刻表、訂飯店、排行程，不需要你一步步下指令。一般 GenAI 工具更像是「智慧搜尋引擎」——你問什麼就回答什麼，回答完就結束。

**Quick check：** 題目出現「自主規劃」「多步驟」「呼叫工具」「觀察-決策-行動」→ 選 AI Agent。

### 3.5 成本分析：顯性成本 vs 隱藏成本

**先懂一句話：** AI 導入的真實成本遠超過 API 費用——隱藏成本才是大頭。

**定義：**
- **顯性成本**（Visible Costs）：容易看到、容易計算的直接費用。
- **隱藏成本**（Hidden Costs）：常被低估或忽略的間接費用，通常佔 TCO（Total Cost of Ownership，總持有成本）的 85–95%。

```
參見 diagrams/04-cost-iceberg.md（成本冰山圖）
```

**成本冰山（Cost Iceberg）：**

```
                    水面上（顯性成本）
        ╔════════════════════════════════╗
        ║  API 費用（依呼叫次數計費）       ║
        ║  授權費（軟體 License）           ║
        ║  硬體（GPU / 伺服器）             ║
        ╚════════════════════════════════╝
～～～～～～～～～水面～～～～～～～～～～～～～～～～
        ╔════════════════════════════════╗
        ║  整合開發（與現有系統串接）         ║  水
        ║  員工培訓（教育訓練費用）           ║  面
        ║  流程改造（變更管理成本）           ║  下
        ║  維護費（模型更新、監控）           ║
        ║  資料治理（清洗、標註、合規）        ║（隱藏成本）
        ║  風險管理（合規審查、保險）         ║
        ╚════════════════════════════════╝
```

| 成本類型 | 典型佔比 | 常見項目 |
|---|---|---|
| 顯性成本（Visible Costs） | 約 5–15% | API 費用、軟體授權費、硬體 |
| 隱藏成本（Hidden Costs） | 約 85–95% | 整合開發、員工培訓、流程改造、維護、資料治理 |

**TCO（Total Cost of Ownership，總持有成本）：**

TCO 是從「系統生命週期」視角評估 AI 導入所有成本總和的框架，包含顯性成本和所有隱藏成本。初級只需理解「TCO 比只算 API 費用更完整」——不需要計算具體金額（中級才考數字計算）。

🗣️ **白話說明：** 就像買車——車價（顯性成本）只是開始，之後還有油錢、保險、定期保養、停車費（隱藏成本）。只看車價而不算總持有成本，容易在預算上踩坑。AI 導入也一樣，API 費用是「車價」，隱藏成本才是「養車費」。

**Quick check：** 「以下何者是最常被忽略的AI導入成本」→ 選隱藏成本類的選項（員工培訓、系統整合、資料治理）。

---

### 3.6 KPI 設計與效能評估

**先懂一句話：** AI 導入後，要用對的 KPI 衡量真正的效益，而不是只看技術指標。

**定義：** KPI（Key Performance Indicator，關鍵績效指標）是用來衡量 AI 導入是否達成業務目標的具體數字指標。

**KPI 四大類型：**

| KPI 類型 | 衡量什麼 | 舉例 |
|---|---|---|
| 效率 KPI | 速度提升、自動化程度 | 處理時間減少 40%、自動化率達 80% |
| 品質 KPI | 準確度、客戶滿意度 | 準確率 ≥ 95%、客訴量下降 30% |
| 成本 KPI | 單位成本降低、人力節省 | 每件成本降低 25%、人力節省 2 FTE |
| 業務 KPI | 收入增長、客戶體驗 | 收入成長 15%、NPS（Net Promoter Score，淨推薦值）提升 10 分 |

**POC（Proof of Concept，概念驗證）成功標準設計：**

在驗證階段，需要事先定義：
1. **量化標準（Quantitative）**：具體數字目標（例：準確率 ≥ 90%）
2. **質化標準（Qualitative）**：主觀感受（例：員工認為流程更順暢）
3. **Exit Criteria（退出標準）**：達到哪些條件才算通過、或哪些條件代表應停止 POC

**MVP（Minimum Viable Product，最小可行產品）vs POC（Proof of Concept，概念驗證）：**

| 比較 | POC | MVP |
|---|---|---|
| 目的 | 驗證技術可行性 | 驗證商業價值 |
| 對象 | 內部技術評估 | 真實用戶使用 |
| 完整度 | 最小化原型 | 基本功能完整 |
| 問題 | 「技術做得到嗎？」 | 「用戶願意用嗎？」 |

🗣️ **白話說明：** KPI 就是你給 AI 系統的「成績單」。開學前先訂好「這學期至少要考過 80 分」（量化標準），上課一個月後（POC 期間）看成績有沒有達標（Exit Criteria）。沒達標就調整方向（回到前一階段），達標才全力推廣（進入實施階段）。

**Quick check：** POC 是「技術驗證」，MVP 是「市場驗證」；Exit Criteria 是「什麼條件代表這個 POC 成功或應該喊停」。

| **評估時機** | 導入**前** | 導入**前** | 導入**前** | 導入**後** ⚠️ |
| **評估對象** | 基礎設施+資料 | 市場上的AI工具 | AI 模型表現 | KPI 成效數據 |

### 4.2 AI Agent vs 一般 GenAI 工具 🔥🔥

| 比較項目 | 一般 GenAI 工具 | AI Agent |
|---|---|---|
| **觸發方式** | 每次需人工輸入提示 | 給目標後自主執行 |
| **步驟數** | 單次輸出 | 多步驟迴圈 |
| **記憶** | 僅限當次 session | 跨步驟保持上下文 |
| **工具使用** | 否（或有限） | 是，呼叫外部 API/工具 |
| **判斷決策** | 被動，等待下一個提示 | 主動，自行決定下一步 |
| **成本** | 按使用量低 | 多步驟累積成本較高 |
| **白話比喻** | 智慧 Google 搜尋 | 自己工作的私人助理 |

**考題判斷：** 「能自主查詢、比價、下單的系統」→ AI Agent；「問一個問題得到一個回答」→ 一般 GenAI 工具。「記憶」「多步驟」「呼叫工具」→ AI Agent 的關鍵詞。

### 4.3 Build / Buy / SaaS 決策比較 🔥🔥

| 比較項目 | Build（自建） | Buy（購買授權） | SaaS（雲端訂閱） |
|---|---|---|---|
| **資料主權控制** | 最高，可私有部署不外傳 | 中等，部分客製化隔離 | 最低，需信任供應商 |
| **客製化程度** | 最高 | 中等 | 最低（標準功能） |
| **上線速度** | 最慢（開發週期） | 中等 | 最快 |
| **初期成本** | 最高 | 中等 | 最低 |
| **維護責任** | 自己 | 共同 | 供應商 |
| **適合情境** | 核心競爭力、高安全需求 | 標準化業務、中度客製 | 快速驗證、非核心、預算有限 |

**考題判斷：** 「資料絕對不能上雲」→ Build；「功能標準，只需小改動」→ Buy；「新創公司快速上線」→ SaaS。

---

### 4.4 POC vs MVP 🔥

| 比較項目 | POC（概念驗證） | MVP（最小可行產品） |
|---|---|---|
| **全名** | Proof of Concept | Minimum Viable Product |
| **中文** | 概念驗證 | 最小可行產品 |
| **目的** | 驗證技術可行性 | 驗證商業/市場價值 |
| **受眾** | 內部技術團隊 | 真實用戶/客戶 |
| **問題** | 「技術上做得到嗎？」 | 「用戶真的需要嗎？」 |
| **在四階段的位置** | 第3階段（驗證） | 第3→4階段過渡 |

**考題判斷：** 「內部小規模驗證技術」→ POC；「真實用戶測試最基本功能」→ MVP。

---

### 4.5 顯性成本 vs 隱藏成本 🔥🔥

| 比較項目 | 顯性成本 | 隱藏成本 |
|---|---|---|
| **TCO 佔比** | 約 5–15% | 約 85–95% |
| **容易識別** | 是，有帳單可查 | 否，常被忽略 |
| **典型項目** | API 費用、授權費、硬體 | 整合開發、員工培訓、流程改造、資料治理、維護 |
| **考試考法** | 較少直接考 | 「以下何者最容易被忽略？」🔥 |

**考題判斷：** 題目問「最容易被低估的成本」或「長期成本考量」→ 選隱藏成本相關選項。

## Section 5 · 口訣

### 口訣 1：四階段框架 —「構方驗實」

> **構**想 → **方**案 → **驗**證 → **實**施
>
> 🗣️ 記法：「**構方驗實**」念起來像「工坊嚴實」——建立一個工坊（工廠）要嚴格審查才能扎實落地。四個字、四個階段，順序不能亂。

---

### 口訣 2：四大評估維度 —「技工模效」

> **技**術評估 → **工**具評估 → **模**型評估 → **效**能評估
>
> 🗣️ 記法：「**技工模效**」念起來像「技工模範」——好的技術工人（技術評估通過）用對了工具（工具評估通過）、選對了模型（模型評估通過），才能產出成效（效能評估看結果）。
>
> ⚠️ 關鍵：前三個（技、工、模）是導入**前**做；效能評估是導入**後**才能量測！

---

### 口訣 3：AI Agent 四大組件 —「腦筆手主」

> **腦**（LLM）+ **筆**記本（Memory）+ **手**（Tool Calling）+ **主**見（Autonomous Planning）
>
> 🗣️ 記法：一個好助理要有腦袋、隨時做筆記、有雙手能做事、還要有自己的主見。少一樣都不算真正的 Agent。

---

### 口訣 4：隱藏成本四大頭 —「整訓改維」

> **整**合開發 + 員工培**訓** + 流程**改**造 + 持續**維**護
>
> 🗣️ 記法：「**整訓改維**」——整合系統要培訓人員，改造流程才能維持運作。這四項才是AI導入的真正大頭，API費用只是零頭。

---

### 口訣 5：Build / Buy / SaaS —「建買租」

> **建**（Build）→ **買**（Buy）→ **租**（SaaS）
>
> 🗣️ 記法：跟住房一樣——自己蓋（Build）最慢但最客製、買成屋（Buy）次快且可小改、租屋（SaaS）最快最彈性但長期可能較貴。上線速度：**SaaS > Buy > Build**。「核心自建，非核心外租」是導入決策的最高原則。

---

### 口訣 6：POC 成功標準三要素 —「量質退」

> **量**化標準 + **質**化標準 + **退**出標準（Exit Criteria）
>
> 🗣️ 記法：做POC前先講清楚「幾分算通過、主觀感受好不好、什麼情況代表要喊停」，三個標準都定好，POC才不會變無底洞。

---

## Section 6 · 考試陷阱

### 陷阱 ❌1：可以跳過 POC 直接實施

❌ 陷阱：「公司時間緊迫，可以跳過 POC 直接進入實施階段」
✅ 正解：四階段是循序漸進的，**不能跳過驗證（POC）**。POC 的目的就是在全面投資前先小規模驗證可行性。可以「快速 POC」，但不能「省略 POC」。

### 陷阱 ❌2：效能評估在導入前做

❌ 陷阱：「導入AI前，應該先做效能評估，確認預期成效」
✅ 正解：效益/效能評估（Benefit/Performance Evaluation）是**導入後**才做的。導入前做的是技術、工作流程、模型三個面向的評估。「效益面」的核心問題是「AI 有沒有帶來實際價值」，這只能在導入後才能量測。

### 陷阱 ❌3：API 費用 = AI 導入的主要成本

❌ 陷阱：「評估導入成本時，只要計算 API 費用就夠了」
✅ 正解：API 費用只佔 TCO（Total Cost of Ownership，總持有成本）的 5–15%。真正的大頭是**隱藏成本**：員工培訓、系統整合開發、流程改造、資料治理、持續維護等。

### 陷阱 ❌4：AI Agent = 聊天機器人

❌ 陷阱：「AI Agent 就是能自動回覆客戶問題的智慧聊天機器人」
✅ 正解：聊天機器人是被動的「你問我答」，AI Agent 的核心是**自主多步驟執行**——給一個目標，Agent 自己規劃步驟、呼叫工具、持續執行直到完成。關鍵差異：自主性 + 多步驟 + 工具呼叫。

### 陷阱 ❌5：選最強模型就對了

❌ 陷阱：「導入 AI 應該選市場上最準確、最強的模型」
✅ 正解：模型選擇要考慮場景需求的三角取捨——準確度、速度（Latency）、成本。最強的模型通常最貴、最慢，對許多場景來說是過度投資（Overkill）。要選「最適合」的，不是「最強」的。

### 陷阱 ❌6：所有問題都需要 AI 解決

❌ 陷阱：「既然要數位轉型，所有業務流程都應該導入AI」
✅ 正解：《AI導入指引》的**構想評估**第一步，就是判斷「這個問題是否適合 AI 解決」。如果用 Excel 公式或傳統程式就能解決，就不需要花大錢導入 AI。不適合 AI 的場景包括：資料量太少、問題邏輯太簡單、輸出準確率要求 100%。

### 陷阱 ❌7：Build 一定比 SaaS 安全

❌ 陷阱：「自建 AI 系統一定比用 SaaS 更安全，所以都應該自建」
✅ 正解：Build 確實對資料主權控制力最強，但不代表「一定更安全」——自建需要團隊的安全工程能力，否則可能比成熟的 SaaS 供應商更脆弱。應根據組織的技術能力、安全需求綜合評估。

### 陷阱 ❌8：POC 成功 = 可以立即全面上線

❌ 陷阱：「POC 驗證成功，AI 系統立刻全公司上線」
✅ 正解：POC 通過後，還需要進入**實施（Implementation）**階段，包括：部署策略規劃、員工培訓、變更管理、KPI 追蹤系統建立等。POC 驗證的是「技術可行性」，實施階段才是「規模化落地」。

### 陷阱 ❌9：技術評估只看硬體

❌ 陷阱：「技術評估就是確認伺服器和網路夠不夠用」
✅ 正解：技術評估包含三個面向：**基礎設施**（硬體/雲端）、**資料品質**（資料量/品質/格式）、**API 整合**（與現有系統的串接難度）。「人員技術能力」（是否有會開發的工程師）也屬於技術評估的範圍。

### 陷阱 ❌10：只算 API 費用就等於算清楚成本了

❌ 陷阱：「導入AI的成本只要看API費用和授權費就夠了」
✅ 正解：TCO（Total Cost of Ownership，總持有成本）涵蓋的是系統生命週期內的所有成本。隱藏成本（整合開發、員工培訓、持續維護等）往往在系統上線後才陸續浮現，只看 API 費用容易嚴重低估長期投入。初級的重點是「知道隱藏成本存在且遠大於顯性成本」。

### 情境題範例解析

---

**情境 3：** 行銷部導入AI文案生成工具三個月後，主管想知道是否值得繼續投資。應進行哪種評估？

→ **效益面評估（Performance Evaluation）**，因為是導入後追蹤 KPI 和 ROI

---

**情境 4：** 公司要開發一個系統，能自動監控庫存、比較供應商報價、並自動送出採購申請。這個系統的AI形式是什麼？

→ **AI Agent**（需要多步驟、跨系統整合、自主執行）

---

**情境 5：** 財務長問：「導入AI的成本大概多少？」IT 主管說「API 費用大概每月 3 萬元」。財務長應如何回應？

→ **不夠完整**——API 費用只佔 TCO 的 5–15%，還需要考慮員工培訓、系統整合開發、流程改造、持續維護等隱藏成本。

## Section 8 · 結尾：快速自我檢查

完成本課學習後，請確認以下每個項目都能用自己的話說出來：

- [ ] 能按順序說出《AI導入指引》四階段：**構想 → 方案 → 驗證 → 實施**（口訣：構方驗實）
- [ ] 能區分四大評估維度（技工模效）的核心問題，特別是「效益面是導入**後**才做」
- [ ] 知道 AI Agent 與一般 GenAI 工具的三大差異：**自主性、多步驟、工具呼叫**
- [ ] 能辨識隱藏成本（整合開發、員工培訓、流程改造、維護）vs 顯性成本（API、授權費、硬體）
- [ ] 了解 Build / Buy / SaaS 三種路徑的適用情境，不是「一定選哪個」
- [ ] 知道 POC（Proof of Concept，概念驗證）的目的是「小規模驗證技術可行性」，**不能跳過**
- [ ] 理解 TCO（Total Cost of Ownership，總持有成本）概念：隱藏成本佔大頭（85–95%）
- [ ] 能用情境關鍵字快速判斷題目在問哪個概念（參考 Section 7）
