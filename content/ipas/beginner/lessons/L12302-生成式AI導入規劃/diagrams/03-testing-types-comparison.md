## 測試類型記憶法

```
    使用者視角（功能符不符合需求？）
         │
         ▼
      UAT  ←────── 業務需求符合性
      Beta ←────── 早期使用者回饋

    技術/品質視角（系統好不好？）
         │
         ▼
      A/B  ←────── 哪個版本更好
      平行 ←────── 新舊是否一致

    安全/風險視角（系統安不安全？）
         │
         ▼
      紅隊 ←────── 主動攻擊找弱點
```

## 常見混淆點比較

```
┌─────────────────────────────────────────────────────────────────────────┐
│  容易混淆的配對                                                          │
│                                                                         │
│  UAT vs Beta                                                            │
│  • UAT = 內部業務人員驗收（系統「合不合格」）                           │
│  • Beta = 外部使用者體驗（系統「好不好用」）                            │
│                                                                         │
│  A/B Test vs Parallel Run                                               │
│  • A/B = 同時測試兩版本找「最佳方案」（改進導向）                       │
│  • Parallel Run = 同時跑新舊確認「輸出一致」（遷移導向）               │
│                                                                         │
│  Red Team vs 一般 QA                                                    │
│  • Red Team = 模擬敵對攻擊，找安全與倫理弱點                           │
│  • 一般 QA = 確認功能符合規格，不主動找攻擊面                          │
└─────────────────────────────────────────────────────────────────────────┘
```

## 考試重點

- **UAT** = 使用者驗收測試，重點是「使用者」與「驗收」，**不是**技術測試
- **紅隊測試**（Red Team Testing）= AI 特有安全測試，題目中若出現「偏見」「幻覺」「資安漏洞」考的就是這個
- **平行運行**（Parallel Run）= 新舊系統**同時**運行，用於**切換過渡期**，不是優化比較
- **A/B 測試** = 找「哪個版本更好」，需要**統計顯著性**才能下結論
