## Section 3 · 核心概念

### 3.1 四階段推廣策略

**先懂一句話：** 不要一次導入全公司——先小後大，用「POC → Pilot → Rollout → Scale」四階段降低風險。

---

#### 第一階段：POC（Proof of Concept，概念驗證）

**目的：** 技術可行性驗證——這個 AI 在我們的環境裡能跑起來嗎？

| 項目 | 內容 |
|---|---|
| 時間 | 2–4 週 |
| 範圍 | 最小化：1–2 個工程師、1 個特定任務 |
| 資料 | 少量樣本資料（不需完整資料集） |
| 目標 | 技術可行性確認、基礎準確率是否達標 |
| 輸出 | 技術報告、繼續/停止建議 |

🗣️ **白話說明**：就像買電器前先插電試試看，POC 就是「先跑個小實驗，看看這個 AI 是否能解決我們的問題」，不涉及正式上線。

**Quick check：** POC 的主要目的是？ → **驗證技術可行性**（不是業務成效，不是擴大部署）

---

#### 第二階段：Pilot（試行）

**目的：** 在真實環境中，讓特定部門/使用者試用，收集真實回饋。

| 項目 | 內容 |
|---|---|
| 時間 | 1–3 個月 |
| 範圍 | 1–2 個部門、10–50 位使用者 |
| 資料 | 真實業務資料（但非全量） |
| 目標 | 使用者接受度、業務 KPI 初步達標 |
| 輸出 | 使用者回饋報告、KPI 達標評估 |

🗣️ **白話說明**：就像餐廳正式開幕前先做「軟開幕」——先請一小批熟客來試吃，根據回饋調整菜單，才正式對外開放。

**Quick check：** Pilot 與 POC 最大差異？ → **Pilot 用真實使用者在真實環境測試業務效益；POC 只驗證技術可行性**

---

#### 第三階段：Rollout（推廣）

**目的：** 逐步擴大部署範圍，從試行部門推廣到更多部門。

| 項目 | 內容 |
|---|---|
| 時間 | 3–6 個月 |
| 範圍 | 多個部門，按部門優先順序推廣 |
| 重點 | 培訓、技術支援、問題快速解決 |
| 目標 | 無重大故障、SLA（服務等級協議）達標 |
| 輸出 | 部門導入報告、問題清單與解決狀況 |

🗣️ **白話說明**：軟開幕成功後，餐廳開始正式對外營業，一家分店開了再開下一家，每次開店前先確認前一家沒問題。

---

#### 第四階段：Scale（規模化）

**目的：** 全組織/全產品線部署，建立長期維運機制。

| 項目 | 內容 |
|---|---|
| 時間 | 持續進行 |
| 範圍 | 全組織、全產品線 |
| 重點 | 自動化、效能監控、持續優化 |
| 目標 | 穩定效益、可測量的 ROI |
| 輸出 | 組織級 AI 能力、長期維運文件 |

**Quick check：** 四個階段的正確順序？ → **POC → Pilot → Rollout → Scale**

---

### 3.2 各階段出口標準（Exit Criteria）

**先懂一句話：** 每個階段都要「通過才能進下一關」，出口標準就是這個關卡的合格條件。

| 階段 | 出口標準 |
|---|---|
| POC 出口 | ✅ 技術可行性確認 ✅ 基礎準確率達到預設門檻 ✅ 無技術性障礙 |
| Pilot 出口 | ✅ 使用者接受度 > 預設門檻 ✅ 核心 KPI 達標 ✅ 無重大安全問題 |
| Rollout 出口 | ✅ 無重大故障（N 天零重大事故） ✅ SLA（服務等級協議）達標 ✅ 培訓完成率達標 |
| Scale 入場 | ✅ 前三階段全部通過 ✅ 維運人員就位 ✅ 長期預算已核准 |

🗣️ **白話說明**：出口標準就像路考的合格條件——倒車入庫必須一次進，才能拿到駕照。導入 AI 也一樣，每個階段都要符合標準才能繼續。

**Quick check：** Pilot 階段最重要的出口標準是？ → **使用者接受度達標 + 業務 KPI 達標**

---

### 3.3 資源規劃四維度（人錢料時）

**先懂一句話：** 任何 AI 導入計劃都需要四種資源：人力、預算、資料、時間——缺一不可。

#### 人（人力資源）

| 角色 | 負責內容 |
|---|---|
| 內部 IT 部門 | 技術整合、基礎設施、安全控管 |
| 業務部門 | 需求釐清、UAT、日常使用 |
| AI/資料科學團隊 | 模型選型、測試、優化 |
| 外部顧問 | 技術指導、最佳實踐建議 |
| 變更管理專員 | 培訓、溝通、阻力處理 |

#### 錢（預算規劃）

| 費用類別 | 說明 |
|---|---|
| API 費 | 呼叫 LLM API 的成本（依使用量計費） |
| 開發費 | 整合開發、客製化 |
| 培訓費 | 員工教育訓練 |
| 維護費 | 長期維運、更新、技術支援 |
| 緊急預備金 | 建議留 10–20% 緩衝 |

#### 料（資料資源）

| 資料類型 | 用途 |
|---|---|
| 訓練資料 | 訓練/微調模型 |
| 測試資料 | 驗證模型準確性 |
| 評估資料 | 持續監控效能 |
| 基準資料 | 建立比較基準線 |

#### 時（時程規劃）

| 要素 | 說明 |
|---|---|
| 里程碑 | 每個階段的預期完成日期 |
| 相依性管理 | 識別哪些任務必須按順序執行 |
| 緩衝時間 | 每個里程碑建議加 20% 緩衝 |
| Review Gate | 各階段完成後的正式評審節點 |

🗣️ **白話說明**：蓋房子前要確認「有多少工人、多少錢、多少建材、多少時間」——AI 導入也是一樣的邏輯，四種資源換成人錢料時。

**Quick check：** 資源規劃四維度的口訣？ → **人錢料時**

### 3.4 測試類型

**先懂一句話：** 不同測試有不同的目的和執行者，要根據測試目標選擇正確的測試類型。

#### UAT（User Acceptance Testing，使用者驗收測試）

- **執行者**：終端使用者（業務人員，非 IT）
- **目的**：驗證系統是否符合業務需求和使用者期望
- **時機**：Pilot 階段或正式上線前
- **通過標準**：使用者確認「符合預期功能」後簽署驗收

#### A/B 測試（A/B Testing）

- **執行者**：產品/資料團隊
- **目的**：對比兩個版本（新舊系統或不同策略）哪個效果更好
- **時機**：評估優化效果、比較不同提示詞策略
- **輸出**：統計顯著性結論，採用表現更好的版本

#### Beta 測試

- **執行者**：小範圍外部用戶（非公司員工）
- **目的**：在正式發布前收集真實使用者回饋
- **時機**：Rollout 前的最後外部驗證
- **注意**：Beta 用戶通常是自願參與的早期採用者

#### Red Team 測試

- **執行者**：安全/品質專家，以「攻擊者」角度
- **目的**：主動尋找 AI 系統的漏洞、偏見、有害輸出
- **時機**：上線前的安全性驗證，高風險應用必做
- **輸出**：漏洞清單、改善建議

#### 平行運行（Parallel Run）

- **執行者**：業務部門 + IT
- **目的**：新舊系統同時處理真實業務，比較輸出結果
- **時機**：Rollout 過渡期，確保新系統可以替代舊系統
- **優點**：風險最低，有問題立即切回舊系統

🗣️ **白話說明**：
- UAT = 客人來試菜，確認口味對不對
- A/B 測試 = 試試哪個菜單版本讓客人點更多
- Beta 測試 = 請老顧客先試吃新菜
- Red Team = 請挑剔的美食評論家來找缺點
- 平行運行 = 新廚師和老廚師同時做同一道菜，比較結果

**Quick check：** 哪個測試類型是以「攻擊者角度」尋找漏洞？ → **Red Team 測試**

### 3.5 應急規劃（Contingency Planning）

**先懂一句話：** AI 系統可能出問題，事前要準備好「如果壞了怎麼辦」的應對計劃。

#### Fallback（備援）

- **定義**：問題發生時，切換回原有的舊系統繼續運作
- **觸發條件**：AI 系統嚴重故障、效能不符 SLA（服務等級協議）
- **特點**：完全切回「原始狀態」，使用舊有流程/規則
- **範例**：AI 客服機器人當機 → 切回人工客服

#### Rollback（回滾）

- **定義**：將 AI 系統撤回到上一個穩定的版本
- **觸發條件**：最新版本有 bug 或效能退化
- **特點**：AI 系統繼續運作，只是退回舊版本
- **範例**：更新到 v2.0 後效果變差 → 退回 v1.5

#### Kill Switch（緊急關閉）

- **定義**：緊急停用所有 AI 功能
- **觸發條件**：發現嚴重安全問題、法規合規疑慮、重大倫理事件
- **特點**：最強硬的應急手段，通常配合人工接管
- **要求**：必須能在「幾秒內」執行，不需複雜授權

#### HITL（Human-in-the-Loop，人機協作回圈）

- **定義**：在 AI 決策流程中，插入人工審核節點
- **觸發條件**：高風險決策（醫療、法律、財務、人事）
- **特點**：**不是出問題才啟動**，而是事前設計在流程中
- **範例**：AI 審核貸款申請後，人工審核員再確認才放款

🗣️ **白話說明**：
- Fallback = 新車壞了，騎舊車去上班
- Rollback = 手機系統更新後 bug 很多，退回上一版
- Kill Switch = 緊急剎車按鈕
- HITL = 重要文件需要主管親簽，AI 起草人工確認

**Quick check：** HITL 與其他三個有什麼根本差異？ → **HITL 是事前設計在流程中的人工審核；其他三個是出問題後的應急措施**

### 3.6 變更管理（Change Management）

**先懂一句話：** AI 導入最大的阻力往往不是技術問題，而是人的問題——員工習慣改變需要系統性管理。

#### 四個核心要素

| 要素 | 說明 |
|---|---|
| 利害關係人管理（Stakeholder Management） | 識別、分析、影響各方利害關係人（支持者/中立者/反對者） |
| 員工培訓與阻力管理 | 技能培訓 + 心理層面的變革接受度管理 |
| 溝通計劃 | 定期向各層級說明進度、成效、未來計劃 |
| 成效追蹤與回報 | 定期量化 AI 導入效益，向管理層報告 |

🗣️ **白話說明**：就像公司搬辦公室，技術問題（裝網路、搬電腦）只是一部分；更難的是讓大家適應新環境、不抗拒新路線——這就是變更管理在做的事。

### 3.7 敏捷導入 vs 瀑布導入

**先懂一句話：** AI 導入通常推薦敏捷方式，因為 AI 專案天生充滿不確定性，需要快速迭代調整。

| 面向 | 敏捷（Agile） | 瀑布（Waterfall） |
|---|---|---|
| 流程 | 迭代式、循環 | 線性、依序 |
| 調整彈性 | 高：每個 Sprint 都可調整 | 低：需求一旦確認難以更改 |
| 適合情境 | AI 導入、需求不確定、快速變化 | 需求明確、交付物固定 |
| 風險 | 需求蔓延（Scope Creep） | 最後才發現方向錯誤 |
| 回饋循環 | 快：2-4 週一次迭代 | 慢：到最後交付才有完整回饋 |
| AI 導入建議 | ✅ 推薦 | 只適用需求完全明確的子項目 |

🗣️ **白話說明**：做 AI 項目就像做料理研發——你不知道客人會不會喜歡新口味，所以要先做小份讓少數人試吃再調整（敏捷），而不是一次做 1000 份才發現大家不喜歡（瀑布）。

**Quick check：** 為什麼 AI 導入推薦敏捷而非瀑布？ → **AI 專案存在不確定性，需要快速迭代調整方向**

## Section 4 · 易混淆概念對照表

### 4.1 四個應急機制比較

| 機制 | 觸發時機 | 對象 | 結果 |
|---|---|---|---|
| **Fallback（備援）** | AI 系統嚴重故障 | 整個 AI 系統 | 切回原有舊系統，AI 停用 |
| **Rollback（回滾）** | 最新版本有問題 | 特定 AI 版本 | 退回上一個穩定版本，AI 繼續跑 |
| **Kill Switch（緊急關閉）** | 緊急安全/倫理問題 | 整個 AI 功能 | 立即完全停用，人工接管 |
| **HITL（人機協作回圈）** | 高風險決策（事前設計） | 特定決策節點 | AI 輸出後人工審核再執行 |

**考題判斷：**
- 「系統更新後出現 bug，退版」→ **Rollback**（退回舊版本）
- 「AI 完全壞掉無法運作」→ **Fallback**（切回人工/舊系統）
- 「發現 AI 輸出帶有歧視性內容」→ **Kill Switch**（緊急停用）
- 「貸款審核 AI 建議後需人工確認才放款」→ **HITL**（流程設計，非應急）

---

### 4.2 五種測試類型比較

| 測試 | 執行者 | 目的 | 最佳時機 |
|---|---|---|---|
| **UAT** | 終端使用者（業務人員） | 確認符合業務需求 | Pilot 階段 |
| **A/B 測試** | 產品/資料團隊 | 對比兩個版本效果 | 優化決策時 |
| **Beta 測試** | 外部小範圍用戶 | 收集外部真實回饋 | Rollout 前 |
| **Red Team** | 安全/品質專家 | 尋找漏洞與有害輸出 | 上線前安全驗證 |
| **平行運行** | 業務 + IT | 新舊系統對比輸出 | Rollout 過渡期 |

**考題判斷：**
- 「讓業務同仁測試看是否符合工作需求」→ **UAT**
- 「同時跑舊系統和新 AI，比較結果」→ **平行運行**
- 「以駭客角度找 AI 的漏洞」→ **Red Team 測試**
- 「測試新版提示詞是否比舊版效果更好」→ **A/B 測試**

---

### 4.3 POC vs Pilot 比較

| 面向 | POC（概念驗證） | Pilot（試行） |
|---|---|---|
| 核心問題 | 「技術上能不能做到？」 | 「業務上有沒有效果？」 |
| 執行者 | 工程師/技術團隊 | 業務使用者 |
| 環境 | 實驗室環境 | 真實業務環境 |
| 資料 | 樣本資料 | 真實業務資料 |
| 時間 | 2–4 週 | 1–3 個月 |
| 輸出 | 技術可行性報告 | 使用者接受度 + KPI 報告 |

**考題判斷：**
- 「確認 AI 模型在公司環境可以正常運作」→ **POC**
- 「業務同仁使用 AI 一個月，評估是否提升效率」→ **Pilot**

---

### 4.4 敏捷 vs 瀑布 in AI 導入

| 面向 | 敏捷 | 瀑布 |
|---|---|---|
| 變更能力 | 隨時可調整 | 需求確定後難以改變 |
| 反饋速度 | 快（每 2-4 週） | 慢（完成後才知道） |
| AI 不確定性適應 | ✅ 好 | ❌ 差 |
| 規劃精確性要求 | 低（迭代調整） | 高（需事前規劃完整） |

**考題判斷：**
- AI 導入為何推薦敏捷？→ **AI 專案充滿不確定性，需要快速迭代調整**
- 哪種情況適合瀑布？→ **需求完全明確、交付物固定的子項目**

---

### 4.5 Rollout 四階段 vs 三階段（常見題型陷阱）

部分題目使用三階段（POC→Pilot→Scale），部分使用四階段（POC→Pilot→Rollout→Scale）。

| 表述方式 | 說明 |
|---|---|
| 三階段：POC→Pilot→Scale | Scale 涵蓋了 Rollout + 規模化，是簡化版本 |
| 四階段：POC→Pilot→Rollout→Scale | Rollout 單獨列出，更精確描述逐步推廣 |
| 考試應對 | 看題目用哪種版本；核心邏輯「從小到大」不變 |

## Section 5 · 口訣

### 口訣一：四階段「小試大推」

> **小（POC）試（Pilot）大（Rollout）推（Scale）**
>
> 小 = 小範圍技術驗證 → 試 = 試用測業務效益 → 大 = 大範圍推廣 → 推 = 推到全組織

### 口訣二：資源規劃「人錢料時」

> **人錢料時，缺一不可**
>
> 人（人力）→ 錢（預算）→ 料（資料）→ 時（時程）

### 口訣三：應急機制「切退停審」

> **切（Fallback 切舊系統）退（Rollback 退舊版本）停（Kill Switch 緊急停用）審（HITL 人工審核）**
>
> 記憶鉤子：出了大事的 SOP「切換→退版→停機→審核」

### 口訣四：測試五類「驗比外攻並」

> **驗（UAT 驗收）比（A/B 比較）外（Beta 外部）攻（Red Team 攻擊）並（Parallel Run 並行）**

### 口訣五：出口標準三組合

> POC 出口：**技術可行 + 準確率達標**
>
> Pilot 出口：**使用者接受 + KPI 達標**
>
> Rollout 出口：**無重大故障 + SLA 達標**

### 口訣六：HITL 記憶點

> **「高風險決策，人工必審」** — 醫療/金融/法律/人事 → HITL 事前設計，不是應急

---

## Section 6 · 考試陷阱

❌ **陷阱 1：把 POC 和 Pilot 搞混**
> ❌ 「POC 是讓業務同仁實際使用評估效益」
> ✅ POC 是**技術團隊驗證可行性**；Pilot 才是業務使用者測試業務效益

❌ **陷阱 2：把 Rollback 誤解為切回舊系統**
> ❌ 「發現 bug 後 Rollback，切換回舊的人工流程」
> ✅ Rollback 是**退回 AI 系統的上一個版本**；Fallback 才是切回原有舊系統/人工流程

❌ **陷阱 3：HITL 當作應急措施**
> ❌ 「AI 出問題時啟動 HITL 應急」
> ✅ HITL 是**事前設計在流程中的人工審核節點**，不是應急措施

❌ **陷阱 4：Red Team 測試的執行者搞錯**
> ❌ 「Red Team 測試由業務使用者執行，模擬真實使用情境」
> ✅ Red Team 測試由**安全/品質專家**以攻擊者角度執行，找漏洞和偏見

❌ **陷阱 5：平行運行（Parallel Run）等同 A/B 測試**
> ❌ 「平行運行是 A/B 測試的另一種說法」
> ✅ 平行運行是**新舊系統同時處理同一批真實業務**；A/B 測試是**分流不同用戶到不同版本比較效果**

❌ **陷阱 6：把 Rollout 放在 Scale 之後**
> ❌ 「POC → Pilot → Scale → Rollout」
> ✅ 正確四階段順序：**POC → Pilot → Rollout → Scale**

❌ **陷阱 7：資源規劃只考慮預算**
> ❌ 「資源規劃主要是做預算估算」
> ✅ 資源規劃有**四個維度：人、錢、料（資料）、時（時程）**，缺一不可

❌ **陷阱 8：敏捷就是「沒有規劃」**
> ❌ 「AI 導入用敏捷方式，所以不需要事前規劃」
> ✅ 敏捷是**迭代式規劃**，每個 Sprint 都有計劃，只是允許彈性調整

❌ **陷阱 9：Beta 測試等於 UAT**
> ❌ 「Beta 測試就是讓員工做使用者驗收測試」
> ✅ Beta 測試是**外部小範圍用戶**（非公司員工）的測試；UAT 是**內部終端使用者**的驗收

❌ **陷阱 10：Kill Switch 需要複雜授權程序**
> ❌ 「Kill Switch 需要高層核准才能觸發，流程至少兩天」
> ✅ Kill Switch 的設計要求是**幾秒內即可執行**，不需要複雜授權，緊急情況立刻啟動

## Section 7 · 情境題快速判斷

### 判斷流程圖

```
收到情境題
    │
    ▼
涉及哪個核心概念？
    │
    ├──→ 導入了幾個月/幾週？ 誰在測試？
    │       ├── 工程師 + 技術驗證 + 2-4週 → POC
    │       ├── 業務使用者 + 真實環境 + 1-3個月 → Pilot
    │       ├── 多部門逐步擴大 + 3-6個月 → Rollout
    │       └── 全組織 + 長期維運 → Scale
    │
    ├──→ 出了問題要怎麼處理？
    │       ├── 新版本有bug → Rollback（退版）
    │       ├── 整個AI壞掉/無法運作 → Fallback（切舊系統）
    │       ├── 緊急安全/倫理問題 → Kill Switch
    │       └── 高風險決策需要人審 → HITL（事前設計，非應急）
    │
    ├──→ 要做什麼測試？
    │       ├── 業務人員確認功能 → UAT
    │       ├── 比較兩個版本優劣 → A/B 測試
    │       ├── 外部小範圍用戶試用 → Beta 測試
    │       ├── 找安全漏洞/偏見 → Red Team
    │       └── 新舊系統同時跑比較 → 平行運行
    │
    └──→ 資源規劃缺了什麼？
            ├── 找不到人做 → 人（人力）
            ├── 預算不夠 → 錢（預算）
            ├── 沒有訓練資料 → 料（資料）
            └── 時程不合理 → 時（時程）
```

### 關鍵字快速對應表

| 關鍵字/情境描述 | 對應概念 |
|---|---|
| 技術可行性驗證、工程師實驗、樣本資料 | POC |
| 業務使用者、真實環境、一個月、KPI | Pilot |
| 逐步擴大到更多部門、SLA 達標 | Rollout |
| 全組織部署、長期維運、標準化 | Scale |
| 業務人員確認功能符合需求 | UAT |
| 新舊系統同時處理業務、過渡期 | 平行運行（Parallel Run） |
| 以攻擊者角度找漏洞、Prompt Injection | Red Team 測試 |
| 發現版本更新有問題，退版 | Rollback |
| AI 完全故障，回到人工流程 | Fallback |
| 緊急安全或倫理疑慮，立即停用 | Kill Switch |
| 高風險決策插入人工審核（事前設計） | HITL |
| 迭代式、快速調整、Sprint | 敏捷（Agile） |
| 線性、需求明確、依序執行 | 瀑布（Waterfall） |

### 情境題演練

**情境 1：**
> 某公司 AI 客服系統更新到 v3.0 版本後，發現誤判率明顯上升，IT 部門決定暫時回到 v2.5 版本，等問題修復後再升級。
>
> 這個動作是什麼應急措施？

**答：Rollback（回滾）** — 退回 AI 系統的上一個穩定版本，AI 持續運作。

---

**情境 2：**
> 某銀行導入 AI 信貸審核系統，系統設計規定：當 AI 評估為「邊界案件」（分數在 60–70 分之間），必須交由信貸專員做最終核准才能放款。
>
> 這個設計對應哪個概念？

**答：HITL（Human-in-the-Loop，人機協作回圈）** — 事前設計在決策流程中的人工審核節點，不是應急措施。

---

**情境 3：**
> 某公司計劃導入 AI 文件摘要工具。第一步：由 2 位工程師用 100 份樣本文件測試 AI 模型能否正確摘要，耗時 3 週。
>
> 這個階段對應哪個導入階段？

**答：POC（Proof of Concept，概念驗證）** — 工程師 + 樣本資料 + 技術可行性驗證。

---

**情境 4：**
> 某公司的 AI 推薦系統被發現會對特定族群產生歧視性推薦，管理層決定立即停用所有 AI 推薦功能，改回人工挑選。
>
> 這個動作對應什麼？

**答：Kill Switch（緊急關閉）** — 因為倫理問題緊急停用 AI 功能，配合人工接管。

---

**情境 5：**
> 某電商平台正在評估更換新版本的商品描述 AI，讓 10% 的用戶看到新版 AI 描述，另外 90% 看到原版，比較點擊率和轉換率的差異。
>
> 這個測試是什麼類型？

**答：A/B 測試** — 分流不同比例的用戶到不同版本，比較效果差異。
