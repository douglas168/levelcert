## 四件套功能對比

### 件套一：Guardrails（護欄）

```
┌─────────────────────────────────────────────────────────┐
│                    Guardrails 護欄                       │
│                                                         │
│  位置：模型外層（不修改模型本身！）                      │
│                                                         │
│  輸入端（Pre-processing）：                             │
│    ✓ 過濾敏感詞/惡意指令                                │
│    ✓ 偵測 Prompt Injection 模式                         │
│    ✓ 禁止輸入 PII（姓名/電話/身分證）                   │
│    ✓ 限制主題範圍（如：只回答客服相關）                  │
│                                                         │
│  輸出端（Post-processing）：                             │
│    ✓ 偵測並過濾 PII 洩漏                                │
│    ✓ 強制添加 AI 生成標示                               │
│    ✓ 攔截違規/有害輸出                                   │
│                                                         │
│  Planner 職責：定義哪些主題/資料類型需要過濾             │
└─────────────────────────────────────────────────────────┘
```

**考試重點：** Guardrails 是在**模型外層**，不是修改模型權重。

---

### 件套二：HITL — Human-in-the-Loop（人機協作回圈）

```
正常低風險流程：
使用者 → LLM → Guardrails → 自動輸出 ✓

高風險情境流程（需 HITL）：
                        ┌──────────────────────┐
使用者 → LLM → 輸出草稿 → │  HITL 介入點         │
                        │  人工審核員           │
                        │  ├── 批准 → 輸出     │
                        │  ├── 修正 → 再輸出   │
                        │  └── 拒絕 → 退回     │
                        └──────────────────────┘

必須設 HITL 的場景：
  🏥 醫療診斷建議      （錯了影響生命）
  💰 金融信貸決策      （歧視/拒絕理由）
  👥 人資篩選決定      （職業歧視）
  📋 法律文件草擬      （幻覺引用假法條）
  🏛️ 政府對外公文      （行政院指引要求）
```

**考試重點：** HITL = 事前/即時介入，不是純事後稽核（Post-hoc Audit）。

### 件套三：Red Team（紅隊演練）

```
時間軸：

上線前（Pre-deployment）
  │
  ├── Red Team 階段 ────────────────────────────┐
  │     · 模擬攻擊者行為                         │
  │     · 嘗試各種 Prompt Injection              │
  │     · 測試越獄（Jailbreak）可能性            │
  │     · 評估偏見與公平性                       │
  │     · 輸出：漏洞報告 + 修補建議              │
  │                                              │
  ├── 根據報告修補 ◄───────────────────────────┘
  │
上線後（Post-deployment）
  │
  ├── 定期 Red Team 再評估
  │
  └── 發現新漏洞 → 修補 → 記錄

Planner 職責：
  · 編列 Red Team 預算與時程
  · 決定是否找外部顧問
  · 要求記錄測試結果與修補歷程
```

**考試重點：** Red Team 是**主動找漏洞**（事前），不是事後修復（那是 Incident Response）。

---

### 件套四：Incident Response（事件應變計畫）

```
AI 事故發生
      │
      ▼
  偵測/通報
  ├── 監控告警觸發（自動）
  └── 使用者/員工回報（手動）
      │
      ▼
  初步評估（Triage）
  ├── 影響範圍有多大？
  ├── 是否涉及個資外洩？
  └── 需要立即下線嗎？
      │
      ▼
  控制止血
  ├── 暫停受影響功能
  ├── 通知相關部門
  └── 保全證據
      │
      ▼
  對外溝通
  ├── 使用者/客戶通知
  ├── 主管機關通報（若涉及個資洩漏→個資法 72 小時）
  └── 媒體/公關回應
      │
      ▼
  根本原因分析 + 修補 + 紀錄

Planner 職責：
  · 事前制定 SOP 文件
  · 指定負責人（RACI）
  · 定期演練
```

## 四件套 vs. 三法 整合應用

```
高風險 AI 應用（如：醫療診斷 AI）

  Guardrails ─────────────────────────────────► 個資法 §27
  （禁止輸入病患身份資訊）                    安全維護義務

  HITL ────────────────────────────────────────► 人工智慧基本法
  （醫師必須在最終診斷前介入覆核）             七大原則：人類自主

  Red Team ───────────────────────────────────► 人工智慧基本法 §16
  （事前找出偏見/幻覺漏洞）                   高風險 AI 事前評估

  AI 標示 + 審計紀錄 ──────────────────────────► 人工智慧基本法
  （明確告知「本建議由 AI 輔助生成」）         七大原則：透明與可解釋

  訓練資料授權檢查 ────────────────────────────► 著作權法
  （醫學文獻/病歷訓練資料是否有授權）          重製需授權或合理使用
```

## 考試快速對應

| 考題關鍵字 | 對應工具/法規 |
|---|---|
| 「限制 AI 回答主題」「過濾輸入」「攔截輸出」| Guardrails（護欄）|
| 「高風險決策必須人審核」「事前介入」| HITL（人機協作回圈）|
| 「事前找漏洞」「主動攻擊測試」| Red Team（紅隊演練）|
| 「出事後通報」「止血流程」「對外溝通」| Incident Response（事件應變）|
| 「高風險 AI 分級」「moda 負責」| 人工智慧基本法 §16 |
| 「個資輸入 prompt」「§27」| 個資法（Personal Data Protection Act）|
| 「AI 產出著作權」「純 AI 不保護」| 著作權法 + 智慧局 2023-06 函釋 |
