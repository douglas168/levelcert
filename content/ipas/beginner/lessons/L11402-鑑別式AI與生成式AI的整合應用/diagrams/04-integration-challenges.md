# 圖解 04：整合挑戰與緩解策略（Integration Challenges）

> 對應 Study Guide Section 3.8

---

## 五大整合挑戰總覽

```
整合AI系統面臨的挑戰
│
├─ 1. 資料流設計（Data Flow Design）
│   問題：各模組的輸入/輸出格式不對齊
│   後果：系統無法正常串接，出現解析錯誤
│
├─ 2. 延遲（Latency）
│   問題：生成式模型推理時間長，串聯時累加
│   後果：整體系統回應變慢，使用者體驗差
│
├─ 3. 幻覺（Hallucination）
│   問題：生成式AI可能產生不準確或捏造的內容
│   後果：整合輸出包含錯誤資訊，影響決策
│
├─ 4. 評估困難（Evaluation Difficulty）
│   問題：端對端評估 vs 各模組分開評估
│   後果：難以找出效能瓶頸或錯誤來源
│
└─ 5. 成本管理（Cost Management）
    問題：生成式AI的Token費用隨用量累積
    後果：串聯多步驟 = 多倍Token費用，成本飆升
```

---

## 詳細說明與緩解策略

### 挑戰 1：資料流設計問題

```
  問題情境：
  ┌──────────────────┐          ┌──────────────────┐
  │ 鑑別式AI 輸出    │  ──→     │ 生成式AI 輸入    │
  │ {fraud: 0.87,   │   格式    │ 只接受純文字     │
  │  type: "card"}  │   不對齊！ │                  │
  └──────────────────┘          └──────────────────┘
          ↓ 結果：系統無法解析，報錯或輸出錯誤

  緩解方向（概念層次）：
  ・設計清楚的模組介面規範（輸入輸出格式協議）
  ・使用結構化輸出（JSON）讓下游模組容易解析
  ・透過提示工程要求生成式輸出固定格式
```

### 挑戰 2：延遲問題

```
  串聯管線的延遲是累加型的：

  鑑別式AI    生成式AI     總延遲
  ┌─────┐    ┌─────┐
  │ 1秒  │ →  │ 4秒  │   = 5秒
  └─────┘    └─────┘

  步驟越多，延遲越長：
  鑑別1秒 → 生成4秒 → 鑑別2秒 = 7秒

  ⚠️ 生成式AI是最大的延遲瓶頸（推理計算量大）

  並聯模式可降低整體延遲：
  ┌─────┐
  │鑑別1秒│ ↘       ≈ max(1秒, 4秒) = 4秒
  └─────┘   融合
  ┌─────┐ ↗
  │生成4秒│
  └─────┘
  （但並聯要求兩個任務互不依賴才可行）
```

### 挑戰 3：幻覺問題

```
  幻覺（Hallucination）= 生成式AI捏造不存在的內容

  高風險場景：
  ┌────────────────────────────────────────────┐
  │ 醫療 AI 系統：生成假的用藥劑量 → 危害病患  │ ⚠️ 高風險
  │ 法律 AI 系統：引用不存在的法條 → 誤導判決  │ ⚠️ 高風險
  │ 客服 AI 系統：聲稱不存在的優惠 → 商業糾紛  │
  └────────────────────────────────────────────┘

  降低策略：
  ・使用 RAG：基於知識庫文件回答，而非憑記憶
  ・加入鑑別式審核層：輸出前自動事實查核
  ・人工確認：高風險場景一律要求人工複核

  ⚠️ 注意：RAG 能「降低」但無法「消除」幻覺
```

### 挑戰 4：評估困難

```
  端對端評估：測試整個系統的最終輸出品質
  優點：反映真實使用效果
  缺點：難以定位出錯在哪個模組
       ↓
  ┌──────────┐ → ┌──────────┐ → ┌──────────┐
  │ 鑑別模組  │   │ 橋接層   │   │ 生成模組  │
  └──────────┘   └──────────┘   └──────────┘
       ↑              ↑              ↑
  最終出錯時，不清楚是哪一個模組的問題

  模組分開評估：分別測試每個模組的效能
  優點：精準找出瓶頸
  缺點：各模組單獨好不代表組合後一定好
```

### 挑戰 5：成本管理

```
  生成式AI 依 Token 數量計費：

  單步驟：提問(100 tokens) + 回覆(200 tokens) = 300 tokens

  串聯三步驟（每步包含前一步輸出）：
  步驟1：300 tokens
  步驟2：300 + 200（步驟1結果）= 500 tokens
  步驟3：500 + 300（步驟2結果）= 800 tokens
  ──────────────────────────────────────────
  總計：1600 tokens（比單步驟貴5倍以上）

  ⚠️ 使用 RAG 也會增加費用（檢索到的文件也算 Token）
```

## 考試速記：五大挑戰

```
  口訣：「資延幻評成」

  資 = 資料流設計（格式對齊）
  延 = 延遲（串聯累加，生成式是瓶頸）
  幻 = 幻覺（生成式可能捏造，RAG 降低但不消除）
  評 = 評估困難（端對端 vs 模組分開）
  成 = 成本管理（Token 費用隨步驟增加）
```

| 挑戰 | 哪種模式影響最大 | 考試考點 |
|------|---------------|---------|
| 資料流設計 | 串聯（格式依賴強）| 介面不對齊→系統報錯 |
| 延遲 | 串聯（累加）| 生成式是瓶頸；並聯延遲取 max |
| 幻覺 | 含生成式的所有整合 | RAG降低不消除；醫療場景風險最高 |
| 評估困難 | 多模組整合 | 模組分開好≠整體好 |
| 成本 | 含多步生成式的串聯 | 步驟越多Token費用越高 |
