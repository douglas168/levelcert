## Vendor Lock-in 不只是成本問題

```
考試陷阱：「Vendor Lock-in 就是換平台要花錢，只要預算夠就能解決」
                              ❌ 錯誤
                              │
                              ▼
實際影響是多維度的：

  直接遷移成本 ──────────────── 這是最表面的影響
        │
  技術策略自主性喪失 ─────────── 被鎖定後，議價籌碼大幅降低
        │                       供應商可任意調漲，企業無談判空間
  資料格式不相容風險 ─────────── 遷移時資料可能無法完整轉換
        │
  服務中斷風險 ──────────────── 遷移期間業務無法正常運作
        │
  重新鎖定風險 ──────────────── 換到新平台後，可能又被鎖定
```

## 降低 Vendor Lock-in 風險的策略

| 策略 | 說明 |
|---|---|
| 選擇支援資料匯出的平台 | 評估平台是否支援標準格式（CSV、JSON、SQL）資料匯出 |
| 使用開放標準 API | 選擇遵循 REST/OpenAPI 標準而非專有 API 的平台 |
| 建立抽象層（Abstraction Layer） | 在自建邏輯與平台之間加入一層隔離，降低直接依賴 |
| 定期評估平台依賴程度 | 每年審查，若依賴過深則評估漸進式遷移計畫 |

> 💡 **考試要點**：Vendor Lock-in 的本質是「技術遷移困難」（技術問題），不只是費用問題。成因是**平台封閉設計**（非使用者選擇）。
