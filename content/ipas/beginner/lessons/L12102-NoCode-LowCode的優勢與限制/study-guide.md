## Section 3 · 核心概念

### 3.1 快速開發（Rapid Development）

**先懂一句話**：NC/LC 把原型開發週期從月縮短到天或週，是最被引用的核心優勢。

傳統開發需要「需求→架構→編碼→測試→部署」的完整週期，動輒 3–6 個月。NC/LC 平台透過視覺化拖放（Drag-and-Drop）、元件複用（Reusable Components）與內建自動化測試，大幅壓縮開發時程。Forrester 研究指出，LCNC 開發速度比傳統全程式碼（Full-Code）開發快約 **20 倍**。

```
傳統開發 vs LCNC 時程對比

傳統開發（Full-Code）
┌────────────────────────────────────────────────────┐
│ 需求 → 架構設計 → 編碼 → 測試 → 部署  ≈ 3–6 個月    │
└────────────────────────────────────────────────────┘

LCNC 開發
┌──────────────────────────────────┐
│ 需求 → 拖放設定 → 測試 → 部署      │ ≈ 數天至數週
└──────────────────────────────────┘
```

🗣️ **白話說明**：你要辦大學社團報名——傳統做法是找資工系同學從零寫系統，可能兩週；用 Google AppSheet 或 Airtable，你午餐時間就能搞定。這就是 20 倍速差距的真實感受。

**Quick check**：「開發速度」優勢的關鍵詞是什麼？→ 「快速原型」「MVP（Minimum Viable Product，最小可行產品）」「20 倍」「Forrester」

---

### 3.2 降低成本（Cost Reduction）

**先懂一句話**：LCNC 不只省人力，而是壓縮了整個開發生命週期（Development Lifecycle）的成本。

| 成本面向 | 傳統開發 | LCNC 開發 |
|---|---|---|
| 人力成本 | 需聘專業工程師團隊 | 業務人員可自行開發 |
| 時間成本 | 3–6 個月 | 數天至數週 |
| 維護成本 | 團隊自維伺服器與程式碼 | 平台負責底層基礎架構 |
| 試錯成本 | 高（失敗 = 大量沉沒成本） | 低（快速迭代、快速放棄） |

🗣️ **白話說明**：台灣 Junior 工程師月薪 4–5 萬，Senior 8–10 萬。一個小型請假系統傳統做法要寫三個月，光薪資就 12–30 萬；用 Power Apps，有 Excel 基礎的人資同事花一週搞定，只需平台訂閱費。

**Quick check**：TCO（Total Cost of Ownership，總持有成本）概念為何重要？→ LCNC 的低前期費用若加計長期授權費與技術債修復成本，TCO 可能反而不比傳統開發低。

---

### 3.3 賦能業務人員（Business Empowerment）

**先懂一句話**：LCNC 讓業務專家直接建構工具，IT 部門從「全包」轉型為「治理 + 賦能」。

Gartner 預測 2026 年 **80%** 的 LCNC 使用者將來自正式 IT 部門以外，反映公民開發者（Citizen Developer）趨勢。效益在於業務部門自主解決需求，不需等 IT 排程（Development Backlog）。

```
需求解決路徑對比

業務部門發現問題
    │
    ├─ 傳統路徑：提需求 → 排隊等 IT → 3 個月後交付
    │
    └─ LCNC 路徑：用核可工具自建 → 1 週搞定 ✅
```

🗣️ **白話說明**：每次要做客戶追蹤表都要提 IT 需求單，可能兩個月後才輪到你。有了 LCNC，你自己用 Airtable 一個下午搞定。

**Quick check**：公民開發者與影子 IT 的差異在哪裡？→ **有無 IT 核可**。核可 = 公民開發者（正面），未核可 = 影子 IT（風險）。

---

### 3.4 易於維護（Ease of Maintenance）

**先懂一句話**：視覺化設計介面讓非技術人員也能修改應用，平台負責底層更新。

NC/LC 平台提供視覺化流程設計器（Visual Workflow Designer），功能修改無需更動程式碼；平台供應商負責底層基礎架構升級與安全更新，降低維護複雜度。

🗣️ **白話說明**：就像 Canva 改版型，拖一拖就好；不像傳統開發要改 CSS、重測、重部署。

**Quick check**：易維護的反面是什麼？→ 技術債（Technical Debt）——一旦快速疊加功能卻不重構，「易維護」就變成「難維護」的反面教材。

---

### 3.5 加速數位轉型（Digital Transformation Acceleration）

**先懂一句話**：LCNC 降低技術門檻，讓更多企業能快速展開數位化。

台灣 43% 企業領導者認為數位轉型最大障礙是技術人才不足（數位時代報導）。LCNC 填補了人才缺口——不需搶工程師，讓現有員工直接動手數位化。同時，LCNC + 生成式 AI（Generative AI）整合正成為新趨勢：用自然語言描述需求，AI 自動產生 App 介面（如 Power Apps Copilot、AppSheet + Gemini）。

**Quick check**：LCNC + AI 是否代表不需要人類參與？→ **否**。AI 降低技術門檻，但業務需求判斷、資料品質把關、AI 偏差監督仍需人工。

---

### 3.6 供應商鎖定（Vendor Lock-in）⚠️ 🔥🔥🔥

**先懂一句話**：深度依賴 NC/LC 平台後，無法輕易將應用遷移到其他平台或自建環境，因為平台通常不允許匯出原始碼。

定義：企業應用的邏輯、資料結構、介面設計都綁在特定平台的專有格式（Proprietary Format）中。多數 NC 平台無法匯出原始碼；部分 LC 平台雖允許匯出，產出程式碼仍高度依賴原廠套件，遷移成本依然極高。

```
Vendor Lock-in 困境

Day 1：開心上線              Day 365：想換平台
┌───────────────┐           ┌─────────────────┐
│ 選了平台 A    │           │ 平台 A 突然漲價  │
│ 快速開發 ✅   │           │ 想搬到平台 B    │
│ 方便又划算    │           └────────┬────────┘
└───────────────┘                    │
                               無法匯出原始碼
                                     │
                                     ▼
                           ┌─────────────────┐
                           │ 遷移 = 從頭重寫  │
                           │ 💸 巨額成本 + 停機 │
                           └─────────────────┘
```

🗣️ **白話說明**：就像把所有照片存在某個雲端相簿裡，三年後對方漲價，想搬家卻發現照片格式是專利格式、匯不出原檔。LCNC 的 Vendor Lock-in 就是這種感覺——進去容易，出來很痛。

Vendor Lock-in 的影響**不只是成本**，更包括：
- 技術策略自主性（Strategic Autonomy）喪失
- 資料格式不相容的遷移風險
- 服務中斷（遷移期間的業務影響）
- 可能換到新平台後又產生新的鎖定

**Quick check**：Vendor Lock-in 是誰造成的？→ **平台本身的封閉設計**（Proprietary Format），不是使用者的決定。

### 3.7 影子 IT 風險（Shadow IT Risk）⚠️ 🔥🔥🔥

**先懂一句話**：LCNC 工具門檻低，特別容易催生「影子 IT」——非 IT 部門未經核可自建的系統。

影子 IT（Shadow IT）是指**組織中未經 IT 部門知情或核可的技術系統與應用**。風險包括：

```
影子 IT 形成路徑與後果

業務部門覺得好用
      │
      ▼
自行註冊 NC/LC 帳號
      │
      ▼
建了一堆內部工具（IT 完全不知道）
      │
      ▼
┌─────────────────────────────────┐
│ 風險一：資料安全漏洞              │
│ 風險二：不符合法規合規（個資法等） │
│ 風險三：系統間無法整合            │
│ 風險四：員工離職後無人維護        │
└─────────────────────────────────┘
```

🗣️ **白話說明**：大學社長自己辦了 Google Form 收社費，把大家銀行帳號存在自己 Sheets 裡。學校資安中心完全不知道這件事。如果 Sheets 權限沒設好，個資就外洩了。公司裡的影子 IT 就是這個情況的放大版。

> ⚠️ **考試重要**：影子 IT 是**風險**，不是優勢。看到「IT 不知情」「未經核可」「員工自行建置」，答案指向 Shadow IT。

**Quick check**：影子 IT 的解法是什麼？→ 建立核准工具清單 + 公民開發者制度 + 定期審查，「疏導」而非「圍堵」。

---

### 3.8 效能限制（Performance Limitations）

**先懂一句話**：大量資料或高并發（高 Concurrent Users）場景下，LCNC 平台效能成為瓶頸。

NC/LC 平台的視覺化流程設計器難以表達深層巢狀邏輯（Nested Logic）；當同時上線人數暴增，平台效能可能無法承受。這是 LCNC 平台的**擴展性瓶頸（Scalability Bottleneck）**。

🗣️ **白話說明**：用 Canva 做一張 IG 貼文沒問題，但要做電影特效，Canva 就力不從心了。LCNC 做報名表單很棒，但要做像蝦皮雙 11 同時百萬人搶購的系統，就超出能力範圍。

**Quick check**：Zapier 的真正限制是什麼？→ **複雜邏輯有限制**（不是「不支援 API」——Zapier 就是靠 API 串接服務的！）

---

### 3.9 資安隱私疑慮（Security & Privacy Risks）

**先懂一句話**：資料存於第三方雲端平台，涉及 GDPR（General Data Protection Regulation，一般資料保護規則）、台灣個資法的合規風險，高敏感產業（金融/醫療）需特別謹慎。

主要疑慮：
- **資料主權（Data Residency）**：全球性 SaaS NC/LC 平台未必在台灣設有伺服器
- **SLA（Service Level Agreement，服務等級協議）差異**：平台保障的可用性與傳統自建系統不同
- **合規認證需求**：ISO 27001、SOC 2 等認證需確認平台是否持有

🗣️ **白話說明**：把公司客戶資料存在美國的雲端服務，若台灣金管會要求資料落地境內，就直接違規了。

---

### 3.10 技術債（Technical Debt）

**先懂一句話**：為了短期快速交付而犧牲長期架構品質，未來修改成本越來越高。

形成路徑：
1. 用 NC/LC 快速做出 MVP → 上線成功
2. 需求持續增加 → 在原有基礎上不斷疊加功能
3. 從未重構（Refactoring）→ 系統越來越混亂
4. 最終：修一個 bug 產生三個新 bug → 維護成本飆升

🗣️ **白話說明**：大學報告第一版趕出來，之後老師改需求就在上面東補西補，到第五版已經亂七八糟、自己都看不懂。LCNC 的技術債就是「一直東拼西湊卻從不整理」的後果。

**Quick check**：技術債是誰的問題？→ **使用者的開發行為**（快速疊加功能但不重構），不是平台本身的問題。

---

### 3.11 客製化受限（Customization Constraints）

**先懂一句話**：平台模板和視覺化工具難以實現超出設計範圍的複雜邏輯。

NC 平台：受限於模板與內建元件，無法實作超出平台預設能力的邏輯。LC 平台：雖可加入少量自訂程式碼，但深度客製化仍受平台架構約束。

🗣️ **白話說明**：樂高積木可以拼出很多形狀，但沒辦法拼出積木設計師沒想到的零件。LCNC 就是那盒積木——超出盒子規格就做不到。

---

### 3.12 NC/LC 適用評估框架（Evaluation Framework）

**先懂一句話**：選 NC、LC 還是傳統開發，要從四個維度評估，沒有絕對答案。

| 評估維度 | NC 適合 | LC 適合 | 傳統開發適合 |
|---|---|---|---|
| **任務複雜度** | 簡單（表單、自動化） | 中等（企業流程） | 複雜（核心演算法） |
| **安全敏感度** | 低（內部工具） | 中高（可加程式碼合規控制） | 極高（金融核心交易） |
| **規模化需求** | 低（小型內部工具） | 中（企業級應用） | 高（高并發、大規模） |
| **ROI 分析** | 快速上線 > 長期成本 | 速度與彈性平衡 | 長期維護 > 短期成本 |

**口訣**：**「安全選 Low，簡單選 No，核心選 Traditional」**

---

### 3.13 決策框架（Build vs Buy vs NC/LC）

**先懂一句話**：企業選型時有三條路，各有適用場景。

| 策略 | 適用場景 | 典型情境 |
|---|---|---|
| **Build（傳統開發）** | 核心競爭力、高度客製化需求 | 金融交易引擎、電商推薦系統 |
| **Buy（購買軟體）** | 標準化功能、快速部署 | HR 系統（SAP）、CRM（Salesforce） |
| **NC/LC** | 快速驗證、非核心流程自動化 | 活動報名、內部審批流程 |

🗣️ **白話說明**：核心業務是你的護城河（Build）；一般辦公室需求買現成就好（Buy）；快速驗證想法或自動化非核心流程就用 NC/LC。

## Section 4 · 易混淆概念對照表

### 4.1 Shadow IT vs Citizen Developer（最常考）🔥🔥🔥

| 面向 | 公民開發者（Citizen Developer） | 影子 IT（Shadow IT） |
|---|---|---|
| **工具** | 組織**核可**的 NC/LC 平台 | **未經核可**的工具 |
| **IT 知情** | 是——在 IT 治理框架下 | 否——IT 完全不知道 |
| **性質** | 正面——企業鼓勵的趨勢 | 負面——需防範的風險 |
| **考試立場** | 題目呈正面語境 | 題目呈風險語境 |

**考題判斷：** 看到「IT 不知情」「未經核可」→ Shadow IT；看到「組織核可工具」「IT 治理框架」→ Citizen Developer

---

### 4.2 Vendor Lock-in vs Shadow IT（兩大風險對比）🔥🔥

| 面向 | Vendor Lock-in（供應商鎖定） | Shadow IT（影子 IT） |
|---|---|---|
| **定義** | 深度依賴平台，無法遷移 | 未經 IT 核可的系統 |
| **風險本質** | 失去技術選擇自主權 | 失去安全與合規控制 |
| **誰造成的** | 平台本身的封閉設計 | 使用者自行繞過 IT |
| **影響對象** | 企業的長期策略 | 企業的資安與合規 |
| **解方** | 評估平台開放性、匯出能力 | 建立治理框架、核可工具清單 |

**考題判斷：** 「無法遷移」「換平台要重寫」→ Vendor Lock-in；「IT 不知情」「未經核可」→ Shadow IT

---

### 4.3 技術債 vs 擴展性瓶頸（易混）

| 面向 | 技術債（Technical Debt） | 擴展性瓶頸（Scalability Bottleneck） |
|---|---|---|
| **起因** | 快速開發後未重構 | 平台本身的技術限制 |
| **誰造成的** | 開發者的行為選擇 | 平台的架構天花板 |
| **症狀** | 改了功能 A 卻壞了 B | 人數增多後效能下降 |
| **解法** | 重構（Refactoring）程式碼 | 遷移高負載模組到傳統架構 |

**考題判斷：** 「越來越難維護」「疊床架屋」→ 技術債；「高併發撐不住」「複雜邏輯超出平台」→ 擴展性瓶頸

---

### 4.4 Low Code「模型」vs ML「模型」（陷阱題）🔥

| 面向 | Low Code 模型（Model） | ML 模型（ML Model） |
|---|---|---|
| **定義** | 抽象描述資料結構、流程與介面 | 從數據中學習的數學函數 |
| **產出** | 可執行的應用程式 | 預測或分類結果 |
| **使用者** | 公民開發者 / 開發者 | 資料科學家 / ML 工程師 |

**考題判斷：** 題目脈絡提到 Low Code 平台 → 「模型」= 設計元件；題目脈絡提到 AI/機器學習 → 「模型」= 訓練出的數學函數

---

### 4.5 NC 優勢 vs LC 優勢

| 面向 | No Code | Low Code |
|---|---|---|
| **門檻** | 最低——零程式碼 | 低～中——需少量程式知識 |
| **速度** | 最快——小時到天 | 快——天到週 |
| **客製化** | 低（受限模板） | 高（可寫程式碼擴展） |
| **IT 治理** | 較難管控 | 較易納入治理框架 |
| **適合使用者** | 非技術業務人員 | 開發者 / 公民開發者 |

**考題判斷：** 看到「零程式碼」「最快上線」「非技術人員」→ No Code；看到「少量程式碼」「IT 治理」「企業級」→ Low Code

## Section 5 · 口訣

### 口訣一：優勢記「快省民」

> **快** → 快速開發（20 倍速）
> **省** → 成本節省（減少工程師依賴）
> **民** → 賦能業務人員 + AI 民主化（Democratization）

### 口訣二：限制記「鎖影效安債客」

> **鎖** → Vendor Lock-in（供應商鎖定）
> **影** → Shadow IT（影子 IT 風險）
> **效** → 效能限制（Performance Limitations，高併發瓶頸）
> **安** → 資安隱私疑慮（Security & Privacy Risks）
> **債** → 技術債（Technical Debt）
> **客** → 客製化受限（Customization Constraints）

### 口訣三：場域選擇原則

> **「安全選 Low，簡單選 No，核心選 Traditional」**

- 高安全合規（金融、醫療）→ **Low Code**
- 簡單標準化（教育、新創 MVP）→ **No Code**
- 高效能核心系統（支付引擎、高并發）→ **Traditional**

### 口訣四：影子 IT vs 公民開發者

> **「核可是公民，偷做是影子」**

- 組織核可 + IT 治理框架 → **公民開發者**（正面）
- 未經核可 + 繞過 IT → **影子 IT**（負面風險）

## Section 6 · 考試陷阱

### 陷阱一：Zapier 不支援 API ❌→✅

❌ 陷阱：「Zapier 的限制是不支援 API 串接」

✅ 正解：Zapier **支援 API 串接**（它的核心功能就是靠 API 串接服務）。真正的限制是**複雜邏輯有限制**——它的 Trigger-Action 模式難以處理深層巢狀邏輯或複雜條件分支。

---

### 陷阱二：LCNC 能完全取代傳統開發 ❌→✅

❌ 陷阱：「NC/LC 發展到最後可以完全取代傳統程式開發」

✅ 正解：LCNC 與傳統開發是**互補**關係（Complementary），不是替代關係。高度客製化、高并發、複雜演算法仍需傳統開發。Gartner 預測 75% 新應用用 LCNC，意味著仍有 25% 需要傳統方式。

---

### 陷阱三：Low Code 的「模型」= ML 模型 ❌→✅

❌ 陷阱：看到「模型（Model）」就聯想到機器學習模型

✅ 正解：Low Code 平台中的「模型」是指**抽象描述資料結構、業務流程與介面邏輯的設計元件**，與 ML 模型（訓練出來做預測的數學函數）完全不同。

---

### 陷阱四：公民開發者越多越好，不需要管理 ❌→✅

❌ 陷阱：「鼓勵公民開發者 = 放任大家自己做，IT 不用管」

✅ 正解：公民開發者的價值**建立在 IT 治理框架之上**。沒有治理的公民開發 = 影子 IT = 風險。企業應提供核可工具清單 + 制定規範 + 定期審查，而非完全放手。

---

### 陷阱五：LCNC + AI = 不需要人類參與 ❌→✅

❌ 陷阱：「LCNC 加上生成式 AI 後，整個開發流程可以全自動、零人工」

✅ 正解：AI 可加速介面生成與內容產出，但**業務需求判斷、資料品質把關、AI 偏差（Bias）監督**仍需人類參與。AI 民主化降低的是技術門檻，不是消除人的角色。

---

### 陷阱六：Vendor Lock-in 只是成本問題 ❌→✅

❌ 陷阱：「Vendor Lock-in 的問題就是換平台要花錢，只要遷移成本可接受就能解決」

✅ 正解：Vendor Lock-in 影響的不只是遷移成本，更包括：技術策略自主性喪失、資料格式不相容風險、遷移期間服務中斷、以及可能換到新平台後又產生新的鎖定。

---

### 陷阱七：Shadow IT 是優勢而非風險 ❌→✅

❌ 陷阱：「員工主動用工具解決問題，是 LCNC 賦能的體現，屬於優勢」

✅ 正解：只有在**IT 核可**下的自主開發才是「公民開發者賦能」（優勢）。未經核可的自主建置就是 **Shadow IT**（風險）——差別在於「有無治理框架」。

---

### 陷阱八：技術債是平台的問題 ❌→✅

❌ 陷阱：「技術債是因為 LCNC 平台設計不好造成的」

✅ 正解：技術債是**使用者的開發行為**造成的——快速疊加功能但從不重構。NC/LC 平台本身的速度優勢反而更容易讓人輕忽重構的必要性。

---

## Section 7 · 情境題快速判斷

### 決策流程圖

```
遇到 NC/LC 情境題
        │
        ▼
問 1：有沒有「IT 不知情」「未核可」的字眼？
   是 → Shadow IT 風險
   否 ↓
問 2：有沒有「無法遷移」「換平台要重寫」的情境？
   是 → Vendor Lock-in
   否 ↓
問 3：有沒有「疊加功能後越來越難改」「改了 A 壞 B」？
   是 → 技術債
   否 ↓
問 4：有沒有「高併發撐不住」「複雜邏輯超出能力」？
   是 → 擴展性瓶頸（效能限制）
   否 ↓
考慮選型方向：安全選 Low、簡單選 No、核心選 Traditional
```

### 關鍵字速查表

| 看到這些關鍵字 | 答案方向 |
|---|---|
| 「無法遷移」「換平台要重寫」「被平台綁住」 | Vendor Lock-in |
| 「IT 不知情」「未經核可」「員工自行建置」 | Shadow IT |
| 「高併發撐不住」「複雜邏輯超出平台」 | 效能限制（擴展性瓶頸） |
| 「疊加功能後越來越亂」「修了 A 壞了 B」 | 技術債 |
| 「快速原型」「MVP」「兩週上線」 | NC/LC 速度優勢 |
| 「非技術人員自行開發」「組織核可工具」 | 公民開發者賦能 |
| 「金融/醫療 + 法規合規」 | Low Code（安全選 Low） |
| 「Zapier 的限制是什麼」 | 複雜邏輯有限制（不是不支援 API！） |
| 「模型」在 LC 平台脈絡 | 抽象描述資料結構、流程與介面（非 ML 模型） |

### 情境題範例

**情境 A**：「某企業業務部自行註冊了 Notion 帳號管理客戶資料，IT 部門完全不知情，事後發現資料存於海外伺服器，可能違反個資法。」

→ **Shadow IT 風險**（判斷依據：IT 不知情 + 未核可 + 合規風險）

**情境 B**：「某企業使用 NC 平台三年後，平台宣布停止支援舊版 API，想遷移至其他平台，卻發現所有應用邏輯無法匯出，必須從頭重寫。」

→ **Vendor Lock-in**（判斷依據：無法遷移 + 依賴平台專有格式）

**情境 C**：「某行銷部門使用 NC 工具快速上線的活動系統，一年後需求越加越複雜，每次改動都會影響到其他功能，維護成本不斷攀升。」

→ **技術債**（判斷依據：快速上線後持續疊加功能，未重構，維護困難）

**情境 D**：「某新創公司初期用 No Code 做的 SaaS 產品，使用者從 100 成長到 10 萬，系統開始頻繁卡頓甚至崩潰。」

→ **效能限制（擴展性瓶頸）**（判斷依據：使用者規模增加 → 平台效能天花板）
