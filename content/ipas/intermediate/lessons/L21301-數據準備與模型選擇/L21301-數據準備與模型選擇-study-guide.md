## 1. Big Picture — 數據準備與模型選擇核心流程

### 整體流程管線

```text
業務問題
  ↓
數據收集（Data Collection）
  ↓  ← 代表性？標籤品質？任務對齊？
數據清洗（Data Cleaning）
  ↓  ← 缺失值、異常值、重複值、格式統一
尺度調整 + 類別編碼（Scaling + Encoding）
  ↓  ← 先切分，再轉換（防止 Data Leakage）
特徵工程（Feature Engineering）
  ↓  ← 特徵選擇、轉換、建立
資料切分（Train / Validation / Test Split）
  ↓
模型選擇（Model Selection）
  ↓  ← 任務 × 資料 × 商業 × 成本
模型評估（Evaluation）
  ↓  ← 類別不平衡時不能只看 Accuracy
部署
```

### 快查表：題目問的是 → 想到

| 題目問的是 | 想到 |
|---|---|
| 樣本偏向某族群、地區、時段 | 代表性不足（Representativeness） |
| 資料有空值、空白欄位 | 缺失值處理（Missing Value Imputation） |
| 某值明顯偏離常識 | 異常值判斷（Outlier） |
| 特徵數值尺度差很大 | 尺度調整（Scaling） |
| 類別欄位轉數字 | 類別編碼（Categorical Encoding） |
| 模型選哪個最好 | 看任務 × 資料 × 商業 × 成本 trade-off |
| 詐欺、流失、罕見故障 | 類別不平衡（Class Imbalance）→ 不能只看 Accuracy |
| 模型離線好、上線差 | Data Leakage 或代表性問題 |
| 選哪個集合來決定最終模型 | Validation Set（驗證集） |
| 最後一次確認模型好壞 | Test Set（測試集） |

## 2. 數據收集（Data Collection）🔥🔥

### 先懂一句話

收資料時，最重要的不是「量多」，而是資料**任務對齊、代表性夠、標籤正確、合法授權**。這四件事缺一，後面蓋再好的模型都沒用。

### Everyday Analogy

你想預測 LINE 群組哪些人會取消追蹤。如果你只收到「從不發言的人」的資料，樣本就嚴重偏掉，模型只學了一種人，上線後根本不準。

### 先問自己一個問題

這份資料真的能代表「未來模型上線時會遇到的人、時間、地區、行為」嗎？

如果答案是否定的，問題通常不是模型太弱，而是資料一開始就收歪了。

### 技術說法

數據收集（Data Collection）要確認四件事：

| 檢查點 | 英文 | 要問的問題 | 常見考題陷阱 |
|---|---|---|---|
| 任務對齊 | Task Alignment | 這些欄位能回答預測目標嗎？ | 資料很多，但和問題無關 |
| 代表性 | Representativeness | 樣本能反映真實母體嗎？ | 只收單一地區、時段、族群 |
| 標籤品質 | Label Quality | 目標欄位正確且標準一致嗎？ | 標註規則不一、舊制標籤混用 |
| 合法授權 | Legal Authorization | 資料來源可合法使用嗎？ | 忽略同意、隱私、授權問題 |

### 流程 / 選擇流程

```text
業務問題
  ↓
確認預測目標（要預測什麼）
  ↓
檢查資料是否任務對齊
  ↓
檢查樣本代表性
  ↓
檢查標籤品質
  ↓
確認合法授權
  ↓
進入資料清洗與前處理
```

### 一步一步例子

情境：公司想預測「客戶是否會在 30 天內流失」。

```text
Step 1：任務是分類（流失 / 未流失）
Step 2：資料要同時有流失與未流失客戶
Step 3：樣本要涵蓋不同地區、客群、使用頻率
Step 4：流失標籤的定義要一致，例如 30 天未登入才算流失
Step 5：確認資料來源可合法用於模型訓練
```

如果資料只包含「已流失」客戶，模型看不到對比，就不是好資料。

### 比較表這樣讀

這張表不是背定義，而是用來判斷「任務需要哪種目標欄位」：

| 任務 | 需要什麼 |
|---|---|
| 分類（Classification） | 類別標籤（Label） |
| 回歸（Regression） | 連續數值目標 |
| 分群（Clustering） | 可沒有標籤，但需要有意義的特徵 |

讀法：題目問「是否流失、是否詐欺」就是分類；問「銷售額、溫度、價格」就是回歸；問「把客戶分成幾群」就是分群。

### 記憶方式

```text
收資料四問：
對不對任務？代表不代表？標籤準不準？能不能合法用？
```

### Exam Rule

```text
樣本偏向特定族群 / 地區 / 時間 → 代表性不足（Representativeness）
標籤標準不一致 / 舊制標籤 → 標籤品質（Label Quality）問題
資料很多但問題答不了 → 任務未對齊，多也沒用
```

### Quick Check

分類任務：公司想預測客戶是否會在 30 天內流失。已收集了 5 萬筆資料，但全都是「已流失」的客戶記錄。這份資料最主要的問題是什麼？

答案：代表性不足（Representativeness）。
理由：沒有「未流失」的客戶資料，模型看不到兩類樣本的對比，無法學到有效的分辨能力。

## 3. 數據清洗與預處理（Data Cleaning and Preprocessing）🔥🔥🔥

### 先懂一句話

原始資料幾乎一定有問題：空值、怪值、重複、格式亂。**數據清洗就是把這些問題在建模前逐一解決**，讓後面的模型拿到乾淨可用的輸入。

### Everyday Analogy

就像整理 LINE 投票結果：先刪掉重複回覆、補上空白欄位、把日期格式統一成同一格式——不做這些，後面的分析就越看越亂。

### 先問自己一個問題

這筆資料是「真的缺、真的異常」，還是只是格式不一致造成看起來有問題？

不要看到空值就刪、看到異常值就刪。考題常考你是否會先判斷來源。

### 技術說法

數據清洗與預處理（Data Cleaning and Preprocessing）是在建模前處理缺失值（Missing Values）、異常值（Outliers）、重複值（Duplicates）、格式一致性（Format Consistency），再進一步做尺度調整（Scaling）與類別編碼（Categorical Encoding）。

### 流程 / 選擇流程

```text
原始資料
  ↓
1. 檢查缺失值（Missing Values）
  ↓
2. 判斷異常值（Outliers）
  ↓
3. 移除或合併重複值（Duplicates）
  ↓
4. 統一格式（日期、單位、大小寫、類別名稱）
  ↓
5. 進入縮放 / 編碼（Scaling / Encoding）
  ↓
可建模資料
```

缺失值決策流程：

```text
看到缺失值
│
├─ 缺失比例低，刪掉不影響代表性 → 刪除（Deletion）
├─ 缺失比例高，資料珍貴 → 補值（Imputation）
└─ 空白本身有業務意義 → 保留為訊號
```

異常值決策流程：

```text
看到異常值
│
├─ 真實但極端的案例（高消費客戶）→ 保留，可考慮縮放或轉換
└─ 錄入錯誤（年齡 350、金額多打一個 0）→ 修正或刪除
```

### 一步一步例子

情境：客戶資料中「年薪」有缺失值，而且少數人的年薪非常高。

```text
Step 1：年薪是數值欄位
Step 2：資料有極端高值，平均數會被拉高
Step 3：補值時選中位數（Median）
Step 4：補完後再進入縮放或建模
```

為什麼不是平均數？

```text
年薪：40、45、50、55、1000
平均數 = (40 + 45 + 50 + 55 + 1000) / 5 = 238
中位數 = 50
```

238 明顯不像多數人的年薪，中位數 50 比較穩健。

### 比較表這樣讀

先看「問題類型」，再選處理方式：

| 情境 | 建議做法 |
|---|---|
| 缺失比例低，刪掉不影響代表性 | 刪除（Deletion） |
| 缺失比例高，資料珍貴 | 補值（Imputation） |
| 空白本身有業務意義 | 保留，當作一種訊號 |

補值方法表要看「欄位類型」和「是否有極端值」：

| 欄位類型 | 常用補法 | 注意 |
|---|---|---|
| 數值欄位，無極端值 | 平均數（Mean） | 有極端值時不穩 |
| 數值欄位，有極端值 | 中位數（Median） | 較穩健 |
| 類別欄位 | 眾數（Mode） | 用最常出現的類別補 |

### 記憶方式

```text
缺失值：少可刪，多要補，有意義就保留。
異常值：先問真假，再決定刪不刪。
```

### Exam Rule

```text
有極端值 → 補值用中位數（Median），不用平均數（Mean）
異常值出現 → 先判斷是真實案例還是錯誤，不是直接刪
缺失比例高且資料珍貴 → 補值而非刪除
```

### Quick Check

某份資料的「年薪」欄位有一筆填 999,999,999 元。處理前應該先做什麼？

答案：先判斷這個值是真實的極端案例還是錄入錯誤，再決定是保留、修正還是刪除。
理由：異常值不是看到就刪，錯誤判斷來源會丟失有意義的資訊或留下錯誤資料。

## 4. 尺度調整與類別編碼（Scaling & Categorical Encoding）🔥🔥🔥

### 先懂一句話

模型看到的是數字。如果不同特徵的數字尺度差很多（薪資 vs 0/1），某些模型會被大數字主導。類別欄位也要轉成數字，但轉法不對會引入假順序或維度爆炸。**這一節的核心是「怎麼轉才對，什麼時候轉」。**

### Everyday Analogy

**縮放：** 把不同科目的成績都換成百分制，才能公平加總比較。沒有換算前，滿分 100 的科目和滿分 10 的科目放在一起，大的永遠佔上風。

**編碼：** 把「北、中、南」這三個詞翻譯成電腦能讀的語言。但如果翻成 1、2、3，電腦可能誤以為「南 > 中 > 北」，這就是假順序問題。

### 先問自己一個問題

這個欄位的數字大小，會不會不公平地影響模型？這個類別欄位，原本有沒有自然順序？

### 技術說法

尺度調整（Scaling）是把不同數值特徵轉到可比較的尺度。類別編碼（Categorical Encoding）是把文字類別轉成模型可讀的數字。兩者都要注意資料洩漏（Data Leakage）：必須先切分資料，再用訓練集學轉換規則。

### 流程 / 選擇流程

```text
全部資料
  ↓
先切 Train / Validation / Test
  ↓
只用 Training Set 學縮放、編碼、補值規則
  ↓
同一套規則套到 Validation / Test
  ↓
訓練與評估模型
```

縮放選擇流程：

```text
需要縮放嗎？
│
├─ 使用 SVM（Support Vector Machine，支援向量機）/ k-NN（k-Nearest Neighbors，k 近鄰）/ 距離型模型 → 需要
├─ 特徵尺度差很大 → 需要
├─ 決策樹 / 隨機森林 → 縮放不是第一優先
└─ 有明顯離群值但仍需縮放 → RobustScaler
```

編碼選擇流程：

```text
類別欄位要轉數字
│
├─ 有自然順序（低 / 中 / 高）→ 序位編碼（Ordinal Encoding）
├─ 無自然順序、類別不多 → One-Hot 編碼（One-Hot Encoding）
└─ 類別很多、高基數 → 目標編碼（Target Encoding），但注意 leakage
```

### 一步一步例子

情境：用 k-NN 預測客戶流失，資料有「年收入」和「是否訂閱電子報」。

```text
年收入：30000 到 3000000
是否訂閱電子報：0 或 1
```

如果不縮放，k-NN 計算距離時，年收入會完全主導距離。

```text
Step 1：先切分 Train / Val / Test
Step 2：只用 Train 算縮放規則
Step 3：把同一套規則套到 Val / Test
Step 4：再訓練 k-NN
```

類別例子：

```text
教育程度：高中 / 大學 / 研究所 → 有順序 → Ordinal Encoding
居住縣市：台北 / 台中 / 高雄 → 無大小順序 → One-Hot Encoding
城市代碼 300 種 → 高基數 → 考慮 Target Encoding，但只能在切分後做
```

### 比較表這樣讀

縮放表先看「有沒有離群值」和「需不需要固定範圍」：

| | 標準化（Standardization, Z-score） | 正規化（Normalization, Min-Max Scaling） | 穩健縮放（RobustScaler） |
|---|---|---|---|
| 結果 | 平均接近 0、標準差接近 1 | 壓到固定區間，常見 0 到 1 | 用中位數和 IQR（四分位距）縮放 |
| 對離群值 | 敏感（mean/std 受極端值影響） | 非常敏感（min/max 直接被拉偏） | 不敏感（中位數、IQR 不受極端值影響） |
| 適用 | 尺度差大、分布近似常態、無明顯離群值 | 需要固定範圍、上下界明確 | 資料有明顯離群值，但仍需縮放 |
| 常搭配模型 | SVM、k-NN | 圖像、特定神經網路輸入 | SVM、k-NN（含離群值資料） |

RobustScaler（穩健縮放器）用中位數（Median）和 IQR（Interquartile Range，四分位距）做縮放，重點是「對離群值不敏感」。

```text
Exam Rule：
資料有明顯離群值 + 需要縮放 → RobustScaler（比標準化更穩健）
需要固定範圍 [0, 1] → MinMaxScaler（正規化）
近似常態 / 大多數 ML 演算法 / 無明顯離群值 → StandardScaler（標準化）
```

模型表要看「模型是不是靠距離或邊界」：

| 在意縮放 | 相對不依賴縮放 |
|---|---|
| SVM | 決策樹（Decision Tree） |
| k-NN | 隨機森林（Random Forest） |

> 注意：「樹模型不依賴縮放」≠ 不用做前處理。缺失值、格式等問題仍要處理。

編碼表先看「有沒有順序」，再看「類別多不多」：

| 方法 | 適用情境 | 風險 |
|---|---|---|
| One-Hot 編碼 | 無順序、類別不多 | 高基數時維度暴增 |
| 序位編碼（Ordinal Encoding） | 本來就有順序（低/中/高） | 無順序時不適用 |
| 標籤編碼（Label Encoding） | 通常不優先 | 容易引入假順序 |
| 目標編碼（Target Encoding） | 高基數欄位 | 切分前使用 → Target Leakage |

### 記憶方式

```text
縮放：距離模型先想它。
編碼：有順序用 Ordinal，沒順序用 One-Hot，類別爆多才想 Target。
防洩漏：永遠先切分，再轉換。
```

### Exam Rule

```text
特徵尺度差很大 / SVM / k-NN → 先想到縮放（Scaling）
有自然順序的類別 → 序位編碼（Ordinal Encoding）
無順序、類別不多 → One-Hot 編碼（One-Hot Encoding）
類別很多、高基數 → 目標編碼（Target Encoding），但注意 leakage
切分前先做縮放或目標編碼 → 資料洩漏（Data Leakage）
樹模型 → 縮放不是第一優先，但其他前處理仍要做
```

### Quick Check

一個資料集有「城市」欄位，共有 300 個不同城市值。要轉成模型可用的形式，應優先考慮什麼問題？

答案：高基數（300 個類別）使用 One-Hot 編碼會造成維度暴增，可考慮目標編碼（Target Encoding），但要注意在切分後才能計算，避免 Target Leakage。
理由：One-Hot 對高基數欄位成本太高；目標編碼能降維，但需要防止洩漏。

## 5. 特徵工程（Feature Engineering）🔥🔥

### 先懂一句話

特徵工程是**把原始資料變成模型更容易學習的形式**。重點不是塞更多欄位，而是讓每個特徵都真的對預測有幫助。

### Everyday Analogy

履歷不是把每件小事都塞進去，挑對重點反而更有競爭力。同理，特徵工程是挑選、轉換、創建最有說服力的「資料特徵」。

### 先問自己一個問題

這個欄位能不能讓模型更容易看出規律？還是只是在增加噪音？

### 技術說法

特徵工程（Feature Engineering）包含特徵選擇（Feature Selection）、特徵轉換（Feature Transformation）、特徵建立（Feature Creation）。目的不是讓欄位越多越好，而是提升訊號、降低噪音、讓模型更容易泛化（Generalization）。

### 流程 / 選擇流程

```text
原始欄位
  ↓
先刪掉無關或重複訊號（Feature Selection）
  ↓
把欄位轉成更好學的形式（Feature Transformation）
  ↓
從多個欄位衍生有意義的新欄位（Feature Creation）
  ↓
確認沒有使用未來資訊或標籤資訊
  ↓
進入模型訓練
```

### 一步一步例子

情境：電商要預測客戶是否會流失。

```text
原始欄位：
交易日期、交易金額、商品類別、會員註冊日、客戶 ID

Step 1：刪除不適合直接使用的客戶 ID（Feature Selection）
Step 2：把交易日期拆成星期幾、月份、是否假日（Feature Transformation）
Step 3：計算近 30 天消費金額、距上次購買天數（Feature Creation）
Step 4：確認這些特徵只使用預測時間點以前的資料
```

如果用到「客戶之後有沒有取消訂閱」來做特徵，就是 Target Leakage。

### 比較表這樣讀

這張表要用來判斷「你是在挑欄位、改欄位，還是造新欄位」：

| 方向 | 說明 | 例子 |
|---|---|---|
| 特徵選擇（Feature Selection） | 從現有欄位挑重要的 | 刪掉與預測無關的欄位 |
| 特徵轉換（Feature Transformation） | 把欄位轉成更適合的形式 | 日期拆成年/月/日/星期；偏態取 Log |
| 特徵建立（Feature Creation） | 從原始欄位衍生新欄位 | 近 30 天總消費、距上次購買天數 |

欄位越多不一定越好，因為：

```text
欄位過多 →
  ├─ 噪音增加（不相關特徵干擾學習）
  ├─ 維度膨脹（Curse of Dimensionality）
  ├─ 計算成本上升
  └─ 過度擬合風險增加
```

### 記憶方式

```text
Feature Selection = 挑重點
Feature Transformation = 換表達方式
Feature Creation = 做出新線索
```

### Exam Rule

```text
題目提到欄位很多、重複資訊多、模型易過擬合 → 特徵選擇（Feature Selection）
日期拆成年月日星期 → 特徵轉換（Feature Transformation）
從交易紀錄算「近 30 天總消費」 → 特徵建立（Feature Creation）
強調「欄位越多越完整」 → 這是陷阱，好的特徵工程重質不重量
```

### Quick Check

模型訓練後在訓練集準確率很高，但驗證集準確率明顯偏低。可能的原因之一是什麼，解法是什麼？

答案：可能是特徵過多導致過度擬合（Overfitting）。
理由：太多不相關的特徵讓模型記住了訓練集的噪音，而非學到真正的模式；解法之一是做特徵選擇（Feature Selection），移除不重要的欄位。

## 6. 資料切分與交叉驗證（Dataset Split & Cross-Validation）🔥🔥🔥

### 先懂一句話

把資料切成三份：訓練集學習、驗證集選模型、測試集最後驗收。**最關鍵的規則：測試集只能用一次，用來最後確認，不能拿來反覆調整。**

### Everyday Analogy

平時練習算法題像訓練集；做一套模擬考卷後請朋友評分、幫你找弱點，像驗證集；最後正式上考場，那次才是測試集。正式考試的成績不會事先告訴你，讓你再複習一輪。

### 先問自己一個問題

這個集合是拿來「學習」、「選模型」，還是「最後驗收」？

只要把測試集拿來反覆調整，它就不再是最後驗收。

### 技術說法

資料切分（Dataset Split）通常分成訓練集（Training Set）、驗證集（Validation Set）、測試集（Test Set）。交叉驗證（Cross-Validation）是在訓練資料內多次切分，用來更穩定估計模型表現，但不能取代獨立測試集。

### 流程 / 選擇流程

```text
特徵工程
  ↓
先切資料
  ↓
Training Set：訓練模型、學前處理規則
  ↓
Validation Set：比較模型、調超參數
  ↓
Test Set：最後一次確認泛化能力
```

交叉驗證流程：

```text
訓練資料切成 k 份
  ↓
輪流拿其中 1 份當驗證，其餘 k-1 份訓練
  ↓
平均多次結果，得到較穩定估計
  ↓
選定模型後，仍用獨立 Test Set 最終驗收
```

### 一步一步例子

情境：有 10,000 筆資料，要比較 SVM、Random Forest、Neural Network。

```text
Step 1：切成 Train 70%、Validation 15%、Test 15%
Step 2：在 Train 訓練三種模型
Step 3：用 Validation 比較三者並調參數
Step 4：選定最終模型
Step 5：只用 Test 做最後一次評估
```

如果 Step 5 後又回去調模型，測試集就被污染了。

### 比較表這樣讀

這張表要看「你現在正在做哪個動作」：

| 集合 | 角色 | 禁忌 |
|---|---|---|
| 訓練集（Training Set）| 讓模型學習 | 不能只有訓練集就上線 |
| 驗證集（Validation Set）| 比較模型、調整超參數 | 不能當最終評估依據 |
| 測試集（Test Set）| 最終驗收 | 不能反覆用來調整模型 |

切分比例只是常見參考，不是死背數字：

```text
全部資料 100%
  ├─ 訓練集 60-80%
  ├─ 驗證集 10-20%
  └─ 測試集 10-20%
```

### 記憶方式

```text
Train 學習，Val 選擇，Test 驗收。
測試集像正式考，不能考完偷看答案再重考。
```

### Exam Rule

```text
「比較不同模型哪個好」→ 用驗證集（Validation Set）
「最後確認泛化能力」→ 用測試集（Test Set）
「測試集成績差，可以再調一下」→ 這是陷阱，測試集會失去公正性
「資料量小，要更穩定估計」→ 交叉驗證（Cross-Validation）
「交叉驗證做完，不需要測試集了」→ 這是陷阱，仍需要獨立測試集
```

### Quick Check

機器學習工程師用測試集結果發現模型偏差，調整後再看測試集，反覆三次終於得到滿意結果。這個做法的問題是什麼？

答案：測試集已經失去公正性，變成了「有標準答案的考試卷」，成績不再能反映模型真實泛化能力。
理由：每次根據測試集回去調整，等同於間接用測試集訓練了模型，上線後可能表現遠不如預期。

## 7. 類別不平衡與模型選擇（Class Imbalance & Model Selection）🔥🔥🔥

### 先懂一句話

當少數類別（如詐欺、故障）非常少見時，模型可能直接「全部猜多數類別」就能得到很高的準確率，但其實根本沒學到任何有用的東西。**這就是為什麼類別不平衡時，Accuracy 是最危險的評估指標。**

### Everyday Analogy

100 天裡只有 1 天異常。系統天天都說「正常」，Accuracy 是 99%，但真正重要的那 1 天它完全失敗。這種模型毫無用處，但數字看起來很漂亮。

### 先問自己一個問題

少數類別是不是業務真正關心的目標？如果是，Accuracy 可能正在騙你。

### 技術說法

類別不平衡（Class Imbalance）是分類資料中某一類樣本遠少於另一類，例如詐欺、流失、故障。模型選擇（Model Selection）不是選「最強模型」，而是依任務型態、資料特性、商業需求、成本限制做取捨。

### 流程 / 選擇流程

```text
看到分類任務
  ↓
檢查類別比例
  ↓
如果少數類別很重要
  ↓
不要只看 Accuracy
  ↓
選處理策略：重抽樣 / 類別權重 / 換評估指標
  ↓
再依任務、資料、商業、成本選模型
```

模型選擇流程：

```text
先看任務：分類 / 回歸 / 分群
  ↓
再看資料：資料量、維度、線性或非線性、是否不平衡
  ↓
再看商業：可解釋性、穩定性、上線速度
  ↓
最後看成本：訓練成本、推論成本、維護難度
```

### 一步一步例子

情境：詐欺偵測資料中，99.5% 是正常交易，0.5% 是詐欺。

```text
模型 A：全部猜正常
正常交易 99.5%，所以 Accuracy = 99.5%
但詐欺一筆都抓不到，Recall = 0
```

考試要看出：Accuracy 很高不代表好，因為業務真正關心的是少數類別有沒有被抓到。

SMOTE（Synthetic Minority Over-sampling Technique，合成少數類別過採樣技術）例子：

```text
Step 1：先切 Train / Val / Test
Step 2：只在 Train 裡對少數類別做 SMOTE
Step 3：SMOTE 在少數類別樣本和近鄰之間插值，生成合成樣本
Step 4：Val / Test 保持原始分布，用來公平評估
```

### 比較表這樣讀

類別不平衡策略表要看「你想改資料、改訓練，還是改評估」：

| 策略 | 說明 |
|---|---|
| 重抽樣（Resampling） | 過採樣（Oversampling）增加少數類別樣本，或欠採樣（Undersampling）減少多數類別樣本。常見過採樣方法：**SMOTE**（合成少數類別樣本，插值生成，非直接複製）。注意：SMOTE 必須在切分後，只套用於訓練集，否則驗證集被污染。 |
| 類別權重（Class Weight） | 讓模型在訓練時更重視少數類別的錯誤 |
| 換評估指標 | 不只看 Accuracy，改用 Precision、Recall、F1-score 等 |

SMOTE 和簡單過採樣的差別：

| 方法 | 做法 | 問題 |
|---|---|---|
| 簡單過採樣（Random Oversampling） | 複製少數類別樣本 | 容易過擬合（完全重複） |
| SMOTE | 插值生成新樣本 | 較自然，泛化能力較好 |

模型選擇表要看「需求優先順序」，不是背誰最好：

| 軸 | 考量 |
|---|---|
| 任務型態 | 分類 / 回歸 / 分群 |
| 資料特性 | 資料量、維度、線性/非線性、類別不平衡 |
| 商業需求 | 可解釋性、穩定性、上線速度 |
| 成本限制 | 訓練成本、推論成本、維護難度 |

常見模型 trade-off：

| 模型 | 優點 | 缺點 |
|---|---|---|
| 線性模型（Linear Model） | 快、可解釋性高 | 難處理複雜非線性關係 |
| 決策樹（Decision Tree） | 直觀、好說明 | 易過度擬合 |
| 隨機森林（Random Forest） | 穩健、效果好 | 較難解釋，成本較高 |
| 支援向量機（SVM, Support Vector Machine） | 小中型資料有效 | 對尺度敏感，大資料成本高 |
| k 近鄰（k-NN, k-Nearest Neighbors） | 直觀 | 推論慢，對尺度敏感 |
| 神經網路（Neural Network） | 表現潛力高 | 資料、算力、維護成本高 |

迴歸評估指標表要看「你想懲罰什麼」：

| 指標 | 公式 | 特點 |
|---|---|---|
| MAE（Mean Absolute Error，平均絕對誤差） | mean(\|y − ŷ\|) | 對離群值不敏感 |
| MSE（Mean Squared Error，均方誤差） | mean((y − ŷ)²) | 放大大誤差，可微分 |
| RMSE（Root Mean Squared Error，均方根誤差） | √MSE | 與 y 同單位，常用 |
| R²（決定係數） | 1 − SS_res/SS_tot | 越接近 1 越好（可為負） |

```text
Exam Rule（迴歸指標）：
迴歸 + 對離群值不敏感 → MAE
迴歸 + 懲罰大誤差 → MSE / RMSE
迴歸 + 解釋模型解釋力 → R²
```

### 記憶方式

```text
不平衡任務：Accuracy 先懷疑。
SMOTE：不是複製，是插值。
選模型：任務、資料、商業、成本四個軸一起看。
```

### Exam Rule

```text
詐欺 / 流失 / 異常 / 罕見疾病 → 類別不平衡，不能只看 Accuracy
「少數類別很重要」→ 想到重抽樣（Resampling）或類別權重（Class Weight）
「SMOTE」→ 插值過採樣，切分後只套用於訓練集
「需要高可解釋性」→ 線性模型或決策樹
「表格資料、效果與穩健性優先」→ 隨機森林
「資料量大、模式複雜」→ 神經網路（但成本高）
「神經網路最強，所以優先選」→ 這是陷阱，要看 trade-off
```

### Quick Check

銀行想建立詐欺偵測模型。測試集中 99.5% 是正常交易，模型 Accuracy 達到 99.3%。工程師是否應該滿意這個結果？

答案：不應該。
理由：在嚴重類別不平衡下，模型可能只是把所有交易都預測為「正常」，Accuracy 就能接近 99.5%。應改用 Recall（召回率）或 F1-score 等能反映詐欺偵測能力的指標。

## 8. Exam Decision Trees

### Tree 1：前處理選擇

```text
題目描述資料問題？
│
├─ 有空白 / 缺值
│   ├─ 缺失比例低 → 刪除（Deletion）
│   ├─ 缺失比例高且資料珍貴 → 補值（Imputation）
│   │   ├─ 有極端值 → 中位數（Median）
│   │   └─ 無極端值 → 平均數（Mean）
│   └─ 空白有業務意義 → 保留為訊號
│
├─ 有明顯偏離值
│   ├─ 真實極端案例 → 保留，考慮縮放
│   └─ 錄入錯誤 → 修正或刪除
│
├─ 特徵尺度差很大 / 使用 SVM / k-NN
│   └─ 縮放（Scaling）→ 標準化或正規化
│
└─ 類別欄位
    ├─ 有自然順序 → 序位編碼（Ordinal Encoding）
    ├─ 無順序、類別不多 → One-Hot 編碼（One-Hot Encoding）
    └─ 高基數 → 目標編碼（Target Encoding），注意 leakage
```

### Tree 2：資料切分與 Leakage 判斷

```text
切分時序出問題了嗎？
│
├─ 全資料先做縮放再切分
│   └─ → Data Leakage（資料洩漏）❌
│
├─ 切分前先做目標編碼
│   └─ → Target Leakage（標籤洩漏）❌
│
├─ 用測試集反覆調整模型
│   └─ → 測試集失效，評估過度樂觀 ❌
│
└─ 先切分，再由訓練集學轉換規則，套到 Val / Test
    └─ → 正確做法 ✅
```

### Tree 3：模型選擇

```text
業務問題是什麼任務？
│
├─ 分類 / 回歸
│   ├─ 需要高可解釋性？
│   │   ├─ 是 → 線性模型 / 決策樹
│   │   └─ 否
│   │       ├─ 資料量大、模式複雜 → 隨機森林 / 神經網路
│   │       └─ 小中型資料、尺度已處理 → SVM / k-NN
│
└─ 分群（Clustering）
    └─ 先確認特徵表示合理，再選分群方法

有類別不平衡？
  └─ 不能只看 Accuracy → 重抽樣 / 類別權重 / 換評估指標
```

### Tree 4：評估集合選擇

```text
題目問哪個集合？
│
├─ 「比較哪個模型較好」→ 驗證集（Validation Set）
├─ 「最後確認泛化能力」→ 測試集（Test Set）
├─ 「資料量小，要更穩定估計」→ 交叉驗證（Cross-Validation）
└─ 「交叉驗證完就夠了，不需要測試集」→ 陷阱，仍需要測試集
```

## 9. Trap Clinic

### Trap 1：資料越多越好

錯。資料多不代表資料對。

Exam fix：

```text
樣本偏向特定族群 / 地區 / 時段 → 代表性不足，量多也沒用
重要的是任務對齊 + 代表性 + 標籤品質 + 合法授權
```

### Trap 2：缺失值一律刪掉

錯。要先看缺失比例和業務代價。

Exam fix：

```text
缺失比例低 → 刪除可以
缺失比例高且資料珍貴 → 補值（Imputation）
空白有業務意義 → 保留當訊號（進階，中級只需了解概念）
```

### Trap 3：異常值一定要刪

錯。先判斷來源，再決定做法。

Exam fix：

```text
異常值 → 先問「是真實案例還是錄入錯誤」
真實高消費客戶 → 保留
年齡填 350、金額多打一個 0 → 修正或刪除
```

### Trap 4：標準化（Standardization）和正規化（Normalization）一樣

錯。兩者都是縮放，但目的不同。

Exam fix：

```text
標準化（Z-score）→ 平均接近 0、標準差接近 1，適合 SVM / k-NN
正規化（Min-Max）→ 壓到固定區間如 0 到 1，適合需要固定範圍的場景
```

### Trap 5：類別欄位直接用標籤編碼（Label Encoding）

錯。無順序的類別用標籤編碼會引入假順序。

Exam fix：

```text
無自然順序 → One-Hot 編碼（One-Hot Encoding）
有自然順序（低/中/高）→ 序位編碼（Ordinal Encoding）
標籤編碼 → 容易讓模型誤以為類別有大小關係
```

### Trap 6：One-Hot 編碼（One-Hot Encoding）永遠最好

錯。高基數欄位會維度暴增。

Exam fix：

```text
類別數不多 → One-Hot 是標準選項
類別數很多（高基數）→ 維度暴增，考慮目標編碼（Target Encoding）
但目標編碼在切分前使用 → Target Leakage
```

### Trap 7：驗證集（Validation Set）和測試集（Test Set）差不多

錯。角色完全不同。

Exam fix：

```text
驗證集（Validation Set）→ 比較模型、調超參數，可多次使用
測試集（Test Set）→ 最後驗收，只能用一次
一直用測試集調模型 → 測試集失效，評估不公正
```

### Trap 8：Accuracy 高就代表模型好

錯。類別不平衡時 Accuracy 會嚴重誤導。

Exam fix：

```text
詐欺 / 流失 / 異常 / 罕見疾病 → 先懷疑 Accuracy 是陷阱
應改用 Precision / Recall / F1-score 等指標
```

### Trap 9：樹模型不看尺度，所以前處理都能省

錯。樹模型只是相對不依賴縮放，其他前處理仍要做。

Exam fix：

```text
決策樹 / 隨機森林 → 縮放不是第一優先
但缺失值 / 重複值 / 格式 / 編碼 → 仍然需要處理
```

### Trap 10：前處理先在全資料上做比較省事

錯。這是最常見的 Data Leakage 來源。

Exam fix：

```text
切分前全資料做縮放 → Data Leakage
切分前做目標編碼 → Target Leakage
正確：先切分 → 只用訓練集學轉換規則 → 同規則套到 Val / Test
```

### Trap 11：神經網路最強，所以優先選

錯。中級考的是 trade-off，不是「誰最強」。

Exam fix：

```text
「神經網路最強」→ 資料量、解釋性、成本都會改變答案
高解釋性需求 → 線性模型 / 決策樹
表格資料 + 穩健 → 隨機森林
大資料 + 複雜模式 + 成本可接受 → 神經網路才合理
```

### Trap 12：交叉驗證做了就不需要測試集

錯。交叉驗證有助比較模型，但不能取代最終測試集。

Exam fix：

```text
交叉驗證（Cross-Validation）→ 更穩定估計泛化能力，適合資料量不大時
但最終仍需要獨立測試集做最後驗收
```

## 10. Practice Questions

### 10.1 數據收集與代表性

**Q1.** 公司想預測台灣全國門市的銷售量，但訓練資料只包含台北市的門市紀錄。這份資料最大的問題是什麼？

答案：代表性不足（Representativeness）。
理由：樣本只來自台北市，無法反映全台不同地區的實際銷售模式，模型上線後預測其他地區會有明顯偏差。

---

**Q2.** 一份標籤資料中，「客戶滿意度」欄位由不同部門各自標註，標準不一致。這屬於什麼問題？

答案：標籤品質（Label Quality）問題。
理由：標籤標準不一致會讓模型學到矛盾的信號，無論模型多強都無法學到正確的預測規則。

---

**Q3.** 分類任務需要什麼樣的資料？

答案：需要類別標籤（Label）。
理由：分類任務的目標是預測類別，沒有標籤就無法進行監督式學習。

---

### 10.2 缺失值、異常值、格式

**Q4.** 某數值欄位有 30% 的缺失值，且這份資料很難重新收集。最合適的處理方式是？

答案：補值（Imputation）。
理由：缺失比例高、資料珍貴時，直接刪除會損失太多資料，應選擇補值策略。

---

**Q5.** 某欄位的數值有明顯的右偏（含少數非常大的值）。若要補缺失值，哪個統計量比平均數（Mean）更穩健？

答案：中位數（Median）。
理由：有極端值（右偏）時，平均數會被大值拉高而不代表多數樣本，中位數不受極端值影響，更能代表中間位置。

---

**Q6.** 某筆資料「年齡」欄位填入了 350 歲。應該怎麼處理？

答案：先判斷是錄入錯誤，再修正或刪除。
理由：350 歲明顯不合理，應視為錄入錯誤處理，不能保留作為「真實極端案例」。

---

### 10.3 縮放與類別編碼

**Q7.** 使用 k-NN（k 近鄰演算法）建模前，為什麼需要對特徵做縮放？

答案：k-NN 基於特徵間的距離計算，尺度大的特徵會主導距離計算，縮放後讓每個特徵可公平比較。
理由：k-NN 對尺度高度敏感，不縮放會導致某些特徵影響力遠大於其他特徵，結果失準。

---

**Q8.** 「教育程度」欄位有三個值：高中 / 大學 / 研究所。應該用哪種編碼方法？

答案：序位編碼（Ordinal Encoding）。
理由：教育程度有自然的高低順序，序位編碼能正確保留這個順序關係。

---

**Q9.** 「居住縣市」欄位有 22 個不同值，且這些縣市之間沒有自然順序。最合適的編碼方法是？

答案：One-Hot 編碼（One-Hot Encoding）。
理由：無順序關係、且類別數量 22 個不算非常多，One-Hot 不引入假順序，是標準選擇。

---

**Q10.** 工程師在切資料集之前，先用全部資料算了均值和標準差來做標準化，再切成訓練集和測試集。這樣的做法有什麼問題？

答案：這是 Data Leakage（資料洩漏）。
理由：測試集的統計資訊（均值、標準差）被用來轉換訓練集，導致模型間接「看到了」測試集，離線評估過度樂觀。

---

### 10.4 特徵工程

**Q11.** 一份資料集有 200 個特徵欄位，但其中很多與預測目標相關性很低。應該採取什麼特徵工程策略？

答案：特徵選擇（Feature Selection）。
理由：移除低相關性特徵可降低噪音、減少維度，降低過度擬合風險並提升模型的可解釋性。

---

**Q12.** 電商平台把「交易時間」欄位拆成「小時」「星期幾」「是否為假日」三個新欄位。這屬於哪種特徵工程？

答案：特徵轉換（Feature Transformation）。
理由：把原始的時間欄位轉成模型更容易學習的有意義形式，屬於轉換而非創造全新資訊。

---

**Q13.** 從客戶交易記錄計算「近 90 天購買次數」作為新特徵。這屬於哪種特徵工程？

答案：特徵建立（Feature Creation）。
理由：從現有資料衍生出原本不存在的新欄位，屬於特徵建立。

---

### 10.5 資料切分與交叉驗證

**Q14.** 訓練完幾個模型後，應該用哪個集合來選出最佳模型？

答案：驗證集（Validation Set）。
理由：驗證集的角色就是比較模型、調整設定，讓測試集保持未見狀態。

---

**Q15.** 工程師在驗證集上選完模型後，用測試集確認了最終效果。之後發現測試集表現比預期低，決定再調整模型後重新測一次。這樣做有什麼問題？

答案：測試集已失去公正性（Data Leakage）。
理由：一旦根據測試集結果回去調整，測試集就不再是真正的未見資料，評估結果不再反映真實泛化能力。

---

**Q16.** 交叉驗證（Cross-Validation）最主要的用途是什麼？

答案：在資料量不大時，更穩定地估計模型的泛化能力（Generalization Ability）。
理由：多次切分輪流驗證，比一次性切分的評估更穩定可靠，適合資料量有限的情境。

---

**Q17.** 做完 5-fold 交叉驗證後，工程師認為已不需要額外的測試集。這個判斷正確嗎？

答案：不正確。
理由：交叉驗證有助比較和選擇模型，但交叉驗證的結果本身可能有選擇偏差；最終仍需要一個完全獨立的測試集做最後驗收。

---

### 10.6 類別不平衡與模型選擇

**Q18.** 詐欺偵測資料集中，詐欺案例只占 0.5%。工程師訓練後，模型 Accuracy 為 99.4%。這個結果能說明模型表現好嗎？

答案：不能直接下結論，需要看 Precision / Recall / F1-score。
理由：若模型把所有交易都預測為正常，Accuracy 仍可達 99.5%；應改用能反映少數類別偵測效果的指標。

---

**Q19.** 面對嚴重的類別不平衡問題，有哪三種常見的處理策略？

答案：重抽樣（Resampling）、類別權重（Class Weight）、換用 Precision / Recall / F1-score 等評估指標。
理由：三種策略分別從資料分布、訓練目標、評估方式三個角度應對不平衡問題。

---

**Q20.** 某個任務需要向業務主管解釋「為什麼模型做出這個預測」，應該優先考慮哪類模型？

答案：線性模型（Linear Model）或決策樹（Decision Tree）。
理由：這兩類模型可解釋性高，容易向非技術人員說明預測邏輯，符合「高可解釋性」需求。

---

**Q21.** 隨機森林（Random Forest）和決策樹（Decision Tree）相比，什麼情況下決策樹反而是更好的選擇？

答案：當業務需要高可解釋性時，決策樹反而更合適。
理由：決策樹的規則直觀易讀，隨機森林雖然效果更穩健，但多棵樹合在一起難以解釋。

---

**Q22.** 使用 SVM 之前，必須特別注意哪個前處理步驟？

答案：縮放（Scaling）。
理由：SVM 基於邊界距離計算，對特徵尺度高度敏感，尺度未處理會導致模型嚴重偏向尺度大的特徵。

---

### 10.7 綜合情境題

**Q23.** 工程師準備建立客戶流失預測模型。資料中有 95% 的客戶「未流失」，5% 「流失」。他應該特別注意哪些事？（至少說出兩點）

答案：（1）類別不平衡問題，不能只看 Accuracy；（2）應考慮重抽樣或類別權重；（3）評估指標應改用 Recall 或 F1-score。
理由：5% 的少數類別是嚴重不平衡，Accuracy 會被 95% 的多數類別主導而掩蓋模型真實表現。

---

**Q24.** 以下哪個步驟順序是正確的：
（A）全資料做 One-Hot 編碼 → 切分 → 訓練
（B）切分 → 全資料做 One-Hot 編碼 → 訓練
（C）切分 → 只用訓練集學 One-Hot 規則 → 套到 Val / Test → 訓練

答案：（C）。
理由：無論哪種前處理（包括 One-Hot），都應先切分資料集，再對各子集做前處理。One-Hot 雖然不涉及數值統計量（mean/std），但切分前做仍可能讓 encoder 的 categories 設定看到完整類別空間（包含 test set 的類別），且流程不一致會埋下壞習慣。統一規則：先切分，再做所有轉換。（A）和（B）都把全資料先做了，（B）甚至在切分後仍對全資料做，都違反正確順序。

---

**Q25.** 某公司選擇使用神經網路，理由是「它最強」。根據本課邏輯，這個決策過程有什麼問題？

答案：模型選擇應基於任務型態、資料特性、商業需求、成本限制四個軸做 trade-off，而不是只看「誰最強」。
理由：神經網路需要大量資料、算力與維護成本；若資料量小或需要高可解釋性，選神經網路反而不合理。

## Final Oral Recall

考前最後 3 分鐘，把這幾句唸一次：

1. **資料收集：** 先看「任務對齊、代表性、標籤品質、合法授權」— 資料多不代表資料對。

2. **缺失值補值：** 有極端值時用中位數（Median），無極端值用平均數（Mean）；缺失比例高且資料珍貴才補值，不是直接刪。

3. **縮放順序：** 先切分，再由訓練集學轉換規則，再套到驗證集和測試集 — 反過來做就是 Data Leakage。

4. **類別編碼：** 有順序用序位編碼（Ordinal Encoding）；無順序、類別不多用 One-Hot；高基數考慮目標編碼（Target Encoding），但切分前不能算。

5. **資料切分：** 訓練集學習、驗證集選模型、測試集最後驗收 — 測試集只能用一次，一直看就失去公正性。

6. **類別不平衡：** 看到詐欺、流失、異常、罕見疾病，先懷疑 Accuracy 是陷阱 — 要改看 Precision / Recall / F1-score。

7. **模型選擇：** 不是選「誰最強」，是看任務 × 資料 × 商業需求 × 成本 — 高解釋性選線性或決策樹，表格資料選隨機森林，大資料複雜模式才考慮神經網路。

## Final Study Advice

L21301 的考題核心是「判斷能力」，不是「記公式」。

遇到情境題，先跑這個思維順序：
**先看任務 → 再看資料有什麼問題 → 決定前處理策略 → 最後才選模型 trade-off。**

最常被考到的三個「看到就反應」規則：
- 看到 SVM / k-NN → 想到縮放敏感
- 看到詐欺 / 流失 / 異常 → 想到類別不平衡，Accuracy 是陷阱
- 看到「切分前做轉換」→ 想到 Data Leakage

把這三個反應練到直覺，本課大多數題目都能正確判斷。
