## 1. Big Picture / Core Pipeline 🔥🔥🔥

### 先懂一句話

L22404 的主軸是「資料層保護」。考試通常不是問模型卡或模型生命週期，而是問資料在蒐集、發布、分析、訓練、儲存、傳輸、合規管理時，該用哪一種保護方式。

### Everyday Analogy

把資料想成一批重要文件。公開前要遮掉能認人的資訊，統計時要避免某個人的影響被看出來，合作訓練時文件最好不離開原單位，儲存與傳輸時要上鎖，最後還要符合公司與法律流程。

### 先問自己一個問題

題目到底在保護哪一個位置的資料？

```text
資料要公開？      → 先想匿名化
資料只輸出統計？  → 先想差分隱私
資料不能集中？    → 先想聯邦學習
資料在儲存傳輸？  → 先想加密
資料誰可以看？    → 先想存取控制
資料是否合法？    → 先想合規制度
```

### 技術說法

本課的技術不是在回答「模型準不準」，而是在回答「資料怎麼被保護」。考試會把保護位置藏在情境文字裡，你要先抓到資料目前處於哪個階段，再選控制方法。

### 流程 / 選擇流程

```text
蒐集目的與合法基礎
→ 資料最小化與分類
→ 匿名化 / 差分隱私
→ 聯邦學習 / 隱私計算
→ 加密、存取控制、金鑰管理
→ 法規與管理制度稽核
```

### 比較表這樣讀

這張表不要逐格硬背。先看左欄的「題目問的是」，把它當成題目關鍵字；右欄就是最可能要選的控制。

| 題目問的是 | 想到 |
|---|---|
| 公開資料但不想被認出是誰 | k-匿名性、匿名化 |
| 同一匿名群內敏感值太一致 | ℓ-多樣性 |
| 群內敏感值分布和全體差太多 | t-相近性 |
| 加雜訊保護統計輸出 | 差分隱私 DP（Differential Privacy，差分隱私） |
| 同一資料集多次 DP 查詢 | 隱私預算組合 |
| 各醫院、各裝置資料不能離開本地 | 聯邦學習 FL（Federated Learning，聯邦學習） |
| 伺服器只能看聚合結果 | 安全聚合 |
| 儲存中資料保護 | 靜態加密、AES（Advanced Encryption Standard，進階加密標準）-256 |
| 傳輸中資料保護 | TLS（Transport Layer Security，傳輸層安全）1.3 |
| 密文狀態下仍可運算 | 同態加密 |
| 誰可以看哪些欄位 | RBAC（Role-Based Access Control，角色式存取控制）、ABAC（Attribute-Based Access Control，屬性式存取控制）、欄位層級安全 |
| Privacy by Design / Default | GDPR（General Data Protection Regulation，一般資料保護規則）Art.25 |
| 72 小時外洩通報 | GDPR Art.33 |
| 隱私資訊管理系統 | ISO/IEC 27701:2025、PIMS（Privacy Information Management System，隱私資訊管理系統） |

本課和 L23401 的邊界要記清楚：

```text
資料集在訓練前、共享前、分析前如何被保護 → L22404
模型訓練完成後的模型卡、推論稽核、生命週期文件 → L23401
```

### 記憶方式

```text
公開前 = 匿名化
統計輸出 = 差分隱私
不離本地 = 聯邦學習
看不懂 = 加密
誰能看 = 存取控制
制度要求 = 合規
```

### Exam Rule

```text
資料發布前降低重識別風險 → 匿名化
統計查詢加入雜訊 → 差分隱私
資料不離院、不離端 → 聯邦學習
儲存中 / 傳輸中 / 密文運算 → 加密類型判斷
法律要求轉成控制措施 → 個資法、GDPR（General Data Protection Regulation，一般資料保護規則）、ISO 27701
```

### Quick Check

某題描述「多家醫院共同訓練模型，但病歷不能離開各醫院」，最先想到哪個概念？

答案：聯邦學習（Federated Learning, FL）。因為核心需求是資料保留在本地，只上傳模型更新或聚合資訊。

## 2. 匿名化三層防護 🔥🔥🔥

### 先懂一句話

匿名化不是只刪掉姓名，而是降低從資料重新識別個人的風險。三個常考層次是：k-匿名性防「認出是誰」、ℓ-多樣性防「猜出敏感屬性」、t-相近性防「從分布偏斜推論」。

### Everyday Analogy

想像公布班上成績資料。只把姓名拿掉還不夠，因為「三年二班、座號、社團、身高」可能還是能猜出某人；如果同一群裡所有人的成績都一樣低，也仍然洩漏敏感資訊。

### 先問自己一個問題

題目是在問「認得出是誰」，還是「雖然認不出人，但猜得出敏感事情」？

```text
認得出是誰？              → k-匿名性不夠
猜得出疾病 / 薪資？       → ℓ-多樣性不夠
群內分布太偏、太不像全體？ → t-相近性不夠
```

### 技術說法

匿名化（Anonymization）是在資料發布或共享前，降低重新識別（re-identification）風險。準識別欄位（quasi-identifiers）不是姓名，但組合起來可能指向某個人；敏感屬性（sensitive attribute）則是你不希望被推論出的資訊。

### 流程 / 選擇流程

```text
原始資料
→ 找出準識別欄位與敏感欄位
→ 一般化 / 抑制
→ 檢查 k、ℓ、t
→ 發布資料或提供分析
```

### 比較表這樣讀

先用一句話分清楚三層：

```text
k-匿名性（k-Anonymity）= 至少混在 k 個人裡
ℓ-多樣性（ℓ-Diversity）= 同一群裡敏感值不能太單一
t-相近性（t-Closeness）= 同一群的敏感值分布不能和全體差太遠
```

準識別欄位（quasi-identifiers）是單獨看不一定能識別個人，但組合起來可能認出人的欄位，例如年齡、郵遞區號、性別、地區、職稱。敏感屬性（sensitive attribute）是疾病、薪資、信用狀態、政治或健康資訊等不希望被推論出的欄位。

| 概念 | 白話定義 | 主要防什麼 | 常見限制 |
|---|---|---|---|
| k-匿名性（k-Anonymity） | 每個等價類至少有 k 筆資料 | 身份揭露（identity disclosure） | 不能單獨防屬性洩漏 |
| ℓ-多樣性（ℓ-Diversity） | 每個等價類的敏感值至少有足夠多樣性 | 屬性揭露（attribute disclosure） | 多樣不代表分布合理 |
| t-相近性（t-Closeness） | 等價類敏感值分布要接近整體分布 | 相似性 / 偏斜攻擊 | 保護更強，但資料可用性可能下降 |

```text
k-anonymity
  解決：是不是唯一那個人
ℓ-diversity
  解決：雖然不是唯一，但敏感值都一樣
t-closeness
  解決：雖然有多種值，但分布太偏
```

### 一步一步例子

**k=2 範例：先看準識別欄位，不看疾病欄位**

| 年齡 | 郵遞區號 | 性別 | 疾病 |
|---|---|---|---|
| 20-29 | 1061* | 女 | 流感 |
| 20-29 | 1061* | 女 | 憂鬱症 |
| 40-49 | 2204* | 男 | 糖尿病 |
| 40-49 | 2204* | 男 | 高血壓 |

檢查步驟：

```text
第 1 組：20-29 + 1061* + 女 → 2 筆
第 2 組：40-49 + 2204* + 男 → 2 筆
所以每個等價類（equivalence class）至少有 2 筆 → 符合 k=2
```

但要注意：符合 `k=2` 不代表完全安全。如果第 1 組兩筆疾病都一樣，就會變成 ℓ-多樣性不足。

常見匿名化操作：

| 操作 | 意思 | 例子 |
|---|---|---|
| 一般化（Generalization） | 把精確值改成較粗範圍 | `27` → `20-29` |
| 抑制（Suppression） | 遮掉過細或高風險值 | `10617` → `1061*` |

注意：雜湊（hashing）不自動等於匿名化。若原始值範圍小或可被字典攻擊，仍可能被反查。

#### 假名化 vs 匿名化

| 概念 | 定義 | 可逆性 | GDPR 判定 |
|---|---|---|---|
| **假名化（Pseudonymisation）** | 以假名取代識別資訊，但保留可逆的對應表 | 可逆（有對應表即可還原） | 仍屬個人資料 |
| **匿名化（Anonymization）** | 完全無法識別個人，不可逆 | 不可逆 | 不再屬個人資料 |

GDPR 明確指出假名化資料仍屬個人資料（因為可逆），但假名化被視為降低風險的技術措施（Art.32 提及）。考試判斷：「保留對應表、可還原」→ 假名化；「完全不可逆」→ 匿名化。

### 記憶方式

```text
k = 幾個人混在一起
ℓ = 敏感值有幾種
t = 群內分布像不像全體

假名化 = 有鑰匙可還原
匿名化 = 沒有鑰匙可還原
```

### Exam Rule

```text
防止身份揭露 / 每組至少 k 人 → k-匿名性
同一匿名群內敏感值都一樣 → ℓ-多樣性
群內分布和整體分布差太多 → t-相近性
27 改 20-29 / 郵遞區號遮星號 → 一般化或抑制
只刪姓名仍可重識別 → 匿名化不足
保留對應表、可還原 → 假名化（仍屬個人資料）
完全不可逆 → 匿名化（不再屬個人資料）
```

### Quick Check

某資料表符合 `k=3`，但某個等價類裡 3 筆資料的疾病全部都是同一種。這主要違反哪個概念？

答案：ℓ-多樣性（ℓ-Diversity）。因為雖然身份不唯一，但敏感屬性太一致，仍可能被猜出疾病。

## 3. 差分隱私（Differential Privacy, DP）🔥🔥🔥

### 先懂一句話

差分隱私是在保護「統計輸出」。它希望資料集中有沒有某一個人的資料，都不會讓查詢結果改變到足以推論那個人的隱私。

### Everyday Analogy

像公布全校最常點的午餐排行榜。大家可以知道整體趨勢，但不應該因為你今天有沒有買雞腿便當，就讓排行榜出現可被追蹤的明顯變化。

### 先問自己一個問題

題目是不是在問「只公布統計結果，但不能讓單一個人的加入或移除被看出來」？

```text
公布原始資料表？        → 先想匿名化
公布查詢結果 / 統計值？ → 先想差分隱私
使用者上傳前先擾動？    → 先想本地差分隱私
```

### 技術說法

差分隱私（Differential Privacy, DP）的核心不是把資料表變匿名，而是讓查詢輸出對單一個人的存在不敏感。直覺上，資料集中有你或沒有你，外部觀察者看到的統計結果都應該差不多。

### 流程 / 選擇流程

```text
原始資料
→ 設定查詢與隱私預算
→ 加入隨機雜訊
→ 輸出統計結果
→ 追蹤累積隱私損失
```

### 比較表這樣讀

DP 題最重要先看 `ε`：它不是準確率，而是隱私損失預算。

| 概念 | 意思 | 考試判讀 |
|---|---|---|
| ε（epsilon） | 隱私損失預算 | 越小越私密、雜訊越多；越大越精準、保護越弱 |
| δ（delta） | 失敗機率 | 越小代表偏離理想 DP 保證的機率越低 |
| 拉普拉斯機制 | 常用於數值查詢 | 人數、總和、平均等 |
| 高斯機制 | 常用於向量或 `(ε,δ)`-DP | 多維統計、梯度、機器學習 |
| 本地差分隱私（Local DP） | 資料上傳前先在裝置端擾動 | 伺服器不直接看原始答案 |

```text
epsilon 小 -----------------------> epsilon 大
保護強、雜訊多                     保護弱、雜訊少

delta 小 -------------------------> delta 大
失敗機率低                         失敗機率高
```

### 一步一步例子

**隱私預算組合（privacy budget composition）**

同一份資料集連續做多次 DP 查詢時，隱私損失會累加。

```text
順序組合：
ε_total = ε1 + ε2 + ... + εn
δ_total = δ1 + δ2 + ... + δn
```

例：同一份員工資料做三次查詢，`ε = 0.3、0.4、0.5`，總隱私損失是：

```text
ε_total = 0.3 + 0.4 + 0.5 = 1.2
```

如果總預算上限是 `1.0`，這三次一起做就超標。

平行組合要另外記：

```text
互斥資料子群 → 取最大 ε
同一資料重複查詢 → ε 相加
```

**DP vs 匿名化**

| 比較 | 匿名化 | 差分隱私 |
|---|---|---|
| 保護對象 | 發布的資料表 | 查詢結果或統計輸出 |
| 常見做法 | 一般化、抑制、分組 | 加雜訊、控制隱私預算 |
| 常考語境 | 資料公開前 | 統計查詢、分析介面、ML 訓練 |

### 記憶方式

```text
DP = 在答案上加霧
ε 小 = 霧很濃，隱私強，答案較不準
ε 大 = 霧很淡，答案較準，隱私弱
同一批資料一直問 = 霧的保護額度會被用掉
```

### Exam Rule

```text
加雜訊保統計 → 差分隱私
ε 越小 → 隱私越強、雜訊越多
ε 越大 → 統計越準、隱私越弱
同一資料集多次查詢 → 預算累加
互斥子群分別查詢 → 平行組合取最大值
裝置端先擾動再上傳 → Local DP
```

### Quick Check

同一資料集做兩次 DP 查詢，`ε=0.2` 和 `ε=0.4`。若是順序組合，總 `ε` 是多少？

答案：`0.6`。因為同一份資料集連續查詢時，順序組合的隱私預算要相加。

## 4. 聯邦學習（Federated Learning, FL）🔥🔥

### 先懂一句話

聯邦學習讓資料留在本地端或各機構內，只把模型更新送去中央伺服器聚合。它降低資料集中外流風險，但不代表完全沒有隱私風險。

### Everyday Analogy

像多家醫院一起寫研究報告，但病歷不能帶出醫院。每家醫院只交出整理後的模型更新，中央再把各家的結果合成新的共同版本。

### 先問自己一個問題

題目是不是一直強調「資料不能離開本地、醫院、手機、分行或機構」？

```text
資料可以集中到一處？      → 不一定需要 FL
資料不能離開各機構？      → 聯邦學習
中央不能看單一更新？      → 安全聚合
同一批人但欄位不同？      → 縱向 FL
不同人但欄位相似？        → 橫向 FL
```

### 技術說法

聯邦學習（Federated Learning, FL）是一種分散式模型訓練方式。每個客戶端（client）用本地資料訓練，中央伺服器只收模型更新（model updates）並聚合成全域模型（global model）。

### 流程 / 選擇流程

```text
中央下發初始模型
→ 各客戶端用本地資料訓練
→ 上傳模型更新
→ 伺服器用 FedAvg / Secure Aggregation 聚合
→ 產生新的全域模型
```

### 比較表這樣讀

這張表先看「資料在哪裡」與「傳出去的是什麼」。FL 的考點通常不是演算法細節，而是資料不集中、更新要聚合。

| 概念 | 重點 | 常見考法 |
|---|---|---|
| Federated Learning | 資料不離本地，只傳模型更新 | 醫院、手機、分行資料不能集中 |
| FedAvg | 依各客戶端資料量加權平均模型 | `n_k / N` 加權概念 |
| Secure Aggregation | 伺服器只看聚合結果，不看單一更新 | 防止單一客戶端更新洩漏 |
| Horizontal FL | 欄位相似、資料列不同 | 不同醫院病人、不同手機使用者 |
| Vertical FL | 同一批人、不同欄位 | 銀行有交易資料，電商有購物資料 |

```text
Horizontal FL：
機構 A: [客戶1, 客戶2] x [年齡, 收入, 地區]
機構 B: [客戶3, 客戶4] x [年齡, 收入, 地區]
→ 列不同，欄位相似

Vertical FL：
銀行: [客戶1, 客戶2] x [收入, 信貸]
電商: [客戶1, 客戶2] x [購物頻率, 客單價]
→ 人相同，欄位不同
```

### 一步一步例子

**FedAvg（Federated Averaging）直覺**

假設三家醫院各自訓練後送出模型更新，資料量不同：

```text
A 醫院：1000 筆
B 醫院：500 筆
C 醫院：500 筆
總資料量 N = 2000
```

中央聚合時，A 醫院權重比較大：

```text
A 權重 = 1000 / 2000 = 0.5
B 權重 = 500 / 2000 = 0.25
C 權重 = 500 / 2000 = 0.25
```

所以 FedAvg 的直覺是「資料量多的客戶端，模型更新影響較大」。

重要陷阱：梯度或更新向量仍可能洩漏資訊，所以 FL 常搭配安全聚合、差分隱私、加密等機制。

### 記憶方式

```text
FL = 資料留在家，只寄作業心得
Secure Aggregation = 老師只看全班平均，不看單一學生答案
Horizontal = 欄位一樣，人不同
Vertical = 人一樣，欄位不同
```

### Exam Rule

```text
資料不能離院 / 不離端 → 聯邦學習
伺服器只看總和或平均，不看個別更新 → 安全聚合
欄位相似、不同人 → 橫向聯邦學習
同一批人、欄位不同 → 縱向聯邦學習
資料不出端但仍可能被梯度反推 → FL 不是絕對安全
```

### Quick Check

銀行和電商掌握同一批客戶，但銀行有信用資料，電商有購物資料。這比較像哪種聯邦學習？

答案：縱向聯邦學習（Vertical FL）。因為資料主體相同，但各方掌握的欄位不同。

## 5. 加密、安全儲存與存取控制 🔥🔥🔥

### 先懂一句話

加密保護的是「看不看得懂」，存取控制保護的是「誰可以碰」。考試常要求你分辨資料在儲存中、傳輸中、運算中，分別該用什麼技術。

### Everyday Analogy

資料像文件。放在櫃子裡要鎖，是靜態加密；寄送路上要用密封信封，是傳輸加密；文件還鎖在盒子裡就能計算結果，是同態加密；誰拿得到鑰匙和能看哪幾頁，是存取控制。

### 先問自己一個問題

資料現在是在「放著」、「路上」、「計算中」，還是在問「誰可以看」？

```text
放在資料庫 / 磁碟 / 備份？  → 靜態加密
走 API / 網路 / 服務間通訊？ → 傳輸加密
加密狀態下仍要計算？        → 同態加密
依角色或條件決定權限？      → 存取控制
```

### 技術說法

加密（encryption）把資料變成未授權者看不懂的密文；存取控制（access control）決定誰能讀、寫、查詢或看到哪些欄位。兩者常一起用，但不是同一件事。

### 流程 / 選擇流程

```text
資料產生
→ 傳輸加密保護上傳與交換
→ 靜態加密保護資料庫、磁碟、備份
→ 存取控制限制人與系統權限
→ 金鑰管理維持加密可信度
→ 需要密文運算時使用同態加密
```

### 比較表這樣讀

先看資料狀態，再選加密類型。

| 類型 | 主要用途 | 代表技術 | 題目關鍵字 |
|---|---|---|---|
| 靜態加密（Encryption at Rest） | 保護儲存中的資料 | AES-256 | 磁碟、資料庫、備份、物件儲存 |
| 傳輸加密（Encryption in Transit） | 保護網路傳輸中的資料 | TLS 1.3 | API、上傳下載、服務間通訊 |
| 同態加密（Homomorphic Encryption, HE） | 不解密仍可在密文上運算 | HE schemes | 加密狀態下做運算、外包計算 |

```text
普通加密：
明文 → 加密 → 密文 → 解密 → 運算

同態加密：
明文 → 加密 → 密文 → 直接運算 → 解密結果
```

| 存取控制 | 判斷方式 | 例子 |
|---|---|---|
| RBAC（Role-Based Access Control，角色式存取控制） | 依角色給權限 | 資料工程師、分析師、客服 |
| ABAC（Attribute-Based Access Control，屬性式存取控制） | 依屬性、資源、環境條件判斷 | 法務部 + 公司設備 + 上班時間 |
| 欄位層級安全 | 不同角色看不同欄位 | 分析師看年齡區間，看不到身分證字號 |

金鑰管理（Key Management）常見考點：

```text
金鑰不要和資料放同一處
定期金鑰輪替（key rotation）
高價值場景使用 HSM（Hardware Security Module，硬體安全模組）保護主金鑰
```

### 記憶方式

```text
At Rest = 放著也要鎖
In Transit = 路上要密封
Homomorphic = 鎖著也能算
RBAC = 看角色
ABAC = 看條件
Column-Level = 看欄位
```

### Exam Rule

```text
AES-256 / 資料庫 / 備份檔 → 靜態加密
TLS 1.3 / API / 網路傳輸 → 傳輸加密
不解密仍可運算 → 同態加密
依角色授權 → RBAC
依使用者、資源、環境條件授權 → ABAC
只遮某些欄位 → 欄位層級安全
金鑰輪替 / HSM → 金鑰管理
```

### Quick Check

某公司希望雲端服務商在看不到原始明文資料的情況下完成計算，應優先想到哪個技術？

答案：同態加密（Homomorphic Encryption, HE）。因為題目關鍵是「密文狀態下仍可運算」。

## 6. 法律合規與管理制度 🔥🔥🔥

### 先懂一句話

合規題不是只背法條號碼，而是要把法律要求翻成技術與組織措施。例如資料最小化、同意管理、權限控管、加密、稽核、事故通報。

### Everyday Analogy

像辦活動收報名資料。你不能因為方便就收一堆無關資料；要說明用途、限制誰能看、保存多久、外洩怎麼通知，還要有內部流程證明你真的有管理。

### 先問自己一個問題

題目是在問「可以收什麼」、「要怎麼保護」、「外洩怎麼通知」，還是「用哪個管理制度」？

```text
收資料不能超過目的？        → 個資法第 5 條 / 資料最小化
非公務機關蒐集處理？        → 個資法第 19 條
外洩通知當事人？            → 個資法第 12 條
設計與預設就要保護隱私？    → GDPR Art.25
處理安全、加密、復原？      → GDPR Art.32
72 小時通報主管機關？       → GDPR Art.33
隱私資訊管理系統？          → ISO/IEC 27701
```

### 技術說法

合規（compliance）不是只背法條，而是把要求落成控制措施（controls）。例如「目的必要範圍」會對應資料最小化；「處理安全」會對應加密、假名化、韌性、復原測試；「管理系統」會對應制度、角色、稽核與持續改善。

### 流程 / 選擇流程

```text
確認蒐集目的
→ 判斷合法基礎
→ 只收必要欄位
→ 設計預設保護與安全措施
→ 留存紀錄、稽核、通報與持續改善
```

### 比較表這樣讀

這張表用法是：左欄看題目出現哪個法規或標準，中欄抓考點，右欄把它翻成實務控制。

| 法規 / 標準 | 高頻考點 | 技術或管理對應 |
|---|---|---|
| 個資法第 5 條 | 比例原則、特定目的必要範圍 | 資料最小化、欄位裁剪、保存期限 |
| 個資法第 19 條 | 非公務機關蒐集或處理個資的合法基礎 | 同意管理、資料來源紀錄、目的標註 |
| 個資法第 12 條 | 個資被竊取、竄改、毀損、滅失或洩漏時通知當事人 | 事故通報、應變流程 |
| 個資法第 27 條 | 舊版常考「適當安全措施」 | 舊教材題目仍可能用它指安全維護 |
| 施行細則第 12 條 | 技術上與組織上安全維護措施 | 風險評估、稽核、紀錄保存、教育訓練 |
| GDPR Art.25 | Privacy by Design / by Default | 預設最小蒐集、最小權限、隱私設計 |
| GDPR Art.32 | Security of Processing | 加密、假名化、CIA（Confidentiality, Integrity, Availability，機密性、完整性、可用性）、韌性、復原測試 |
| GDPR Art.33 | 72 小時外洩通報主管機關（有前提條件，見下方注意）| personal data breach、supervisory authority |
| ISO/IEC 27701:2025 | PIMS 隱私資訊管理系統 | 2025 版設計為可獨立使用的管理系統標準（2019 版不可獨立使用）；仍可與 27001/27002 整合 ⚠️ |

版本提醒：原始 guide 提到 2025-11-11 修法已公告刪除個資法第 27 條但施行日未定。考試若沿用舊教材，第 27 條仍常代表「非公務機關適當安全措施」；若題目明示最新修法，則要看新版條文架構。

> **GDPR Art.33 注意**：72 小時通報義務有前提條件。GDPR Art.33 規定，**「除非該外洩不太可能對自然人的權利和自由造成風險」**，否則才需要在 72 小時內向主管機關通報。並非所有資料外洩都必須通報。考試常見選項以「72 小時」為標準答案；若遇到對比選項「任何外洩都必須通報」vs「視風險程度決定是否通報」，後者更精確。

#### 被遺忘權 / 刪除權（Right to Erasure, GDPR Art.17）

資料主體在特定條件下，有權要求資料控管者刪除其個人資料（例如：蒐集目的已消失、撤回同意、資料被非法處理）。

大數據系統的挑戰：
- 資料已複製到多個節點、備份或二級分析系統，逐一刪除技術上複雜。
- 如果資料已進入機器學習模型權重，刪除「模型記憶」目前仍是研究課題（machine unlearning）。
- 實務上需設計能依請求定位並刪除特定個人資料的管線。

考試判斷：「資料主體要求刪除、不再使用其個資」→ GDPR Art.17 被遺忘權。

罰鍰記憶以考試層級抓方向即可：

| 義務類型 | 常見方向 |
|---|---|
| 第 19 條等核心蒐集處理義務 | 較高罰鍰級距，原 guide 提到現行公告版本為 5 萬到 50 萬 |
| 第 12 條通知義務等 | 較低罰鍰級距，原 guide 提到現行公告版本為 2 萬到 20 萬 |
| 已公告但未生效的新修法 | 安全維護計畫等義務可能提高到 15 萬到 1,500 萬級距 |

### 記憶方式

```text
第 5 條 = 不能收太多
第 19 條 = 要有合法基礎
第 12 條 = 外洩通知當事人
GDPR 25 = 設計和預設就保護
GDPR 32 = 處理安全
GDPR 33 = 72 小時通報主管機關（有風險條件）
GDPR 17 = 刪除權 / 被遺忘權
ISO 27701 = PIMS 隱私管理系統
```

### Exam Rule

```text
蒐集不得超過目的必要範圍 → 個資法第 5 條
非公務機關蒐集處理要有合法基礎 → 個資法第 19 條
外洩通知當事人 → 個資法第 12 條
舊教材適當安全措施 → 個資法第 27 條
技術與組織安全措施清單 → 施行細則第 12 條
隱私從設計和預設開始 → GDPR Art.25
處理安全、加密、韌性、復原 → GDPR Art.32
72 小時通報主管機關（視風險條件）→ GDPR Art.33
資料主體要求刪除個資 → GDPR Art.17 被遺忘權
PIMS / 2025 可獨立使用 → ISO/IEC 27701:2025
```

> ⚠️ **ISO/IEC 27701 版本提醒**：ISO/IEC 27701:**2019** 需以 ISO 27001 為基礎，**不可獨立使用**。若文中或題目提及 **2025 年修訂版**允許獨立使用，應確認該版本已正式發布（非 DIS/FDIS 草稿）後再引用。考試若題目明確指 2019 版，請選「需以 27001 為基礎」；若明確指 2025 版，則可選「可獨立使用」。

### Quick Check

題目出現「系統預設只開啟最小必要個資處理與最小權限」，最接近 GDPR 哪一條？

答案：GDPR Art.25。因為這是在考 Privacy by Design / by Default，不是事後補救。

## 7. 情境選擇：看到需求先選控制 🔥🔥🔥

### 先懂一句話

情境題的第一步不是背定義，而是抓題目要求的保護位置：發布前、查詢時、訓練時、儲存傳輸時、授權時、合規時。

### Everyday Analogy

像修水管要先看漏水點在哪裡。水還沒流出前、管線中、儲水槽、使用者水龍頭，對應的修法都不一樣。

### 先問自己一個問題

題目最強的關鍵字是哪一類？

```text
公開 / 去識別 / 重識別       → 匿名化
統計查詢 / 加雜訊 / ε        → 差分隱私
不能離院 / 不離端 / 多方訓練 → 聯邦學習
AES / TLS / 密文運算         → 加密
角色 / 屬性 / 欄位           → 存取控制
條文 / 通報 / PIMS           → 合規
```

### 技術說法

情境選擇題本質上是在做控制選擇（control selection）。不要先背定義，先找風險位置：識別風險、推論風險、集中風險、竊聽風險、未授權存取風險，或法規遵循風險。

### 流程 / 選擇流程

```text
先看資料在哪個階段
→ 判斷風險是識別、推論、集中、竊聽、未授權或法規違反
→ 選對技術或管理控制
```

### 比較表這樣讀

這張表是考場用的「需求 → 控制」對照表。左欄像題目描述，中間欄看資料進出，右欄才是答案。

| 任務 / 場景 | 輸入 | 輸出 | 常見答案 |
|---|---|---|---|
| 公開資料集前降低重識別 | 原始表格 | 去識別後表格 | k-匿名性、ℓ-多樣性、t-相近性 |
| 統計查詢但不暴露個人影響 | 查詢與資料集 | 加雜訊統計結果 | 差分隱私 |
| 使用者端先擾動再上傳 | 本地原始資料 | 私有化資料 | Local DP |
| 多院共同訓練但資料不離院 | 各院本地資料 | 全域模型 | 聯邦學習 |
| 伺服器不能看個別更新 | 客戶端更新 | 聚合更新 | Secure Aggregation |
| 資料庫、備份、硬碟保護 | 儲存資料 | 密文資料 | AES-256 / at rest |
| API 或服務間傳輸保護 | 網路資料流 | 加密通道 | TLS 1.3 / in transit |
| 外包計算但不暴露明文 | 加密資料 | 解密後結果 | 同態加密 |
| 不同人只能看不同資料 | 使用者與資源 | 授權決策 | RBAC / ABAC / 欄位層級安全 |
| 隱私管理制度建置 | 組織流程 | PIMS | ISO/IEC 27701:2025 |

### 記憶方式

```text
先判斷資料在哪裡
再判斷怕什麼
最後才選名詞
```

常見配對：

```text
公開資料怕認人 → k / ℓ / t
統計結果怕推個人 → DP
訓練資料不能集中 → FL
資料怕被看懂 → encryption
權限怕太大 → RBAC / ABAC
流程怕不合規 → PDPA / GDPR / ISO
```

### Exam Rule

```text
先判斷保護階段，再選工具
公開資料 → 匿名化
統計輸出 → 差分隱私
分散訓練 → 聯邦學習
儲存 / 傳輸 / 運算 → 加密類型
誰能看 → 存取控制
法律義務 → 合規控制
```

### Quick Check

題目說「同一份資料會被查詢很多次，每次都符合 DP，所以一定不會超過風險」。這句話哪裡錯？

答案：忽略隱私預算組合。同一份資料多次 DP 查詢時，總隱私損失會累加，可能超過預算上限。

## 9. Trap Clinic

### Trap 1：刪掉姓名就等於匿名化

錯。姓名只是直接識別欄位，年齡、郵遞區號、性別、職稱等準識別欄位組合起來仍可能重識別。

Exam fix：

```text
只刪姓名仍可猜出個人 → 匿名化不足，檢查準識別欄位
```

### Trap 2：k-匿名性能防所有隱私洩漏

錯。k-匿名性主要防身份揭露，若同一等價類敏感值都一樣，仍會發生屬性揭露。

Exam fix：

```text
群內敏感值都一樣 → ℓ-多樣性
```

### Trap 3：ε 越大代表差分隱私越安全

錯。ε 越小，隱私越強；ε 越大，雜訊較少、結果較準，但保護較弱。

Exam fix：

```text
ε 小 → 隱私強
ε 大 → 精度高但隱私弱
```

### Trap 4：每次查詢都有 DP，所以查幾次都沒關係

錯。同一份資料連續查詢時，隱私預算會累加。

Exam fix：

```text
同一資料集多次 DP 查詢 → sequential composition，ε 相加
```

### Trap 5：聯邦學習資料不出端，所以絕對不會洩漏

錯。模型更新或梯度仍可能洩漏資訊，因此常需要安全聚合、差分隱私或加密搭配。

Exam fix：

```text
資料不離端但更新可能洩漏 → FL 仍需 Secure Aggregation / DP
```

### Trap 6：同態加密只是傳輸加密

錯。同態加密的重點是密文可運算；傳輸加密的重點是網路通道保護。

Exam fix：

```text
不解密仍可運算 → Homomorphic Encryption
網路傳輸保護 → TLS
```

### Trap 7：加密等於存取控制

錯。加密讓未授權者看不懂資料；存取控制決定誰可以存取資料。

Exam fix：

```text
看不懂 → 加密
誰能看 → RBAC / ABAC / 欄位層級安全
```

### Trap 8：72 小時外洩通報就是台灣個資法

錯。72 小時是 GDPR Art.33 的經典考點。台灣個資法第 12 條重點是外洩通知當事人，不能直接套成 72 小時。

Exam fix：

```text
72 小時 + supervisory authority → GDPR Art.33
台灣個資外洩通知當事人 → 個資法第 12 條
```

### Trap 9：ISO/IEC 27701 可以隨便選哪個版本

注意版本：**2019 版**需要 ISO 27001 為基礎，**不可獨立使用**；**2025 版**設計為可獨立使用，但仍可與 27001/27002 整合。

Exam fix：

```text
ISO/IEC 27701:2019 → 需 ISO 27001 基礎，不可獨立使用
ISO/IEC 27701:2025 → PIMS，可獨立使用的 MSS（確認已正式發布後引用）
```

## 10. Practice Questions

### 10.1 匿名化

**Q1.** 一張公開資料表中，每個「年齡區間 + 郵遞區號 + 性別」組合至少有 5 筆資料。這最接近哪個概念？

答案：k-匿名性（k-Anonymity）。
理由：題目重點是每個等價類至少有 k 筆，避免唯一識別。

**Q2.** 某資料表符合 `k=4`，但某一組 4 人的疾病欄位全部相同。主要風險是什麼？

答案：屬性揭露（attribute disclosure）。
理由：雖然不知道是哪一個人，但可以推論該群人的敏感疾病。

**Q3.** 等價類內有多種疾病，但疾病分布和整體資料相比極度偏斜，應想到哪個概念？

答案：t-相近性（t-Closeness）。
理由：t-相近性要求等價類敏感屬性分布接近全體分布。

**Q4.** 把年齡 `27` 改成 `20-29`，這是哪一種匿名化操作？

答案：一般化（Generalization）。
理由：精確值被改成較粗的範圍。

### 10.2 差分隱私

**Q5.** DP 中 `ε` 越小代表什麼？

答案：隱私保護越強，通常雜訊越多。
理由：小 ε 限制單一個人對輸出的可觀察影響。

**Q6.** 同一資料集做三次 DP 查詢，ε 分別是 `0.1`、`0.2`、`0.4`，順序組合的總 ε 是多少？

答案：`0.7`。
理由：同一資料集多次查詢的 ε 要相加。

**Q7.** 對三個互斥子群分別用 `ε=0.5` 查詢，平行組合下整體 ε 通常是多少？

答案：`0.5`。
理由：互斥資料子群適用平行組合，取最大值而不是相加。

**Q8.** 使用者手機先把資料擾動後才送到伺服器，這是哪一種 DP？

答案：本地差分隱私（Local Differential Privacy, LDP）。
理由：隱私化在資料離開裝置前完成。

### 10.3 聯邦學習

**Q9.** 多家醫院想共同訓練模型，但病歷不能離院，應選哪個架構？

答案：聯邦學習（Federated Learning, FL）。
理由：FL 讓資料留在本地，只傳模型更新。

**Q10.** 伺服器只能看到多個客戶端更新的總和，看不到單一客戶端更新。這是什麼機制？

答案：安全聚合（Secure Aggregation）。
理由：它保護個別更新不被中央伺服器直接觀察。

**Q11.** 不同手機使用者都有相似欄位，但資料列是不同人，較像哪種 FL？

答案：橫向聯邦學習（Horizontal FL）。
理由：欄位相似，但資料主體不同。

**Q12.** 銀行和電商對應同一批客戶，但掌握不同欄位，較像哪種 FL？

答案：縱向聯邦學習（Vertical FL）。
理由：資料主體相同，但欄位不同。

### 10.4 加密與存取控制

**Q13.** 資料庫與備份檔要加密保存，常見代表技術是什麼？

答案：AES-256 / 靜態加密。
理由：資料處於儲存狀態，重點是 encryption at rest。

**Q14.** App 與 API 之間傳輸個資時，常見保護是什麼？

答案：TLS 1.3 / 傳輸加密。
理由：資料正在網路傳輸中，重點是 encryption in transit。

**Q15.** 不解密資料也能在密文上計算，應選哪個技術？

答案：同態加密（Homomorphic Encryption, HE）。
理由：HE 的核心特徵是密文可運算。

**Q16.** 權限依「部門、地點、時間、設備狀態」等條件動態判斷，較像 RBAC 還是 ABAC？

答案：ABAC。
理由：ABAC 依屬性與環境條件做授權，不只是角色。

### 10.5 合規與混合陷阱

**Q17.** 題目說「蒐集個資不得超過特定目的必要範圍」，對應個資法哪個高頻條文？

答案：個資法第 5 條。
理由：第 5 條常考比例原則與特定目的必要範圍。

**Q18.** 題目出現「Privacy by Design and by Default」，應選 GDPR 哪一條？

答案：GDPR Art.25。
理由：Art.25 的核心就是設計即保護與預設保護。

**Q19.** 題目出現「personal data breach、supervisory authority、72 hours」，應想到什麼？

答案：GDPR Art.33。
理由：72 小時通報主管機關是 Art.33 的典型關鍵字。

**Q20.** ISO/IEC 27701:2025 的定位是什麼？

答案：隱私資訊管理系統（PIMS）標準，且 2025 版可獨立使用。
理由：它不再只能簡化成 ISO/IEC 27001 的附錄，但仍可與 27001/27002 整合。

## Final Oral Recall

考前最後 3 分鐘，把這幾句唸一次：

1. L22404 是資料層保護，不是模型卡、推論稽核或模型生命週期文件。
2. 匿名化三層要記成：k 防身份、ℓ 防屬性、t 防分布。
3. 差分隱私是保護統計輸出，ε 越小越私密，同一資料集多次查詢會累加預算。
4. 聯邦學習是資料不離端，但模型更新仍可能洩漏，所以常搭配安全聚合或 DP。
5. AES-256 是儲存中，TLS 1.3 是傳輸中，同態加密是密文仍可運算。
6. GDPR Art.25 是設計與預設保護，Art.32 是處理安全，Art.33 是 72 小時通報。
7. ISO/IEC 27701:2025 是 PIMS，重點是可獨立使用，但仍可與 27001/27002 整合。

## Final Study Advice

考試時先抓題目關鍵字，再判斷資料處於哪個階段。公開前想匿名化，統計輸出想差分隱私，資料不離機構想聯邦學習，儲存傳輸運算想加密，誰能看想存取控制，條文與制度想合規控制。
