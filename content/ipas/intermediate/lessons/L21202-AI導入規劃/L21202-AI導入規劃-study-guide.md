## 1. Big Picture — AI 導入規劃的核心流程 🔥🔥🔥

### 整個課程的位置

這堂課不是在「選工具」（那是 L21201），也不是「管風險」（L21203）或「上線維運」（L21302）。

L21202 的任務只有一件事：**方案選定之後，規劃怎麼做到上線。**

```text
L21201 評估選工具
    ↓ Go/No-Go gate 通過
L21202 AI導入規劃  ← 你現在在這裡
    ↓ 規劃四大產出物完成 + Pilot gate 通過
L21302 部署與維運（MLOps, Machine Learning Operations，機器學習維運）
```

### L21202 的五階段導入路線圖（5-Phase Adoption Roadmap）

```text
P1 需求分析 Requirements
    → gate：利害關係人簽核、L1-L5 五層對應表完成
          ↓
P2 方案設計 Solution Design
    → gate：功能/資料/模型/整合 四面向藍圖 + 預算核准
          ↓
P3 資源規劃 Resource Plan
    → gate：RACI（Responsible / Accountable / Consulted / Informed，負責 / 當責 / 諮詢 / 告知）+ 排程 + 合約 + 預算分配表
          ↓
P4 試點 Pilot
    → gate：五條件全過才進 Rollout（指標 + 可接受度 + 風險 + 經濟 + SOP, Standard Operating Procedure，標準作業程序）
          ↓
P5 擴展 Rollout / Scale
    → gate：全量部署完成 → 移交 L21302
```

口訣記法：**需·方·資·試·擴**

### 考試速查表

| 題目問的是 | 想到 |
|---|---|
| 「需求分析」「業務流程對映 AI」 | 需求分析五層（業·流·決·資·技） |
| 「誰負責誰簽核」「跨部門分工」 | RACI（R 執行 / A 當責 / C 諮詢 / I 告知） |
| 「要多少錢、多少人、多少時間」 | 資源分配五向量（錢·人·算·料·時） |
| 「試點驗證」「小範圍真實部署」 | Pilot（P4 階段） |
| 「單一衡量整體進展的指標」 | 北極星指標 North Star Metric（只能一個） |
| 「選工具 / 算 ROI（Return on Investment，投資報酬率）/ TCO（Total Cost of Ownership，總持有成本）」 | 那是 L21201，不是本課 |
| 「EU AI Act / 風險分級」 | 那是 L21203，不是本課 |
| 「k8s / CI/CD（Continuous Integration / Continuous Delivery，持續整合 / 持續交付）/ 模型漂移」 | 那是 L21302，不是本課 |

## 2. 需求分析（Requirements Analysis）🔥🔥🔥

### 先懂一句話

需求分析就是把「我們想用 AI（Artificial Intelligence，人工智慧）」這句話，一層一層拆解成「用什麼資料、做什麼決策、解決哪段流程的問題」——共分五層，不能跳層。

### Everyday Analogy

你去診所看病，醫生不會一進門就說「你需要開刀」。他會先問症狀（L1），找到是哪個器官（L2），確認是什麼病因（L3），查哪些檢驗數值（L4），才決定用哪種藥或手術（L5）。直接跳到「開刀」就是跳層，可能會開錯。AI 導入的需求分析也一樣。

### 先問自己一個問題

在寫需求之前，先問：**我現在是在描述「業務問題」，還是在偷渡「技術答案」？**

如果一句話已經出現 ChatGPT、RAG（Retrieval-Augmented Generation，檢索增強生成）、GPU（Graphics Processing Unit，圖形處理器）、模型 fine-tune，通常代表你太快跳到 L5 技術層。考試看到這種題目，要先往回補 L1-L4。

### 技術說法

需求分析（Requirements Analysis）不是把需求寫成一串功能清單，而是把 business need 轉成 AI 可執行的 problem framing。最重要的工具是五層對應法：

```text
Business Goal → Process → Decision → Data → Technique
業務目標 → 流程 → 決策 → 資料 → 技術
```

### 流程 / 選擇流程

```text
L21201 選定方案
    → 進入 P1 需求分析（本節）
    → 產出：需求說明書（Requirements Doc）
    → gate 通過後 → P2 方案設計
```

### Key Concepts

#### 五層對應法（業·流·決·資·技）

```text
L1  Business Goal（業務目標）
    「為什麼做 AI？要解決什麼商業問題？」
         ↓
L2  Process（業務流程）
    「哪一段流程會被 AI 改善？」
         ↓
L3  Decision（決策點）
    「AI 要幫人做什麼判斷？回答什麼問題？」
         ↓
L4  Data（資料）
    「這個決策需要哪些輸入資料？輸出是什麼？標註誰來做？」
         ↓
L5  Technique（AI 技術）
    「用分類 / 回歸 / 檢索 / 生成 / 多模態哪種？」
```

需求分析的判斷流程：

```text
題目只說「改善某個業務結果」
    → 放在 L1，繼續追問流程

題目說「哪一段作業卡住」
    → 放在 L2，繼續追問決策點

題目說「AI 要判斷什麼」
    → 放在 L3，繼續追問輸入/輸出資料

題目說「有哪些資料、誰標註、資料能不能用」
    → 放在 L4，最後才選 L5 技術
```

#### 蝦皮客服 AI — 五層範例

| 層 | 內容 |
|---|---|
| L1 Business Goal | 首回時間從 5 分鐘降到 30 秒；月支出 NT$150 萬降到 NT$60 萬 |
| L2 Process | 消費者傳訊 → 客服讀訊 → 查訂單 → 回覆 |
| L3 Decision | 「這是物流問題、退款問題還是商品問題？」意圖分類 + 草擬回覆 |
| L4 Data | 過去 12 個月 50 萬筆客服對話（已脫敏）、訂單 API（Application Programming Interface，應用程式介面）、退款政策文件 |
| L5 Technique | 意圖分類（discriminative）+ RAG（retrieval）+ 草擬（generative） |

### 一步一步例子

把「我們要用 ChatGPT 改善客服」改寫成合格需求：

```text
Step 1 L1 業務目標：
首回時間從 5 分鐘降到 30 秒，客服成本從 NT$150 萬/月降到 NT$60 萬/月。

Step 2 L2 流程：
消費者傳訊 → 客服讀訊 → 查訂單 → 判斷問題類型 → 回覆。

Step 3 L3 決策：
AI 要判斷「這是物流、退款、商品、付款，還是其他問題」。

Step 4 L4 資料：
需要歷史對話、訂單狀態 API、退款政策文件，並先處理 PII（Personally Identifiable Information，個人可識別資訊）脫敏。

Step 5 L5 技術：
意圖分類 + RAG + 草擬回覆。這時才可以說用 ChatGPT 或其他 LLM。
```

### 比較表這樣讀

下面這張利害關係人表不是要背職稱，而是要判斷「哪種問題要先找誰」。題目問決策邏輯與例外情況，找 SME（Subject Matter Expert，領域專家）；題目問資料能不能拿，找 IT/Data；題目問個資與合規，找法遵/資安。

#### 利害關係人盤點（考題常出現「第一步要做什麼」）

| 類型 | 為什麼要訪談 |
|---|---|
| 業務 SME（客服主管、理賠主任） | 最清楚 decision 的邏輯與例外情況 |
| IT / Data（DBA, Database Administrator，資料庫管理員；資料工程師） | 知道資料能不能拿、合規限制在哪 |
| 法遵 / 個資（法務、資安） | 早期就要知道合規線，避免後期翻盤 |
| 最終使用者（一線客服） | 他們才是被 AI「輔助」的人，要設計 fallback |
| 高管 Sponsor | 給預算、擋跨部門阻力 |

### 記憶方式

```text
業·流·決·資·技
= 先問「為什麼」→ 再問「卡在哪」→ 再問「判斷什麼」→ 再問「資料在哪」→ 最後才問「用什麼技術」
```

### Exam Rule

```text
題目問「規劃階段第一步要做什麼」→ 利害關係人盤點 + 需求訪談
題目問「需求分析的順序」→ L1 業務目標 → L2 流程 → L3 決策 → L4 資料 → L5 技術
題目說「我們想用 ChatGPT 做客服」→ 這是直接跳到 L5，跳層錯誤
```

### Quick Check

公司說「我們要導入 AI 來改善客戶體驗」，這句話對應五層需求分析的哪一層？

答案：L1 Business Goal（業務目標層）。
理由：「改善客戶體驗」是商業目標的陳述，還沒指出哪段流程、哪個決策點、需要什麼資料或用哪種技術。這是需求分析的起點，不是終點。

## 3. 技術應用方案設計（Technical Solution Design）🔥🔥

### 先懂一句話

方案設計就是把「我們用 GPT + RAG 做客服 AI」這個決定，展開成四張設計圖：功能怎麼跑、資料怎麼流、模型怎麼選、整合點在哪——但不畫伺服器架構，那是 L21302 的事。

### Everyday Analogy

蓋房子的設計圖（L21202）和蓋房子的施工規範（L21302）是不同的東西。設計圖告訴你「客廳在哪裡、門怎麼開、水管走哪」（邏輯架構），施工規範告訴你「用幾號鋼筋、水泥比例多少、電線走法」（部署細節）。本課只畫設計圖。

### 先問自己一個問題

在設計方案時，先問：**我是在設計「AI 怎麼完成工作」，還是在設計「伺服器怎麼部署」？**

如果答案開始出現 Kubernetes、Prometheus、autoscaler、CI/CD，那就已經越界到 L21302。L21202 只需要把功能、資料、模型、整合講清楚。

### 技術說法

技術應用方案設計（Technical Solution Design）是把需求轉成 solution blueprint。它不是選型報告，也不是部署規格，而是描述 AI 解決方案在業務流程裡如何運作。

### 流程 / 選擇流程

```text
P1 需求分析完成（有 Requirements Doc）
    → P2 方案設計（本節）
    → 產出：解決方案藍圖（Solution Blueprint）
    → gate：四面向藍圖 + 預算核准
    → P3 資源規劃
```

### Key Concepts

#### 四面向方案設計

| 面向 | 內容 | 產出物 |
|---|---|---|
| **功能設計** | User flow、UI wireframe、fallback（AI 答不出來時轉人工）、信心分數門檻 | User journey map、線框圖 |
| **資料設計** | 來源、清洗 SOP、標註 rubric、版本控管、PII 處理 | Data dictionary、資料卡 |
| **模型設計** | Prompt template / RAG index 結構 / fine-tune 樣本量（引用 L21201 的選型決策） | Model card、Prompt spec |
| **整合設計** | 與 CRM（Customer Relationship Management，客戶關係管理；如 Salesforce）、LINE OA、內部 ticketing 的 API 串接 | 系統序列圖 |

### 一步一步例子

把客服 AI 的方案藍圖拆成四面向：

```text
Step 1 功能設計：
使用者送出問題 → AI 判斷意圖 → 查政策/訂單 → 草擬回覆 → 低信心轉人工。

Step 2 資料設計：
歷史對話先脫敏，政策文件建立版本，訂單資料只讀必要欄位。

Step 3 模型設計：
意圖分類負責分流，RAG 負責引用政策，LLM 負責草擬自然語言回覆。

Step 4 整合設計：
LINE OA 接收訊息，CRM 查客戶狀態，ticketing 系統接手人工 fallback。
```

### 比較表這樣讀

四面向表的讀法是：**先看使用者流程，再看資料能否支持流程，再看模型如何做判斷，最後看系統接點。** 如果題目問 fallback，通常是功能設計；問標註 rubric 或 PII，通常是資料設計；問 prompt/RAG/fine-tune，通常是模型設計；問 API 或 CRM，通常是整合設計。

#### 邏輯架構 vs 部署架構的界線（重要！）

```text
┌──────────────────────────────────────────┐
│  L21202 畫到這：邏輯架構                 │
│                                          │
│  [使用者] → [Web App] → [API Gateway]    │
│                          ↓               │
│                    [LLM Service]         │
│                          ↓               │
│                    [Vector DB] ← [資料源] │
└──────────────────────────────────────────┘
              ↓（交給 L21302 繼續）
┌──────────────────────────────────────────┐
│  L21302 才畫：部署架構                   │
│  k8s namespace / node pool / autoscaler  │
│  Istio service mesh / Prometheus / Grafana│
└──────────────────────────────────────────┘
```

### 記憶方式

```text
方設四圖：功·資·模·整
功能回答「怎麼用」
資料回答「吃什麼」
模型回答「怎麼判斷」
整合回答「接到哪裡」
```

### Exam Rule

```text
題目說「方案設計要包含什麼」→ 功能 / 資料 / 模型 / 整合 四面向
題目說方案設計要畫 k8s、Prometheus、Istio → 錯，那是 L21302 部署架構
題目說「fallback 機制在哪個設計面向」→ 功能設計（User flow 的一部分）
```

### Quick Check

技術應用方案設計中，「決定 AI 答不出來時要轉接真人客服」，這屬於四面向中的哪一項？

答案：功能設計（Functional Design）。
理由：轉接真人客服是一種 fallback 機制，屬於 User flow 的設計範疇，對應方案設計的「功能面向」，而非資料、模型或整合面向。

## 4. 目標設置（Goal Setting）🔥🔥🔥

### 先懂一句話

目標設置是用三層金字塔來對齊「我們要做到哪裡」——最頂層是一個北極星指標（不能多），中層是 OKR / SMART 目標，底層是每週追蹤的 3+2+1 KPI。

### Everyday Analogy

車子的儀表板上有很多錶：油量、水溫、轉速、時速。只有一顆**最大的時速錶**是北極星指標——它告訴你「現在開得快還是慢」。其他的錶（KPI）是輔助，但主錶只有一個。如果同時有三顆「最重要的錶」，你開車時就不知道看哪一個。

### 先問自己一個問題

設定目標前，先問：**如果這個指標變好，使用者真的得到價值、公司也真的得到成果嗎？**

如果答案只是「看起來很多人用」或「模型分數很漂亮」，通常只是 Vanity Metric（虛榮指標），不能當 North Star Metric。

### 技術說法

目標設置（Goal Setting）要把 strategy 轉成 measurable outcomes。最上層用 North Star Metric（北極星指標，NSM）定方向，中層用 OKR（Objectives and Key Results，目標與關鍵結果）/ SMART 寫成果，下層用 3+2+1 KPI（Key Performance Indicator，關鍵績效指標）做週期追蹤。

### 流程 / 選擇流程

```text
P1 需求分析完成（確認業務目標）
    → P2 方案設計階段同步進行目標設置（本節）
    → 產出：OKR / SMART 目標書
    → 連結到 P4 Pilot gate 的成功指標
```

### Key Concepts

#### 三層目標金字塔

```text
          ┌────────────────────┐
          │  North Star Metric │  ← 一個專案只能有一個
          │  北極星指標（NSM）  │     反映「使用者價值 + 商業成功」
          └────────────────────┘
                     │
          ┌────────────────────┐
          │  OKR / SMART       │  ← 季 / 年度可衡量目標（2-5 個 KR）
          └────────────────────┘
                     │
          ┌────────────────────┐
          │  3+2+1 KPI         │  ← 每週 / 月追蹤：3 營運 + 2 業務 + 1 風險
          └────────────────────┘
```

#### 北極星指標（North Star Metric, NSM）

**鐵律：一個專案只能有一個 NSM。**

| 專案 | 好的 NSM | 不好的 NSM |
|---|---|---|
| 蝦皮客服 AI | 「每週首回 < 30 秒的訊息佔比」 | 「客戶滿意度」（太抽象）、「API 呼叫次數」（虛榮） |
| LINE Bank 反詐 | 「每月可疑交易攔截金額」 | 「模型 accuracy」（內部指標，不反映業務） |
| 104 履歷配對 | 「每週完成面試的配對數」 | 「履歷閱覽次數」（沒有轉換） |

### 一步一步例子

把「改善客服品質」改成可考、可追蹤的目標：

```text
Step 1 先選一個 NSM：
每週首回 < 30 秒的訊息佔比。

Step 2 寫 Objective：
讓客服回覆速度明顯提升，同時維持可接受的服務品質。

Step 3 寫 Key Results：
KR1：首回 < 30 秒佔比從 20% 提升到 70%，於 2026-Q3 達成。
KR2：客服人力成本從 NT$150 萬/月降到 NT$60 萬/月。
KR3：CSAT 維持在 4.5/5 以上。

Step 4 拆 3+2+1 KPI：
3 operational：latency、覆蓋率、fallback 率。
2 business：首回時間、CSAT。
1 risk：幻覺/錯答投訴率。
```

### 比較表這樣讀

NSM 表不是在比較哪個指標「聽起來高級」，而是在看它有沒有同時連到使用者價值與商業成功。若指標只描述模型內部表現，例如 accuracy、F1-score、API 呼叫次數，通常要降格成 KPI。

#### SMART 原則（每個目標都要通過這五關）

| 字母 | 中文 | 白話檢查 |
|---|---|---|
| **S**pecific | 具體 | 「改善客服」❌ → 「降低首回時間」✅ |
| **M**easurable | 可衡量 | 「大幅改善」❌ → 「從 5 分鐘降到 30 秒」✅ |
| **A**chievable | 可達成 | 「accuracy 99.9%」對客服意圖分類通常 ❌ |
| **R**elevant | 相關 | 要與 North Star 對齊，不做離題指標 |
| **T**ime-bound | 有期限 | 「長期目標」❌ → 「2026-Q3 達成」✅ |

SMART 表的讀法是：先抓最容易缺的兩格，通常是 Measurable（沒有基線/數字）和 Time-bound（沒有期限）。考題常把「+10%」「大幅改善」寫得很像目標，但缺基線或期限就不是 SMART。

#### OKR 結構

- **Objective（O）**：一句話的方向感，帶 aspiration（有志向的）。
  - 例：「讓蝦皮客服成為台灣電商最快的回覆體驗。」
- **Key Result（KR）**：3-5 個可量化成果，用 SMART 規則寫。
  - KR1：每週首回 < 30 秒的訊息佔比 ≥ 70%（現狀 20%）
  - KR2：每月客服人力成本 ≤ NT$60 萬（現狀 150 萬）
  - KR3：客戶 CSAT ≥ 4.5 / 5（現狀 4.0）

**OKR vs SMART 的關係**：SMART 檢查「單一目標的品質」；OKR 是「一組（1 個 O + 多個 KR）的結構」。常一起用——用 SMART 的規則去寫 KR。

#### 3+2+1 KPI 組合 🔥

| 類別 | 數量 | 蝦皮客服 AI 範例 |
|---|---|---|
| **營運 KPI**（Operational） | 3 | (a) p95 latency < 2s、(b) AI 覆蓋率 ≥ 80%、(c) fallback 轉人工率 ≤ 15% |
| **業務 KPI**（Business） | 2 | (a) 首回時間中位數 < 30s、(b) CSAT ≥ 4.5 |
| **風險 KPI**（Risk） | 1 | 幻覺 / 錯答投訴率 ≤ 0.5% |

記憶鉤子：先看系統跑得穩不穩（3 operational），再看業務有沒有賺（2 business），最後看會不會出事（1 risk）。

#### Vanity Metrics 反模式（看了爽但沒用的指標）

| Vanity（虛榮）❌ | Actionable（可行動）✅ |
|---|---|
| Total API 呼叫次數 | 每次呼叫的成功率、單位成本 |
| 「模型 accuracy 95%」 | 生產環境 p95 latency + 客戶 CSAT |
| 「客戶滿意度 +10%」（沒基線） | 「CSAT 從 4.0 → 4.5，2026-Q3 達成」 |

**Vanity 辨識法**：問自己「這個指標變差了，我明天要做什麼？」答不出來 → 就是 vanity。

### 記憶方式

```text
一星、二到五 KR、三二一 KPI
一星 = NSM 只能一個
二到五 KR = OKR 的可衡量結果
三二一 = 3 營運 + 2 業務 + 1 風險
```

### Exam Rule

```text
「單一、能反映使用者價值與商業成功」→ 北極星指標（只能有一個）
「一個 O + 3-5 個 KR」→ OKR 結構
「Specific / Measurable / Achievable / Relevant / Time-bound」→ SMART 原則
「3 營運 + 2 業務 + 1 風險 KPI」→ 3+2+1 KPI 組合
「看了儀表板但不知道下一步」→ Vanity metric 反模式
「『客戶滿意度 +10%』沒有基線也沒有期限」→ 不是 SMART 目標
「一個專案有三個北極星指標」→ 錯誤，只能有一個
```

### Quick Check

某 AI 專案設定了「每週可疑交易攔截金額」作為追蹤指標，同時還設定了「模型 F1 score」、「系統 uptime」、「客訴率」另外三個「北極星指標」，請問這樣的做法有什麼問題？

答案：一個專案只能有一個北極星指標。
理由：「太多北極星 = 沒有北極星」。當四個指標同時宣稱是北極星，團隊決策時就沒有唯一依據，等於沒有指北針。應挑選其中一個最能同時反映「使用者價值 + 商業成功」的指標（本例宜選「可疑交易攔截金額」），其餘降格為 KR 或 KPI。

## 5. 資源分配（Resource Allocation）🔥🔥🔥

### 先懂一句話

AI 專案的資源分配要看五個向量，用口訣「**錢·人·算·料·時**」記。缺任何一個都會讓專案卡住。

### Everyday Analogy

登山出發前，你要準備：**錢**（登山費）、**人**（誰和你一起）、**裝備**（算力，鞋子、帳篷）、**食物**（料，補給品）、**時間**（幾天行程）。只帶了錢、忘帶食物，還是爬不到山頂。AI 專案也一樣，五個向量缺一不可。

### 先問自己一個問題

做資源規劃前，先問：**這份計畫是不是只寫了錢，卻忘了 AI 專案真正會卡住的人、算力、資料、時間？**

考題很常給一份「看似完整」的預算表，但其實只列總預算。L21202 要看的不是一個數字，而是五個向量是否都被規劃到。

### 技術說法

資源分配（Resource Allocation）是把 solution blueprint 轉成可執行的 resource plan。AI 專案和傳統 IT 最大不同是資料、算力、模型實驗、再訓練與變革管理都會吃資源，不能只用人月與硬體採購估算。

### 流程 / 選擇流程

```text
P2 方案設計完成（有藍圖）
    → P3 資源規劃（本節）
    → 產出：資源配置表 + 專案章程（Project Charter）
    → gate：RACI + 排程 + 合約 + 預算分配表
    → P4 試點開始
```

### Key Concepts

#### 五向量資源分配

| 向量 | 口訣 | 內容 | AI 專案的獨特坑 |
|---|---|---|---|
| Budget | **錢** | 五年 TCO；AI 初期估算建議 × 2 備用 | 資料清洗、再訓練不是一次性成本 |
| Headcount | **人** | PM（Project Manager，專案經理）/ DS（Data Scientist，資料科學家）/ MLE（Machine Learning Engineer，機器學習工程師）/ DE（Data Engineer，資料工程師）/ SME / QA（Quality Assurance，品質保證） | DS/MLE 年薪 NT$120-200 萬，難招 |
| Compute | **算** | GPU 型號、雲或地端、峰值 vs 常態 | 訓練用 A100/H100；推論 L40S/T4 夠用 |
| Data | **料** | 量、品質、標註成本、合規（清洗 2-8 週是常態） | 最常被低估；沒資料就沒模型 |
| Timeline | **時** | AI 落地**至少 6 個月**；含 pilot gate | 比傳統 IT 專案多 30-50% |

### 一步一步例子

把「客服 AI 預算 NT$500 萬」拆成可執行資源計畫：

```text
Step 1 錢 Budget：
列出 API/雲端、標註、資料清洗、教育訓練、合規、fallback 人力的預算。

Step 2 人 Headcount：
PM、DS、MLE、DE、SME、QA、法務/資安窗口都要有 owner。

Step 3 算 Compute：
判斷是 Managed API、雲端 GPU，還是地端推論；估尖峰與常態用量。

Step 4 料 Data：
確認歷史對話、訂單 API、政策文件是否可用，並安排 2-8 週清洗/標註。

Step 5 時 Timeline：
至少規劃 6 個月，並在 P4 前預留 Pilot gate，不把上線日當唯一里程碑。
```

### 比較表這樣讀

五向量表的讀法是找缺口：題目只給「錢」就是不完整；給了「錢 + 人」但沒有資料與算力，AI 也做不起來；給了上線日但沒有 Pilot gate，代表時程規劃失真。

#### AI 預算分配 heuristic 🔥

實務上沒有一個普遍認可的「AI 預算黃金比例」。以下是兩個常被引用的觀點：

**BCG 10/20/70 法則**（最常出現在考試與文獻中）：

```text
┌─────────────────────────────────────┐
│  10%  演算法 / 技術 Algorithms       │
│       模型選型、訓練、微調           │
├─────────────────────────────────────┤
│  20%  資料 Data                      │
│       資料工程、清洗、標註           │
├─────────────────────────────────────┤
│  70%  人員與流程 People & Process    │
│       變革管理、教育訓練、流程再設計 │
└─────────────────────────────────────┘
```

> ⚠️ **考試陷阱**：若題目問「BCG AI 投資比例法則」，記住 **70% 在人員與流程**，不是技術——與一般直覺相反。

**實務 Build 專案的成本結構**（Andrew Ng / Sculley et al. 隱藏技術債觀察，非命名比例）：
自建（Build / Self-host）ML 專案中，基礎設施（資料管線、整合、工具）往往是最大成本項，模型訓練本身反而佔少數。若採用 Managed API（OpenAI / Claude / Gemini API），基礎設施 CapEx 趨近於 0，主要成本轉為 API 呼叫 OpEx，整體結構大幅改變。

#### 人力配置模板（中型 AI 專案 5-10 人）

| 角色 | 縮寫 | 人數 | 主要責任 |
|---|---|---|---|
| 專案經理 | PM | 1 | 排程、溝通、風險追蹤 |
| 資料科學家 | DS | 1-2 | 模型選型、評估、實驗設計 |
| 機器學習工程師 | MLE | 1-2 | 訓練、部署（與 L21302 交界） |
| 資料工程師 | DE | 1 | pipeline、清洗、標註平台 |
| 業務 SME | SME | 1（兼） | 需求、rubric、UAT |
| QA / 測試 | QA | 0.5-1 | 離線評估 + 人評 |

AI 團隊 vs 傳統 IT 最大差異：多了 DS 與 DE，且 DS 工作充滿不確定性（實驗結果可能全推翻）——所以規劃要預留 buffer。

#### Build vs Buy 對資源分配的影響

| 決策 | 錢 | 人 | 算 | 料 |
|---|---|---|---|---|
| **Buy（Managed API）** | OpEx 為主 | MLE 少，工程師多 | 接近 0 | 中等 |
| **Build（Self-host）** | CapEx 大 | MLE 2+、DE 2+ | GPU 需要 | 量大 |
| **Hybrid** | CapEx + OpEx | 全配 | 地端 + 雲端 | 中大 |

Build vs Buy 表的讀法是：Buy 把算力與模型維運外包，成本變成 OpEx；Build 把控制權拿回來，但要自己扛 GPU、資料管線、MLE/DE 人力；Hybrid 則同時吃兩邊成本。考題若問「Managed API 情境」，不要直接套自建 ML 的成本結構。

### 記憶方式

```text
錢·人·算·料·時
= 有錢買車、有司機、有引擎、有燃料、有路線時間
少任何一個，車都開不到終點
```

### Exam Rule

```text
「規劃要同時考慮哪五項」→ 錢·人·算·料·時（缺一不可）
「AI 專案資料清洗要多久」→ 2-8 週（比傳統 IT 長）
「AI 落地週期最少多久」→ 6 個月起跳（含 pilot gate）
「BCG AI 投資比例法則哪個最多」→ 70% 人員與流程（不是技術/演算法）
「Build 專案成本大頭是什麼」→ 基礎設施（資料管線、整合），不是模型訓練本身
「Managed API 情境下 70/20/10 是否適用」→ CapEx 結構完全不同，不可直接套用
「規劃書只有預算欄位，沒有人力/算力/資料/時間」→ 失分，五向量缺一不可
```

### Quick Check

某新手 PM 把 AI 專案預算分配為「70% 模型訓練、20% 基礎設施、10% 其他」，這樣的分配有什麼問題？

答案：與 BCG 10/20/70 法則的精神相反，也忽視了自建 ML 專案中基礎設施是最大成本項的實務觀察。
理由：BCG 法則強調 **70% 應投入人員與流程**（變革管理、教育訓練），而非技術或模型訓練。在自建 ML 專案的實務中，資料管線、系統整合等基礎設施也往往遠超模型訓練本身的成本。把 70% 預算全丟給模型訓練，常導致資料管線沒建好、組織未準備好而上線延宕數個月。

## 6. 5 階段導入路線圖 + Pilot Gate 🔥🔥🔥

### 先懂一句話

AI 導入不是一口氣衝到上線，而是走五個階段，每個階段結束前都有一道「gate」必須通過才能繼續，Pilot gate 是其中最關鍵的一道——沒過就不能 rollout。

### Everyday Analogy

新藥上市要先做臨床試驗，Phase 1/2/3 每個階段都有嚴格的 safety gate，沒過就不能進下一期。AI 專案的五階段 + Pilot gate 就是同樣的邏輯：在小規模驗證（P4 Pilot）確認安全有效之後，才放量給所有用戶（P5 Rollout）。省略 gate 就像跳過臨床試驗直接賣藥。

### 先問自己一個問題

看到驗證活動時，先問：**現在要證明的是「做得出來嗎」、「有人要用嗎」，還是「小規模真實流程跑得順嗎」？**

這三個問題分別對應 PoC（Proof of Concept，概念驗證）、MVP（Minimum Viable Product，最小可行產品）、Pilot。考試最常把它們混在一起。

### 技術說法

5 階段導入路線圖（5-Phase Adoption Roadmap）是 L21202 的主流程。它用 gate 控制風險：每個階段完成必要產出後才能往下走，尤其 P4 Pilot gate 必須全部通過才可以 Rollout。

### 流程 / 選擇流程

```text
P1 需求分析 → P2 方案設計 → P3 資源規劃
    → P4 Pilot（本節重點）
    → gate：五條件全過
    → P5 Rollout / Scale（移交 L21302）
```

### Key Concepts

#### Pilot Gate 五條件（全部通過才能進 Rollout）🔥

1. **成功指標達成**：PoC 開始前就已書面化，不可事後修改
2. **使用者可接受度**：CSAT / NPS 達門檻
3. **風險紅線無觸發**：錯答率 / 偏見 / 隱私事件在容忍範圍
4. **經濟效益確認**：實際 TCO 不超預估 20%
5. **營運 SOP 就緒**：fallback、事故處理、退場機制完備

記憶鉤子：**指標 + 接受度 + 風險 + 經濟 + SOP**，五個都要打勾。

### 一步一步例子

客服 AI 從 PoC 走到 Rollout 的決策步驟：

```text
Step 1 PoC：
用 10 筆歷史對話測試「意圖分類 + RAG」做不做得出來。
答案是「做得出來嗎？」

Step 2 MVP：
做最小功能集，讓少數客服主管試用草擬回覆功能。
答案是「有人要用嗎？」

Step 3 Pilot：
開 5-20% 真實流量，測 latency、CSAT、錯答率、fallback、實際成本。
答案是「一路到底順不順？」

Step 4 Pilot gate：
五條件全過才進 Rollout；沒過就回 P1/P2 修正，不是降低門檻。
```

### 比較表這樣讀

PoC / MVP / Pilot 表的讀法是看「受眾 + 規模 + 核心問題」。內部技術團隊、小樣本、問可行性 = PoC；早期客戶、最小功能、問需求 = MVP；真實使用者子集、5-20% 流量、問全流程 = Pilot。

#### PoC vs MVP vs Pilot — 三者不能混用 🔥🔥

| 維度 | PoC | MVP | Pilot |
|---|---|---|---|
| 目的 | 可行性證明 | 最小可交付驗證 | 小規模真實部署 |
| 受眾 | 內部技術團隊 | 早期客戶 | 真實使用者子集 |
| 規模 | 1-10 筆樣本 | 最少功能集 | 5-20% 正式流量 |
| 所在階段 | L21201 Go/No-Go 之前 | L21202 方案設計（P2）中 | L21202 P4 |
| 核心問題 | 「**做得出來嗎？**」 | 「**有人要用嗎？**」 | 「**一路到底順不順？**」 |

#### AI 導入 vs 傳統 PMP 的差異（規劃時必須涵蓋的特殊性）🔥🔥

| AI 特殊議題 | 對規劃的影響 | 為何傳統 PMP 不夠 |
|---|---|---|
| **資料不確定性** | 清洗 2-8 週，要預留 buffer | PMP 假設需求穩定 |
| **模型不確定性** | 需 PoC / MVP / Pilot 三重 gate | Waterfall 假設「設計完就能做」 |
| **持續再訓練** | 是 OpEx 而非一次性 CapEx | PMP 把「專案結束」當終點，AI 沒有 |
| **人才稀缺** | DS/MLE 招聘 3-6 個月 | PMP 假設人力可調度 |
| **Feedback loop** | 上線後使用者行為改變，模型衰退 | PMP 沒有上線後持續再進化的概念 |
| **合規介面** | 早期就要把 L21203 風險納入預算 | PMP 把合規當末端檢查 |

#### Waterfall vs Agile（for AI）

| 維度 | Waterfall | Agile（for AI） |
|---|---|---|
| 需求 | 一次定死 | 滾動調整 |
| 產出 | 大版本，數月才一交付 | 2-4 週 sprint 持續交付 |
| AI 適用性 | ❌ 差（模型不確定性） | ✅ 好（可 pilot → 調 → 再 pilot） |

Waterfall vs Agile 表的讀法是：AI 的資料與模型結果會改變需求，所以通常要 Agile 加 Pilot gate。考題若只有 WBS、Gantt、Critical Path，沒有資料 buffer 與模型驗證 gate，就是傳統 PMP 套錯場景。

### 記憶方式

```text
PoC = 做得出來嗎
MVP = 有人要用嗎
Pilot = 一路到底順不順
Pilot gate = 指標、接受、風險、經濟、SOP 五個都過
```

### Exam Rule

```text
「AI 專案適合 waterfall 還是 agile」→ 多數 agile（因為模型不確定性），要補 pilot gate
「Pilot gate 要滿足哪五個條件才能 rollout」→ 指標 + 接受度 + 風險 + 經濟 + SOP
「PoC 在哪個階段」→ L21201 Go/No-Go 之前（問「做得出來嗎？」）
「MVP 在哪個階段」→ L21202 P2 方案設計（問「有人要用嗎？」）
「Pilot 在哪個階段」→ L21202 P4（問「一路到底順不順？」）
「只寫 WBS / Gantt / Critical Path」→ AI 規劃不夠，失分
「規劃階段不需要 pilot」→ 錯，AI 專案的模型不確定性讓 pilot gate 不可省略
```

### Quick Check

某 PM 在 AI 導入規劃中宣稱「只要技術可行性確認（PoC），就可以直接進入全量部署」，這樣的說法有什麼問題？

答案：PoC 通過只代表技術可行，無法代表可以全量部署；中間還缺少 MVP 驗證（有人要用嗎）和 Pilot（5-20% 真實流量的全路徑驗證）。
理由：Pilot gate 的五個條件（指標 + 接受度 + 風險 + 經濟 + SOP）一個都沒驗證。省略 Pilot 直接 rollout，AI 專案出問題的代價是傳統 PMP 式失敗的 5-10 倍。

## 7. 角色協作與產業參考框架 🔥🔥

### 先懂一句話

RACI 是一張「誰負責誰簽核」的對照表；產業框架（Microsoft CAF、Google CAF、IBM AI Ladder、Gartner、CRISP-DM）是幾家大廠把 AI 導入經驗整理出來的參考路線圖——考試只考「哪個框架屬於哪家、代表什麼概念」，不用背完整細節。

### Everyday Analogy

餐廳開一道新菜：廚師**負責（R）**做；主廚**當責（A）**對口味品質負最終責任；營養師**諮詢（C）**提供健康建議（雙向回饋）；收銀台**告知（I）**知道菜上了（單向通知）。每道菜只能有一個主廚（A）——兩個主廚都扛責任，等於沒有主廚。

### 先問自己一個問題

填 RACI 前，先問：**這件事出問題時，最後誰要負責拍板與扛責？**

很多人會把所有主管都填 A，但 RACI 的 A 不是「有關心的人」，而是唯一當責者。每列只能有一個 A。

### 技術說法

RACI 是責任分工矩陣（Responsibility Assignment Matrix）。它把每項活動拆成 Responsible（執行）、Accountable（當責）、Consulted（諮詢）、Informed（告知），用來避免跨部門 AI 導入時沒人負責或多人互相推諉。

### 流程 / 選擇流程

```text
P3 資源規劃
    → 產出 RACI 矩陣（本節）
    → gate：RACI + 排程 + 合約
    → P4 Pilot 執行
```

### Key Concepts

#### RACI 四字母定義

| 字母 | 中文 | 作用 | 每列可以有幾個？ |
|---|---|---|---|
| **R** Responsible | 負責（執行） | 實際動手做的人 | 可以多個 |
| **A** Accountable | 當責（簽核） | 扛專案最終成敗的人 | **每列只能 1 個** 🔥 |
| **C** Consulted | 諮詢 | 事前雙向討論 | 可以多個 |
| **I** Informed | 告知 | 事後單向通知 | 可以多個 |

口訣：**負·當·諮·告**

### 一步一步例子

填「Pilot 成功指標書面化」這一列：

```text
Step 1 找 A：
誰對成功標準是否合理負最終責任？通常是 PO 或業務 owner。

Step 2 找 R：
誰實際整理指標、寫文件、追版本？通常是 PM。

Step 3 找 C：
誰要給雙向意見？DS/MLE 給技術可行性，客服主管給業務門檻，法務給風險限制。

Step 4 找 I：
誰只需要知道結果？高管 Sponsor 可列 I，除非他是 Go/No-Go 的 A。
```

### 比較表這樣讀

RACI 表的讀法不是逐格背，而是先檢查每一列有沒有唯一 A，再看 R 是否是實際執行者，最後分辨 C/I：會參與討論的是 C，只收結果通知的是 I。

#### RACI 矩陣範例（蝦皮客服 AI）

| 專案活動 | PM | PO | DS | MLE | 客服主管 | 法務 | 高管 Sponsor |
|---|---|---|---|---|---|---|---|
| 利害關係人訪談 | R | **A** | C | I | C | C | I |
| 需求說明書撰寫 | R | **A** | C | I | C | C | I |
| 資料可得性評估 | I | C | R | I | I | C | I |
| Pilot 成功指標書面化 | R | **A** | C | C | C | C | I |
| 上線 Go/No-Go 會議 | R | C | C | C | C | C | **A** |

#### 常見 RACI 錯誤

| ❌ 錯 | ✅ 對 |
|---|---|
| 一列出現 3 個 A | 每列只能 1 個 A |
| R 和 A 都是同一人所以省略其一 | R + A 可以同一人，但兩個都要標出 |
| 把 C 填成 I（省事） | C 要收到雙向回饋；I 只是通知 |
| RACI 完成後不公開 | RACI 是共識文件，要公開、讓所有人簽核 |

#### RACI vs DACI

| 框架 | 強調 | 何時用 |
|---|---|---|
| **RACI** | 日常協作分工 | 每週例行任務、常態流程 |
| **DACI** | 重大決策（Driver / Approver / Contributors / Informed） | 一次性大決定（選方案、定路線） |

DACI 的 A = 拍板的 Approver；RACI 的 A = 扛最終成敗的 Accountable。考試頻率：**RACI >> DACI >> RAPID**。

RACI vs DACI 表的讀法是：如果題目是「日常任務誰做誰簽核」，選 RACI；如果題目是「一次重大決策誰推動誰拍板」，選 DACI。

#### 產業參考框架速查表

| 框架 | 代表廠商 | 關鍵結構 | 記憶鉤子 |
|---|---|---|---|
| **Microsoft CAF for AI** | 微軟 | 6 方法論（Strategy / Plan / Ready / Adopt / Govern / Manage）+ Secure 橫跨 | Plan 階段 = 本課範圍；Secure 不是第 7 個階段而是 cross-cutting |
| **Google Cloud AI Adoption Framework** | Google | 4 大支柱，跨多成熟度階段 | **4 pillars：People / Process / Technology / Data**（注意：只有 4 個，Governance 與 Culture 不是獨立支柱） |
| **IBM AI Ladder** | IBM | Collect → Organize → Analyze → Infuse | **先有資料才有 AI**（資料梯度） |
| **Gartner AI Maturity** | Gartner | 5 級（Awareness → Active → Operational → Systemic → Transformational） | 五級成熟度；只有約 6% 企業到最高級 |
| **CRISP-DM** | 通用 | 6 階段（業務理解 → 資料理解 → 資料準備 → 建模 → 評估 → 部署）**循環式，非線性** | **業·資·備·模·評·部**；最廣用的 DM/ML 方法論；Evaluation 後可 loop 回 Business Understanding |

產業框架表的讀法是記「廠商 + 關鍵鉤子」：Microsoft = CAF 六方法論 + Secure 橫跨；Google = 4 pillars；IBM = AI Ladder 資料梯；Gartner = 五級成熟度；CRISP-DM = 六階段循環式。

### 記憶方式

```text
RACI = 負·當·諮·告
每列只有一個「當」

框架鉤子：
微軟六法、Google 四柱、IBM 資料梯、Gartner 五級、CRISP 六步循環
```

### Exam Rule

```text
「每列 A 只能有一個」→ RACI 的 Accountable 鐵律
「R 和 A 有什麼不同」→ R 是執行，A 是扛最終成敗
「一次性重大決策用什麼框架」→ DACI
「AI 導入成熟度分五級」→ Gartner AI Maturity Model
「Collect / Organize / Analyze / Infuse」→ IBM AI Ladder
「4 pillars：People / Process / Technology / Data」→ Google Cloud AI Adoption Framework（不是 6 pillars；Governance/Culture 不是獨立支柱）
「6 方法論 + Secure cross-cutting」→ Microsoft CAF for AI
「業·資·備·模·評·部 六階段」→ CRISP-DM
「CRISP-DM 是線性還是循環」→ **循環式**；Evaluation 後若未達目標，loop 回 Business Understanding（考試常出「線性 vs 循環」選擇題）
```

### Quick Check

專案的「資料可得性評估」活動，哪個角色應該填 A（Accountable，當責）？

答案：通常是資料工程師（DE）或資料負責人，因為他們對資料是否可用、合規性負最終責任。
理由：RACI 中 A 必須是對該活動的成果「最終負責扛鍋」的人。資料可得性是資料工程師 DE 最能掌控也最終負責的事，PM 只是協調，DS 是執行方（R）。

## 8. Exam Decision Trees（考試決策樹）

### 決策樹 1：看到題目 → 判斷屬於哪個單元

```text
題目描述的是…
│
├─ 「評分矩陣 / TCO / ROI / 選工具 / 雙約束過濾」？
│  └─ L21201 評估，不是 L21202
│
├─ 「EU AI Act / NIST RMF / ISO 42001 / 風險分級 / 風險登記冊」？
│  └─ L21203 風險，不是 L21202
│
├─ 「k8s / CI/CD / MLOps / 監控 / 模型漂移 / Canary」？
│  └─ L21302 部署，不是 L21202
│
└─ 「需求分析 / RACI / OKR / 資源分配 / pilot gate / roadmap」？
   └─ L21202 規劃，答這個 ✅
```

### 決策樹 2：判斷目標類型

```text
題目在問什麼目標？
│
├─ 「整個專案最重要的單一指標，反映使用者價值和商業成功」？
│  └─ 北極星指標（North Star Metric），只能有一個
│
├─ 「一個方向性陳述 + 3-5 個可量化成果」？
│  └─ OKR（Objective + Key Results）
│
├─ 「每個目標是否具體可衡量可達成相關有期限」？
│  └─ SMART 原則（用來檢查目標品質）
│
└─ 「每週 / 月追蹤的指標，分三層」？
   └─ 3+2+1 KPI（3 營運 + 2 業務 + 1 風險）
```

### 決策樹 3：判斷驗證活動類型

```text
題目在問哪種驗證？
│
├─ 「技術可行性、1-10 筆樣本、內部驗證、L21201 之前」？
│  └─ PoC（Proof of Concept）
│  　 核心問題：「做得出來嗎？」
│
├─ 「最小功能集、對外部客戶驗證、方案設計階段」？
│  └─ MVP（Minimum Viable Product）
│  　 核心問題：「有人要用嗎？」
│
└─ 「5-20% 真實流量、全路徑驗證、L21202 P4」？
   └─ Pilot
   　 核心問題：「一路到底順不順？」
```

### 決策樹 4：判斷資源分配是否完整

```text
資源配置表有沒有涵蓋五向量？
│
├─ 只寫了預算（錢）？
│  └─ 不完整，還缺人 / 算 / 料 / 時
│
├─ 有錢有人但沒有算力和資料？
│  └─ 不完整，AI 專案沒資料就沒模型
│
└─ 錢·人·算·料·時 都有？
   └─ 基本完整 ✅
   再確認：若題目問 BCG 法則，70% 在人員與流程，不是技術或基礎設施
```

### 決策樹 5：判斷 RACI 是否正確

```text
RACI 矩陣合理嗎？
│
├─ 某一列有 2 個以上的 A？
│  └─ 錯誤，每列只能 1 個 A
│
├─ C 和 I 搞混（把應回饋的人填 I）？
│  └─ 錯誤，C 是雙向諮詢，I 是單向告知
│
└─ 每列一個 A、R 可多個、C/I 分清楚？
   └─ 正確 ✅
```

## 9. Trap Clinic（考試陷阱診所）

### Trap 1：把 L21202 當成 L21201（拿規劃題去算評分矩陣）

錯。L21201 是「選工具」，L21202 是「選定後怎麼做」。看到「AI 導入規劃」就去畫加權評分矩陣、算 ROI，是把兩個單元混用。

Exam fix：

```text
題目說「規劃阶段」「RACI」「roadmap」「資源分配」→ L21202
題目說「評分矩陣」「TCO」「ROI」「雙約束」→ L21201
```

---

### Trap 2：在規劃書裡寫 EU AI Act / NIST RMF

錯。這些是 L21203（風險）的內容。L21202 只是把「預留合規預算」當作規劃輸入，不展開風險框架。

Exam fix：

```text
「EU AI Act 分級」「NIST RMF 對映」「ISO 42001 條文」→ L21203，不是 L21202
L21202 只寫：「預留 10% 預算給 governance」，不寫框架細節
```

---

### Trap 3：方案設計時畫 k8s / Prometheus / Istio

錯。L21202 只畫邏輯架構（誰呼叫誰、資料流、整合點）；k8s、CI/CD、監控全都是 L21302 的事。

Exam fix：

```text
方案設計提到 k8s / autoscaler / Prometheus → 越界，答案應排除這些
方案設計只問到：功能 / 資料 / 模型 / 整合 四面向
```

---

### Trap 4：RACI 的 R 和 A 搞混

錯。R 是「負責執行的人」（可以多個），A 是「出事要扛責任的人」（每列只能一個）。把兩個混用或省略其一都是錯誤。

Exam fix：

```text
「實際動手做的人」→ R（Responsible）
「扛最終成敗的人，每列只能一個」→ A（Accountable）
一列出現兩個 A → 違反 RACI 原則
```

---

### Trap 5：北極星指標可以有很多個

錯。「太多北極星 = 沒有北極星」。一個專案只能有一個 NSM，其餘指標應降格為 KR 或 KPI。

Exam fix：

```text
「反映使用者價值與商業成功的單一指標」→ North Star Metric（NSM），只能一個
「有三個北極星指標」→ 錯，只能選一個，其餘降為 KPI
```

---

### Trap 6：以為「客戶滿意度 +10%」是 SMART 目標

錯。沒有基線、沒有期限，「+10%」是 Vanity metric。

Exam fix：

```text
「大幅改善」「+10%」「長期目標」→ 不是 SMART，缺 Measurable / Time-bound
SMART 正確寫法：「CSAT 從 4.0 提升至 4.5，於 2026-Q3 達成」
```

---

### Trap 7：PoC / MVP / Pilot 三者混用

錯。三者在不同階段回答不同問題，不能互換。

Exam fix：

```text
「可行性驗證、1-10 筆樣本、L21201 Go/No-Go 之前」→ PoC
「最小可交付、對客戶驗證、方案設計階段」→ MVP
「5-20% 真實流量、P4 階段、全路徑驗證」→ Pilot
```

---

### Trap 8：以為「70%」要給模型或技術

錯。BCG 10/20/70 法則中，70% 投入**人員與流程**（變革管理、教育訓練、流程再設計），演算法/技術只佔 10%——與新手直覺完全相反。

Exam fix：

```text
題目問「BCG AI 投資比例法則，哪個佔 70%？」→ 人員與流程（People & Process），不是技術
「10/20/70 法則」→ 10% 演算法、20% 資料、70% 人員與流程
考試陷阱：「70% 基礎設施」或「70% 模型訓練」都不是 BCG 命名法則的說法
```

---

### Trap 9：需求分析跳層（L1 直接跳 L5）

錯。「我們想做 AI」（L1）→「我們用 ChatGPT」（L5）跳過了 L2 流程、L3 決策、L4 資料三層。

Exam fix：

```text
需求分析順序：業務目標 → 流程 → 決策 → 資料 → 技術（業·流·決·資·技）
跳層 = 需求書不完整，上線後必出問題
```

---

### Trap 10：AI 導入規劃等同傳統 PMP（只寫 WBS + Gantt）

錯。AI 規劃必須額外涵蓋：資料不確定性 buffer、模型不確定性 pilot gate、持續再訓練 OpEx、人才稀缺 HR 計畫、feedback loop 監測。只有 PMP 模板一定失分。

Exam fix：

```text
AI 規劃 vs 傳統 PMP 的五大差異：
「資料不確定性」「模型不確定性」「持續再訓練 OpEx」「人才稀缺」「feedback loop」
少寫任何一點都要失分
```

---

### Trap 11：Pilot Gate 通過條件可以事後修改

錯。成功指標必須在 PoC 開始前就**書面化**，Pilot 結束後不可以因為「差一點」就降低門檻（post-hoc rationalization）。

Exam fix：

```text
Pilot 成功指標 → 必須在開始前書面化，不可事後修改
Pilot 未過 → 回到 P1/P2 重新修正，不是降低標準強行 rollout
```

## 10. Practice Questions（練習題）

### 10.1 需求分析（Requirements Analysis）

**Q1.** 一家銀行希望導入 AI 來「改善理賠效率」，這句話落在需求分析五層（L1-L5）的哪一層？

答案：L1（Business Goal 業務目標層）。
理由：「改善理賠效率」是商業目標的陳述，尚未指出哪段流程（L2）、哪個決策點（L3）、需要什麼資料（L4）或用哪種技術（L5）。這是需求分析的起點。

---

**Q2.** 需求分析的正確順序是？

答案：業務目標 → 流程 → 決策 → 資料 → 技術（L1 → L2 → L3 → L4 → L5）。
理由：每一層都必須從上一層推導出來，跳層會導致需求書不完整，上線後必定出現問題。

---

**Q3.** 需求分析階段，「規劃第一步要做什麼」的正確答案是？

答案：利害關係人盤點 + 需求訪談。
理由：在動筆寫需求書之前，必須先確認「誰是利害關係人、他們的需求和限制是什麼」，這才是規劃階段真正的起點。不是「寫程式」，也不是「選 GPU」。

---

**Q4.** 某公司的需求分析文件寫著：「我們要用 RAG 技術處理客服問題」，這份文件哪裡有問題？

答案：從 L1 業務目標直接跳到 L5 技術，跳過了 L2 流程、L3 決策、L4 資料三層。
理由：需求分析五層必須逐層對應，直接指定技術而不說明「哪段流程有問題」「AI 要替代哪個決策」「需要哪些資料」，是典型的跳層錯誤。

---

### 10.2 目標設置（Goal Setting）

**Q5.** 「每週可疑交易攔截金額」和「模型 F1-score」哪一個更適合作為北極星指標？

答案：「每週可疑交易攔截金額」。
理由：北極星指標必須同時反映「使用者價值」和「商業成功」。「可疑交易攔截金額」直接連結到業務成果（保護用戶財產）；「模型 F1-score」是內部技術指標，不反映業務價值，屬於 KPI 而非 NSM。

---

**Q6.** 以下哪個目標符合 SMART 原則？
(A) 「改善客服品質」
(B) 「大幅提升首回時間」
(C) 「首回時間從 5 分鐘降到 30 秒，於 2026-Q3 達成」
(D) 「客戶滿意度 +10%」

答案：(C)。
理由：(A) 不具體；(B) 沒有可衡量基準；(D) 沒有基線（從多少 +10%？）也沒有期限；只有 (C) 同時滿足 Specific（首回時間）、Measurable（5 分鐘 → 30 秒）、Achievable（技術可行）、Relevant（與業務目標對齊）、Time-bound（2026-Q3）五個條件。

---

**Q7.** 3+2+1 KPI 中，「1」代表什麼類型的 KPI？

答案：風險 KPI（Risk KPI）。
理由：3+2+1 的組合是：3 個營運 KPI（Operational）+ 2 個業務 KPI（Business）+ 1 個風險 KPI（Risk）。「1」是風險指標，例如「幻覺 / 錯答投訴率 ≤ 0.5%」，一旦超過門檻就觸發 L21203 的風險處理流程。

---

**Q8.** 「Total API 呼叫次數達 100 萬次」，這是 North Star Metric 還是 Vanity Metric？

答案：Vanity Metric（虛榮指標）。
理由：「API 呼叫次數」只反映系統使用量，不反映使用者價值（呼叫了卻沒解決問題也算一次）。用 Vanity 辨識法：「這個指標變差，我明天要做什麼？」答不出來就是 vanity。

---

### 10.3 資源分配（Resource Allocation）

**Q9.** AI 專案的資源分配必須涵蓋哪五個向量？

答案：錢（Budget）、人（Headcount）、算（Compute）、料（Data）、時（Timeline）。
理由：「錢·人·算·料·時」口訣，缺少任何一個都會讓專案中途卡住。特別是「料」（資料）在 AI 專案最常被低估。

---

**Q10.** BCG 10/20/70 法則中，應分配最多比例（70%）的項目是？

答案：人員與流程（People & Process），佔 70%。
理由：BCG 法則強調 AI 專案的成敗關鍵在於人員變革管理、教育訓練和流程再設計，而非演算法本身（只佔 10%）。這與一般人以為「AI 的錢要花在技術上」的直覺相反，是常見的考試陷阱。

---

**Q11.** 若題目描述一個自建（Build/Self-host）ML 專案的成本結構，基礎設施（資料管線、整合）通常佔多大比例？這和 BCG 10/20/70 法則衡量的是同一件事嗎？

答案：實務上基礎設施確實是自建 ML 專案的最大成本項，但這與 BCG 10/20/70 法則是不同維度的觀察。
理由：BCG 法則衡量的是「AI 轉型投資組合」的分配重心（人員 70%），不是單一技術專案的 CapEx 明細。自建 ML 專案的 CapEx 成本大頭是基礎設施（非模型訓練），但採用 Managed API 時基礎設施 CapEx 趨近於 0，主要成本轉為 API 呼叫 OpEx。這兩個觀察維度不同，不可混用。

---

**Q12.** AI 導入的資料清洗作業，通常需要多少時間？

答案：2-8 週（比傳統 IT 專案長）。
理由：AI 的資料清洗涉及格式標準化、標註、合規（PII 處理）等複雜作業，且常有意外發現（資料品質比預期差）。規劃時必須預留這段 buffer，否則整個 roadmap 時程都會被拖延。

---

### 10.4 Pilot 與 RACI

**Q13.** Pilot Gate 的五個通過條件是哪些？

答案：
1. 成功指標達成（事前書面化，不可事後修改）
2. 使用者可接受度達門檻（CSAT / NPS）
3. 風險紅線無觸發（錯答率 / 隱私事件在容忍範圍）
4. 經濟效益確認（實際 TCO 不超預估 20%）
5. 營運 SOP 就緒（fallback、事故處理、退場機制）

理由：五個條件要全部通過才能進 Rollout；任何一個未達標，就必須回到 P1/P2 修正，而不是強行 rollout 或降低門檻。

---

**Q14.** RACI 矩陣中，某一列出現了三個 A，代表什麼問題？

答案：違反 RACI 原則，每列只能有一個 A（Accountable）。
理由：Accountable 的意義是「出事要扛最終責任的人」。若有三個人都是 A，代表沒有任何人真正負最終責任——結果是出問題時三個人互相推諉。RACI 要求每列 A 有且只有一個。

---

**Q15.** 「Driver / Approver / Contributors / Informed」是哪個框架？它與 RACI 的使用場景有何不同？

答案：DACI 框架。DACI 適合「一次性重大決策」（選方案、決路線）；RACI 適合「日常協作的常態流程」。
理由：DACI 的 Approver（A）= 拍板的人；RACI 的 Accountable（A）= 扛最終成敗的人，兩者意義不同。考試頻率：RACI >> DACI >> RAPID。

---

### 10.5 產業框架與混合概念

**Q16.** IBM AI Ladder 的四個階段依序是？

答案：Collect → Organize → Analyze → Infuse。
理由：IBM 強調「先有資料（Collect / Organize）才有 AI（Analyze / Infuse）」，這個「資料梯度」是 AI 導入的前置條件。記憶鉤子：**先有資料才有 AI**。

---

**Q17.** Microsoft Cloud Adoption Framework（CAF）for AI 的 Secure 在框架中是什麼定位？

答案：Secure 是 cross-cutting（橫跨式）面向，貫穿全部 6 個方法論，不是第 7 個線性階段。
理由：Microsoft CAF 有 6 個方法論（Strategy / Plan / Ready / Adopt / Govern / Manage），Secure 不是其中一個順序階段，而是套用在所有 6 個方法論之上的橫跨式安全要求。

---

**Q18.** CRISP-DM 的六個階段依序是？CRISP-DM 是線性還是循環流程？

答案：業務理解 → 資料理解 → 資料準備 → 建模 → 評估 → 部署；**循環式（iterative）**，不是線性。
理由：口訣「**業·資·備·模·評·部**」。CRISP-DM 是最廣用的資料科學 / ML 專案方法論，L21202 的 5-phase roadmap 可以對應到 CRISP-DM 的前 4-5 個階段。重要：評估（Evaluation）階段後，若模型未達目標，流程**回到業務理解（Business Understanding）**重新定義問題，而不是直接結束——這是常見考試陷阱（選「線性」vs「循環」）。

---

**Q19.** Gartner AI Maturity Model 最高等級（Level 5）的名稱是什麼？全球企業中約有多少比例達到這個等級？

答案：Level 5 = Transformational（轉型化）；全球約只有 6% 的企業達到這個等級。
理由：Gartner 五級分別為 Awareness → Active → Operational → Systemic → Transformational。考題若問「AI 成熟度五級分類」→ 答 Gartner AI Maturity Model。

---

### 10.6 綜合情境 / Mixed Traps

**Q20.** AI 規劃書寫了「使用 Agile 方法論，2 週 sprint，並在 P4 結束後設置 Pilot Gate」，這樣的規劃方法是否合適？

答案：合適。
理由：Agile 適合 AI 專案（模型不確定性高，需要滾動調整），2 週 sprint 可持續交付；Pilot Gate 是 AI 規劃的必要設計，確保小規模驗證通過後才全量部署。這個規劃同時包含了 Agile 的靈活性和 Pilot Gate 的風險控制。

---

**Q21.** 某 AI 規劃書列出：「(1) 成功指標：CSAT ≥ 4.5；(2) 利害關係人：客服主管；(3) 資源分配：預算 NT$500 萬；(4) 里程碑：6 個月上線」。請問這份規劃書缺少什麼重要元素？

答案：缺少：需求分析五層（業·流·決·資·技）、RACI 矩陣、五向量資源分配（只寫了預算，沒有人力/算力/資料/時間）、預算分配說明（各項目如何分配）、技術應用方案設計四面向（功能/資料/模型/整合）、Pilot Gate 五條件。
理由：L21202 的四大產出物是：需求說明書、解決方案藍圖、OKR / SMART 目標書、資源配置表 + 專案章程。每個產出物都有對應的必要內容，光寫成功指標、人員、總預算、時程是遠遠不夠的。

---

**Q22.** 「我們計畫導入 AI 後追蹤：(1) 系統 uptime 99.9%、(2) API 成功率 98%、(3) AI 覆蓋率 80%、(4) 首回時間中位數 < 30s、(5) CSAT ≥ 4.5、(6) 幻覺投訴率 ≤ 0.5%」。請用 3+2+1 KPI 框架對這 6 個指標進行分類。

答案：
- 3 個營運 KPI（Operational）：系統 uptime 99.9%、API 成功率 98%、AI 覆蓋率 80%
- 2 個業務 KPI（Business）：首回時間中位數 < 30s、CSAT ≥ 4.5
- 1 個風險 KPI（Risk）：幻覺投訴率 ≤ 0.5%

理由：3+2+1 組合剛好對應「系統跑得穩（營運）→ 業務有沒有賺（業務）→ 會不會出事（風險）」三層，這個分類同時也符合本題的指標特性。

---

**Q23.** 「AI 專案和傳統 PMP 專案最主要的差異有哪些？」請列出至少四點。

答案：
1. **資料不確定性**：清洗 2-8 週是常態，需求常隨資料發現而改變
2. **模型不確定性**：需要 PoC / MVP / Pilot 三重 gate，不能直接走 waterfall
3. **持續再訓練**：上線後持續有 OpEx（再訓練、版本更新），不是一次性 CapEx
4. **人才稀缺**：DS/MLE 招聘需要 3-6 個月，規劃要提前啟動 HR
5. （加分）**Feedback loop**：上線後使用者行為改變，模型會衰退，需要持續監測

理由：只寫 WBS / Gantt / Critical Path 的傳統 PMP 模板無法涵蓋以上特殊性，在 IPAS 中級考試中一定失分。

## Final Oral Recall

考前最後 3 分鐘，把這幾句唸一次：

1. **需求分析走五層，不能跳層**：業務目標 → 流程 → 決策 → 資料 → 技術（業·流·決·資·技）。直接從「想做 AI」跳到「選 ChatGPT」是跳層錯誤。

2. **資源分配用五向量**：錢·人·算·料·時，缺一不可。**BCG 10/20/70 法則**：10% 演算法、20% 資料、**70% 人員與流程**——與直覺相反，70% 不是給技術或模型。

3. **北極星指標只能有一個**：同時反映「使用者價值 + 商業成功」的單一指標。多個北極星 = 沒有北極星。其餘指標分 3+2+1 KPI 追蹤。

4. **Pilot Gate 五條件全過才能 Rollout**：指標達成 + 使用者可接受度 + 風險紅線無觸發 + 經濟效益確認 + SOP 就緒。成功指標必須在開始前書面化，不可事後改。

5. **RACI 每列只能一個 A**：R = 執行（可多個），A = 當責（每列只有一個），C = 諮詢（雙向），I = 告知（單向）。R 和 A 不能混用，一列多個 A 是違反 RACI 原則。

6. **L21202 的邊界要守住**：選工具算分數 = L21201；EU AI Act / 風險分級 = L21203；k8s / CI/CD / 模型漂移 = L21302。方案設計只畫邏輯架構，不畫伺服器配置。

7. **AI 規劃不等於傳統 PMP**：AI 規劃必須額外包含「資料不確定性 buffer、模型不確定性 pilot gate、持續再訓練 OpEx、人才稀缺 HR 計畫、feedback loop 監測」這五個傳統 PMP 沒有的元素。

---

## Final Study Advice

不要只背口訣和名詞。IPAS 中級 L21202 真正在測的是：**當題目給你一個 AI 導入場景，你能不能判斷「現在在哪個階段、要做什麼事、不能做什麼事」。**

考試遇到不確定的題目，先問自己三個問題：
1. 「這個描述是在**選工具**（L21201）、**規劃**（L21202）、**管風險**（L21203）還是**部署**（L21302）？」
2. 「題目描述的是哪個產出物？是需求書、方案藍圖、OKR 目標書、還是資源配置表 + RACI？」
3. 「如果答案有 k8s / EU AI Act / 加權評分矩陣，那十有八九是讓你選『這不是 L21202 的內容』。」

守住邊界，把五層需求分析、四面向方案設計、三層目標架構、五向量資源分配、五階段 Roadmap、RACI、Pilot Gate 五條件這七個核心架構說清楚——就能拿到 L21202 應得的分數。
